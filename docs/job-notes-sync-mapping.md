# 找工作笔记同步映射说明

## 目的

这份说明用来固定 `/Users/shifusong/Projects/找工作` 与站内 `notes/面试学习` 的同步关系。

以后更新 `/Users/shifusong/Projects/找工作` 时，优先先看这份文档，再决定哪些内容是：

- 直接同步
- 重命名后同步
- 只在站内保留增强版
- 不应纳入站点

这样可以避免两类问题：

1. 源目录更新了，但站内漏同步
2. 站内已经做过结构增强，后续同步时被误覆盖

## 总原则

### 1. 源目录和站内不是完全镜像关系

`/Users/shifusong/Projects/找工作` 是你的原始学习与素材工作区。  
站内 `notes/面试学习` 是面向阅读、搜索、导航和移动端消费的发布版知识库。

所以站内允许存在这些增强：

- 目录级 `README.md` 导航页
- 英文版资料
- 更适合网站展示的标题与路径
- 站内可跳转的 Markdown 内链
- 少量面向岗位表达的泛化命名

### 2. 不改动源目录原文件

同步时遵循：

- 读取 `/Users/shifusong/Projects/找工作`
- 复制或参考内容进入 `notes/`
- 不回写源目录

### 3. 站内的“增强文件”不反向要求源目录存在

例如：

- 英文面试资料
- 站内 README 导航页
- 面试冲刺导航页
- 为了网站结构新增的汇总页

这些可以只存在于站内。

## 每次同步的标准流程

### 1. 先看源目录有没有新增或修改

重点关注：

- `/Users/shifusong/Projects/找工作/面试学习/AI产品经理`
- `/Users/shifusong/Projects/找工作/面试学习/开发`
- `/Users/shifusong/Projects/找工作/面试学习/项目管理产品经理`
- `/Users/shifusong/Projects/找工作/工作后的开发项目`
- `/Users/shifusong/Projects/找工作/实习相关项目`
- `/Users/shifusong/Projects/找工作/AI相关项目`

### 2. 判断属于哪一类同步

- `1:1 同步`
- `重命名后同步`
- `项目包重组同步`
- `站内增强，无需回源`

### 3. 复制或更新站内文件

同步时同时处理：

- 标题
- 站内路径链接
- README 导航关系
- 必要的命名统一

### 4. 重新生成索引

```bash
cd /Users/shifusong/notes-site
python3 scripts/generate-index.py
```

### 5. 做两类校验

- 链接校验
- 结构校验

结构校验至少看：

- 分类树是否正确
- README 是否仍然是入口页
- 新文件是否出现在搜索和列表里

## 判断差异时的口径

以后如果比对出差异，优先按下面口径判断：

### 属于正常差异

- 绝对路径改成站内路径
- 增加英文版
- 增加 README 导航页
- 目标公司名称被泛化为岗位表达
- 项目包为了表达而重写结构

### 属于需要警惕的差异

- 源目录新增了一篇主线笔记，站内完全没有
- 源目录更新了关键业务事实，站内还是旧版本
- 站内出现明显笔误、重复词或断链
- README 导航没有把新材料串进去

## 推荐做法

以后每次你说“把 `/找工作` 的新内容同步到网站”，就按这份说明执行：

1. 先比对源目录和站内目录
2. 区分“主线同步”与“站内增强”
3. 只把真正的新内容和有效结构变化并入站内
4. 最后统一做索引与链接校验

这样同步会稳定很多，也能避免把站内已经打磨好的结构反复覆盖掉。
