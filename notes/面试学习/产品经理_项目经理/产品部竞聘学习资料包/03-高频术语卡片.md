# 高频术语卡片

本文件用于快速纠正口径。每张卡片按“一句话解释、业务场景、容易混淆点、面试表达”整理。

## 1. 产品方案

| 项目 | 内容 |
|---|---|
| 一句话解释 | 产品方案是把客户需求、投资策略、风险收益和产品形态转成可审批、可申报、可发行的业务设计 |
| 业务场景 | 新产品、新策略、新系列、产品矩阵补位 |
| 容易混淆点 | 不是单纯营销材料，也不是说明书初稿 |
| 面试表达 | 产品方案决定后续产品委员会审批、风险评级、销售文件和系统字段，是产品部把市场需求产品化的起点 |

## 2. 产品委员会

| 项目 | 内容 |
|---|---|
| 一句话解释 | 产品委员会是产品规划、产品方案审批和跨部门共识形成机制 |
| 业务场景 | 定期复盘、下阶段产品计划、新产品或新策略审批 |
| 容易混淆点 | 不只是开会审批，更承接市场、同业、销售和投资视角 |
| 面试表达 | 产品委员会体现产品部从单只产品执行到产品矩阵规划的能力 |

## 3. 投资经理初始设置/定投资经理

| 项目 | 内容 |
|---|---|
| 一句话解释 | 定投资经理是为新发产品明确主/副投资经理并进入投资运作链条的前置流程 |
| 业务场景 | 新发产品申报登记前 |
| 容易混淆点 | 培训中的“定投”不是基金定投，而是内部简称 |
| 面试表达 | 定投资经理解决“谁管理产品、如何进入投资和系统链条”的问题 |

## 4. 申报登记

| 项目 | 内容 |
|---|---|
| 一句话解释 | 申报登记是向理财登记中心报送基础材料并取得理财产品登记编码 |
| 业务场景 | 产品发行前的登记编码获取 |
| 容易混淆点 | 申报登记不是具体上架销售，很多发行字段可能尚未最终确定 |
| 面试表达 | 申报登记偏产品资格和基础信息报送，核心是取得登记编码 |

## 5. 发行登记

| 项目 | 内容 |
|---|---|
| 一句话解释 | 发行登记是把本次募集、成立、到期、渠道、份额、费率等具体发行要素补齐并报送 |
| 业务场景 | 已取得登记编码，准备具体起募和渠道上架 |
| 容易混淆点 | 发行登记比申报登记更贴近销售上架 |
| 面试表达 | 发行登记解决“这次怎么卖、谁来卖、卖多少、费用怎么收”的问题 |

## 6. 理财产品登记编码

| 项目 | 内容 |
|---|---|
| 一句话解释 | 理财产品登记编码是产品通过理财登记中心申报登记后取得的产品基础编码 |
| 业务场景 | 申报登记完成后进入发行登记 |
| 容易混淆点 | 不等同于行内标识码、理财产品代码或销售代码 |
| 面试表达 | 登记编码是后续发行登记和渠道上架的基础前提之一 |

## 7. 销售协议+四书

| 项目 | 内容 |
|---|---|
| 一句话解释 | 发行登记常见材料口径，四书为产品说明书、投资协议书、风险揭示书、投资者权益须知 |
| 业务场景 | 新产品发行、说明书修改、募集期或存续期修改 |
| 容易混淆点 | 不建议泛称“五书”，资料口径更准确的是销售协议+四书 |
| 面试表达 | 销售协议+四书是面向投资者和渠道的核心销售文件组合 |

## 8. 三书

| 项目 | 内容 |
|---|---|
| 一句话解释 | 三书通常指投资协议书、风险揭示书、投资者权益须知 |
| 业务场景 | 申报登记材料中常见“销售协议及三书、07说明书” |
| 容易混淆点 | 三书不含产品说明书，说明书常单独列示 |
| 面试表达 | 申报材料里看到三书，要和说明书、销售协议合起来理解为销售文件体系 |

## 9. 产品说明书

| 项目 | 内容 |
|---|---|
| 一句话解释 | 产品说明书是说明产品条款、投资范围、费用、估值、风险、信披等内容的核心文件 |
| 业务场景 | 申报、发行、销售文件修改、存续期变更 |
| 容易混淆点 | 说明书字段最密集，不只是对客户的介绍文本 |
| 面试表达 | 产品说明书是销售文件中最关键、最容易牵动系统要素和报送字段的文件 |

## 10. TA 要素表

| 项目 | 内容 |
|---|---|
| 一句话解释 | TA 要素表是给 TA 系统、运营和渠道使用的结构化产品字段集合 |
| 业务场景 | 新发、增发份额、渠道新增、额度调整、开放日或披露规则设置 |
| 容易混淆点 | 销售文件给人看，TA 要素给系统读 |
| 面试表达 | TA 要素决定产品能否被系统正确销售、交易、清算、披露和额度控制 |

## 11. 产品要素导出表

| 项目 | 内容 |
|---|---|
| 一句话解释 | 产品管理系统导出的结构化字段表，可用于生成或转换 TA 要素表 |
| 业务场景 | TA 接口上线前后、新发和变更 |
| 容易混淆点 | 它不是销售文件，而是系统字段底稿 |
| 面试表达 | 产品要素导出表体现了产品管理从文档化向要素化、数据化转变 |

## 12. 去纸化审批/OA 去纸化流程

| 项目 | 内容 |
|---|---|
| 一句话解释 | 去纸化审批是内部线上审批、运营处理和归档留痕流程 |
| 业务场景 | 申报、发行、TA 要素、销售文件修改、存续期要素变更 |
| 容易混淆点 | 不是微信沟通，也不是泛泛电子化，而是正式审批链路 |
| 面试表达 | 去纸化审批保证产品发行和存续期变更有流程、有节点、有留痕 |

## 13. 风险评级表

| 项目 | 内容 |
|---|---|
| 一句话解释 | 风险评级表可对应理财产品风险评估及评级意见审批单，用于形成产品风险等级 |
| 业务场景 | 产品申报登记前或风险等级调整 |
| 容易混淆点 | 不是主观填一个等级，而是结合资产、比例、估值和风险评分 |
| 面试表达 | 风险评级连接产品投资范围、销售适当性和风险揭示，是产品部必须关注的合规基础 |

## 14. 流动性风险评估

| 项目 | 内容 |
|---|---|
| 一句话解释 | 流动性风险评估主要用于开放式产品，评估产品应对申赎和流动性压力的能力 |
| 业务场景 | 开放式产品申报登记前 |
| 容易混淆点 | 封闭式产品通常不需要开放式流动性风险评估 |
| 面试表达 | 开放式产品持续面临申赎，流动性风险评估是产品设计和风险控制的重要前置工作 |

## 15. 业绩比较基准

| 项目 | 内容 |
|---|---|
| 一句话解释 | 业绩比较基准是基于投资范围、策略和市场环境测算的参考收益指标，不是预期收益承诺 |
| 业务场景 | 产品方案、说明书、发行登记、TA 要素、信披 |
| 容易混淆点 | 不是保本保收益，也不是产品实际收益 |
| 面试表达 | 业绩比较基准既影响销售展示，也可能影响浮动管理费规则和客户理解 |

## 16. 固定管理费

| 项目 | 内容 |
|---|---|
| 一句话解释 | 固定管理费是管理人按约定费率收取的管理费用 |
| 业务场景 | 产品说明书、系统费率模块、TA 要素 |
| 容易混淆点 | 多份额产品不同份额可能费率不同 |
| 面试表达 | 固定管理费字段要在说明书、系统和 TA 中保持一致 |

## 17. 销售服务费

| 项目 | 内容 |
|---|---|
| 一句话解释 | 销售服务费是与销售服务相关的费用，可能按资产净值计提 |
| 业务场景 | 费用条款、渠道销售、TA 设置 |
| 容易混淆点 | 不同渠道、份额可能涉及不同费用安排 |
| 面试表达 | 销售服务费既影响客户费用，也影响渠道和系统展示口径 |

## 18. 托管费

| 项目 | 内容 |
|---|---|
| 一句话解释 | 托管费是托管机构提供托管服务对应的费用 |
| 业务场景 | 说明书费用章节、系统费率设置 |
| 容易混淆点 | 通常不是产品部单独决定，但需在材料中准确体现 |
| 面试表达 | 托管费属于产品费用体系，需与说明书和系统计提规则一致 |

## 19. 浮动管理费

| 项目 | 内容 |
|---|---|
| 一句话解释 | 浮动管理费是与产品收益表现、业绩比较基准或阶梯规则挂钩的管理人费用 |
| 业务场景 | 费用条款、说明书、TA 要素、系统费率模块 |
| 容易混淆点 | 不是固定收取，计算规则可能按份额和区间区分 |
| 面试表达 | 浮动管理费是高风险字段，产品部要关注计提条件、计费基准、份额差异和披露口径 |

## 20. 合同费率与执行费率

| 项目 | 内容 |
|---|---|
| 一句话解释 | 合同费率是说明书约定费率，执行费率是优惠后的实际费率 |
| 业务场景 | 产品管理系统费率模块、销售经理费率优惠 |
| 容易混淆点 | 执行费率不得高于合同费率，生效日期不得早于产品成立日 |
| 面试表达 | 合同费率和执行费率体现了“法律文本口径”和“系统执行口径”的差异，产品部要管住一致性和权限边界 |

## 21. 母子份额/多份额

| 项目 | 内容 |
|---|---|
| 一句话解释 | 母子份额或多份额产品是在同一产品下设置多个份额类型，可能对应不同费率、渠道、业绩基准和销售代码 |
| 业务场景 | 公募母子份额、增发子份额、多渠道销售 |
| 容易混淆点 | 不能只按产品维度看，要按份额维度校验字段 |
| 面试表达 | 多份额产品会显著提高字段一致性管理难度，是产品部流程优化的重要场景 |

## 22. 存续期管理

| 项目 | 内容 |
|---|---|
| 一句话解释 | 存续期管理是产品成立后围绕要素变更、销售文件修改、信披、渠道、份额、到期终止等事项的持续管理 |
| 业务场景 | 开放式产品日常运营、多份额增发、渠道新增、说明书修改 |
| 容易混淆点 | 产品发行结束不代表产品部工作结束 |
| 面试表达 | 产品部的价值贯穿全生命周期，尤其开放式和多份额产品更考验存续期管理能力 |

## 23. 投资库

| 项目 | 内容 |
|---|---|
| 一句话解释 | 投资库是产品可投资资产范围和风险监控边界的重要管理基础 |
| 业务场景 | 新产品发行、主题产品发行、投资范围调整、投决会和风险合规确认 |
| 容易混淆点 | 它不是产品说明书里的泛泛投资范围，而是发行后投资运作和风控监控要能承接 |
| 面试表达 | 产品拓展要前置确认投资库能否承接，尤其主题产品要关注主题投资库和风控指标落地 |

## 24. 主题型投资库

| 项目 | 内容 |
|---|---|
| 一句话解释 | 主题型投资库是在常规投资库基础上，围绕特定主题策略形成的可投资资产集合 |
| 业务场景 | 碳中和、红利、长三角等主题策略产品发行 |
| 容易混淆点 | 主题产品不是只改产品名称或营销口径，还要有对应投资库和风控指标支持 |
| 面试表达 | 主题策略产品化时，产品部要同步推动主题投资库、产品方案、投决会和销售文件口径一致 |

## 25. 风控指标

| 项目 | 内容 |
|---|---|
| 一句话解释 | 风控指标是对投资库比例、计算规则、监控起点和超限处置等风险边界的约束 |
| 业务场景 | 新产品发行前风险合规确认、投决会审议、发行去纸化审批 |
| 容易混淆点 | 不是发行后再看，而是申报和发行排期中需要前置确认 |
| 面试表达 | 产品部要把风控指标作为产品落地条件管理，避免产品能设计但发行后无法稳定监控 |

## 26. 理财登数据元

| 项目 | 内容 |
|---|---|
| 一句话解释 | 理财登数据元是理财登记系统要求报送的结构化字段标准 |
| 业务场景 | 申报登记、发行登记、募集期总量信息、产品要素修改 |
| 容易混淆点 | 理财登不是只上传文件，核心是按数据元规范准确报送字段 |
| 面试表达 | 产品部工作从本质上是“材料+数据元”双线管理，要保证说明书、系统字段和监管报送一致 |

## 27. 申报要素修改

| 项目 | 内容 |
|---|---|
| 一句话解释 | 申报要素修改是对已申报产品的监管申报字段或相关说明书口径进行调整 |
| 业务场景 | 产品基础要素、说明书、申报字段需要修改 |
| 容易混淆点 | 不等同于发行要素修改，先判断影响的是申报字段还是本次发行字段 |
| 面试表达 | 申报要素修改要确保监管报送、说明书修订、用印和运营同步形成闭环 |

## 28. 发行要素修改

| 项目 | 内容 |
|---|---|
| 一句话解释 | 发行要素修改是对募集、成立、渠道、份额、费率等发行登记字段进行调整 |
| 业务场景 | 募集期调整、渠道增删、费率优惠、份额信息调整 |
| 容易混淆点 | 只改系统字段不够，还可能影响销售文件、TA、信披和渠道 |
| 面试表达 | 发行要素修改要从影响范围出发，检查销售文件、TA、渠道、理财登和公告是否都要同步 |

## 29. 销售文件更新时间

| 项目 | 内容 |
|---|---|
| 一句话解释 | 销售文件更新时间是系统和运营识别销售文件版本更新、生效和报送下载的重要时间字段 |
| 业务场景 | 销售文件修改、临时公告、信披接口、运营下载 |
| 容易混淆点 | 不是随便填一个日期，它会影响文件版本和下游处理节奏 |
| 面试表达 | 存续期文件修改要关注销售文件更新时间、流程结束时间和接口传输时间，保证版本按时生效 |

## 30. 报备人员信息

| 项目 | 内容 |
|---|---|
| 一句话解释 | 报备人员信息是产品申报和理财登报送中涉及产品审批人、设计人、投资经理、联系人等人员信息的基础数据 |
| 业务场景 | 定投资经理、申报登记、产品管理系统导入校验 |
| 容易混淆点 | 不是临时填写的联系人信息，缺失或未维护会导致系统导入失败 |
| 面试表达 | 报备人员信息体现了产品管理系统主数据治理的重要性，人员基础数据不准会卡住申报链路 |

## 31. 销售商额度/发行规模上限

| 项目 | 内容 |
|---|---|
| 一句话解释 | 销售商额度和发行规模上限用于控制不同渠道或份额的销售规模边界 |
| 业务场景 | 发行登记、TA 要素、渠道上架、多份额产品 |
| 容易混淆点 | 销售商额度模块和份额发行规模上限在系统中的承接程度可能不同，需以当前系统为准 |
| 面试表达 | 渠道和份额额度是产品上架的关键控制点，系统化时要把总额、份额上限和渠道额度校验打通 |

## 32. 四类存续期流程

| 项目 | 内容 |
|---|---|
| 一句话解释 | 产品管理系统存续期常见流程包括销售文件管理流程、要素变更流程、信披公告管理流程、估值条款涉及变更流程 |
| 业务场景 | 销售文件修改、要素调整、临时公告、估值条款变更 |
| 容易混淆点 | 不是所有变更都走同一个流程，要先判断是否改四书、是否公告、是否涉及估值条款 |
| 面试表达 | 存续期管理的关键是先选对流程，再控制版本、公告、TA、理财登和渠道的一致性 |
