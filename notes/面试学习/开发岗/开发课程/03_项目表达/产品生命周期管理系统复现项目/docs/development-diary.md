# PLM 复现项目开发过程记录

这份文档用来记录我们是如何一步一步推进这个项目的。它的作用不是写流水账，而是帮助你回忆当时做开发时应该经历过的思考过程。

## 第一步：先读建设背景，不急着写代码

我们先读了 `PLM系统建设说明-20240402.docx`。

读完后先确认系统不是一个普通产品信息管理页面，而是为了解决产品生命周期中的重大事件缺乏监控、批量产品信息收集困难、跨部门协作遗漏风险这三类问题。

这一步的开发意义是：

```text
先确定系统为什么存在，再决定代码要复现什么。
```

如果跳过这一步，容易把 PLM 错讲成“产品 CRUD 系统”。但真实项目更重要的是重大事件监控和提醒。

## 第二步：再读两个具体功能需求

我们继续读了两个功能文档：

```text
PLM应用-分红功能需求规格说明书20240412.docx
产品生命周期管理系统-目标盈支持功能-需求说明书_1.3.docx
```

读完后发现这两个功能代表了 PLM 的两类典型能力：

```text
分红功能：日期计算 + 方案监控 + 提醒
目标盈功能：收益率监控 + 连续达标判断 + 提醒
```

这一步的开发意义是：

```text
从需求文档里提炼系统能力，而不是照着页面字段机械写表。
```

## 第三步：抽出系统边界

我们明确了：

```text
PLM 负责录入附加要素、计算规则、跑批提醒、日志追踪。
TA / 数据中台负责提供产品基础信息、工作日历、行情、分红方案。
投管 / 邮件系统负责接收提醒。
```

这一步很重要，因为面试官可能会问：

```text
这些产品数据是 PLM 自己维护的吗？
PLM 是否直接修改 TA？
提醒是怎么送到投管的？
```

正确回答应该是：

```text
PLM 不替代 TA，而是在 TA/数据中台数据基础上补充业务附加要素，并做生命周期事件监控和提醒。
```

## 第四步：拆模块和动作

我们把系统拆成产品基础数据、工作日历、行情、分红要素、分红日期计算、分红提醒、目标盈要素、目标盈监控、文件接口、日志报告等模块。

这一步对应开发里的架构设计。

模块拆清楚后，后面的代码就不会糊在一起：

```text
日期计算归日期计算。
提醒筛选归跑批。
CSV 文件输出归文件接口。
日志归日志模块。
```

## 第五步：确定本地复现边界

本地没有真实 TA、数据中台、投管和邮件系统，所以我们采用模拟：

```text
H2 数据库模拟 PLM/TA 相关表
CSV 文件模拟数据中台输入和投管提醒输出
CLI 命令模拟每日跑批
Markdown 报告模拟跑批结果归档
```

这一步的意义是：

```text
复现项目要能讲清真实系统，又不能假装接了不存在的外部系统。
```

## 第六步：把需求里的规则翻译成代码模块

我们先实现了三个规则类：

```text
WorkdayService
  负责产品工作日判断和工作日偏移。

DividendScheduleCalculator
  负责根据成立日、首次分红间隔、分红频率、测算前提醒天数、测算后监控天数计算分红日期表。

TargetYieldMonitor
  负责判断目标盈产品今日是否达标、是否连续 N 个工作日达标、是否需要止盈提醒。
```

这一步对应真实开发里的“把需求文档里的自然语言规则变成可测试的代码”。

比如文档里说“当前系统日期-N+1（工作日）大于等于业绩观察期起始日期，且最新连续 N 个不同发布日期成立以来年化收益率都大于等于目标止盈收益率”，代码里就要拆成：

```text
1. 找到当前产品的工作日历
2. 从当前批次日向前取 N 个工作日
3. 读取这 N 天的行情年化收益率
4. 每一天都和目标止盈收益率比较
5. 判断最早观察日是否已经进入观察期
6. 满足后生成提醒
```

## 第七步：第一次运行时遇到 SQL 方言问题

本地 H2 开启了 Oracle compatibility mode，用来贴近真实项目里的 Oracle 环境。

第一次运行目标盈监控时，我在 SQL 里写了：

```sql
ORDER BY confirm_date DESC
LIMIT ?
```

但 H2 的 Oracle 兼容模式不支持这种 MySQL/PostgreSQL 风格的 `LIMIT ?` 写法，导致跑批失败。

这个问题的复盘价值很高：

```text
本地 demo 使用 H2 是为了方便运行，但真实项目用 Oracle。
使用 H2 模拟 Oracle 时，不能混用不同数据库方言。
跨数据库开发时，要么坚持标准 SQL，要么把分页/截取逻辑放到 Java 层。
```

最终我把 SQL 改成查询后在 Java 里 `stream().limit(limit)` 截取最近 N 条。

这不是最适合大数据量的生产实现，但对本地复现项目足够清楚，也避免了数据库方言干扰主线。

## 第八步：跑通第一版本地批处理

第一版本地批处理命令是：

```bash
mvn clean package
./scripts/prepare_env.sh
./scripts/seed_demo_data.sh
./scripts/run_daily_batch.sh 20260506
```

它做了这些事：

```text
1. 初始化产品基础数据
2. 初始化产品工作日历
3. 初始化分红附加要素
4. 初始化目标盈附加要素
5. 初始化目标盈行情数据
6. 计算分红日期表
7. 筛选 2026-05-06 当日提醒
8. 生成投管提醒 CSV
9. 生成 Markdown 批处理报告
```

本次样例生成了两条提醒：

```text
DIVIDEND_SCHEMA_MISSING
  定期分红产品已经过测算日，但没有发现 TA 分红方案。

TARGET_YIELD_TAKE_PROFIT
  目标盈产品连续 3 个工作日成立以来年化收益率不低于目标止盈收益率。
```

这说明第一版已经能讲清 PLM 的核心能力：

```text
不是简单维护产品资料，而是基于产品生命周期数据做重大事件监控和提醒。
```

## 第九步：重新纠正业务模块边界

第一版为了尽快跑通，把每日批处理做成了一个统一流程：

```text
PLM Daily Batch
  ├─ 计算分红提醒
  └─ 计算目标盈提醒
```

这个做法在代码上能跑，但在业务表达上不够好。因为定期分红和目标盈不是同一个业务，只是都属于产品生命周期管理系统中的重大事件监控。

所以我们重新调整了表达和代码结构：

```text
定期分红业务线
  ├─ DividendScheduleCalculator
  ├─ DividendReminderService
  └─ DividendBatchWorkflow

目标盈业务线
  ├─ TargetYieldMonitor
  └─ TargetYieldBatchWorkflow

公共总控
  └─ PlmDailyBatchWorkflow
```

这次调整的开发意义是：

```text
业务模块要按业务域拆分，而不是按“都在每日跑批里执行”混在一起。
```

面试时也应该这样讲：

```text
PLM 是一个产品生命周期重大事件管理平台。定期分红和目标盈是两个独立业务场景，它们分别有自己的录入要素、监控规则和提醒逻辑。它们共用产品基础信息、工作日历、行情数据、文件输出和日志这些公共支撑能力。
```

这也是一个真实开发细节：第一版设计经常会先从“能跑通”出发，之后要根据业务边界做模块重构。这个重构本身很有价值，因为它让代码结构更贴合业务语言。
