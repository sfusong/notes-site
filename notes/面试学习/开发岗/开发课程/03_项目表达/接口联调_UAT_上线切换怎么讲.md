# 接口联调、UAT、上线切换：怎么讲

一句话定位：

`我熟悉金融系统从需求确认、接口联调、UAT 到上线切换的真实交付链路，知道功能写完以后如何安全落到业务、供应商和外部机构协作场景里。`

适用经历：TA 系统对接、理财登接口升级切换、托管行/代销机构接入、产品管理系统供应商交付。

## 三个概念

| 概念 | 核心 | 金融项目里的重点 |
| --- | --- | --- |
| 接口联调 | 两个系统按约定真正跑通 | 字段口径、枚举、文件格式、批处理时点、异常返回 |
| UAT | 业务方验收系统是否符合真实业务口径 | 场景覆盖、流程结果、边界情况、业务签字确认 |
| 上线切换 | 从旧版本/旧接口/旧流程平稳切到新链路 | 上线窗口、影响范围、回滚预案、新旧数据衔接 |

## 接口联调怎么讲

“接口联调不只是接口能访问，而是业务链路能跑通。金融系统里常见问题不是 HTTP 通不通，而是字段口径不一致、枚举值不一致、文件交互时点不一致，或者异常场景没有定义清楚。”

你可以讲的具体动作：

- 对齐字段名称、类型、长度、是否必填、枚举值。
- 确认文件交互方式、生成时点、读取时点和重传机制。
- 准备联调数据和典型场景。
- 记录问题，区分是我方改、对方改还是做映射转换。
- 联调通过后沉淀接口文档和上线检查项。

## UAT 怎么讲

“UAT 是业务验收，不是开发自测的延伸。它关注的是业务在真实流程里能不能接受系统结果，比如字段展示是否符合口径、审批流程是否正确、异常场景有没有覆盖。”

你可以讲的具体动作：

- 和业务确认核心流程和边界场景。
- 把业务流程转成测试案例。
- 组织业务、测试、供应商或开发一起执行。
- 缺陷分级：必须本版本修、可接受、后续优化。
- 形成测试报告、会议纪要和上线前置材料。

## 上线切换怎么讲

“上线切换比普通发布更强调风险控制。特别是金融系统涉及外部机构、接口升级和批处理链路时，需要明确切换窗口、影响范围、验证步骤和回滚方案。”

你可以讲的具体动作：

- 明确上线窗口和业务低峰时点。
- 梳理新旧接口或新旧流程差异。
- 准备切换步骤、验证脚本、联系人和回滚路径。
- 上线后验证关键交易、文件、批处理和报表。
- 保留观察期，跟踪异常并快速收口。

## 常见追问

| 问题 | 答法 |
| --- | --- |
| 字段口径不一致怎么办？ | 先确认下游真实需求，再对齐字段含义、格式、枚举和时点，评估由上游改、下游改还是中间做映射转换。 |
| UAT 发现问题怎么处理？ | 先判断是否影响上线核心流程；影响主流程的必须本版本修，不影响主流程但业务可接受的记录为后续优化。 |
| 上线最怕什么？ | 影响范围不清、回滚路径不清、上线后验证不足、外部机构协同不到位。 |
| 金融项目联调难在哪里？ | 难在业务口径、批处理时点、文件交互、外部机构协作和上线窗口都要同时对齐。 |

## 1 分钟完整版

“我理解接口联调、UAT 和上线切换是金融系统交付里非常关键的一条链路。接口联调不只是连通接口，而是要把字段口径、枚举、文件交互、批处理时点和异常处理真正跑通；UAT 是让业务方按真实流程验收系统结果，不只是技术测试；上线切换则更强调上线窗口、影响范围、验证步骤和回滚预案。结合我在 TA 对接、理财登升级和产品管理系统里的经历，我比较熟悉如何在业务、供应商、测试和外部机构之间推进问题收口，让系统能力安全落地。”
