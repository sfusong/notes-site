# Apprentice（学徒）面试资料

## 1. 30 秒项目介绍

Apprentice 是一个面向企业系统的、配置驱动的 AI 操作层。  
它不是简单的聊天机器人，而是想验证一条完整闭环：

- 观察员工在业务系统中的真实操作
- 生成可复用的员工画像
- 理解自然语言请求
- 补齐缺失参数
- 在受控边界内执行真实后端操作

目前这个项目用 `FastAPI + React/Vite + YAML 配置 + Spring Boot 参考业务系统` 搭了一个可运行原型。

## 2. 可确认事实

来自项目文档可确认的点：

- Agent backend: `FastAPI`
- Frontend: `React + Vite`
- 配置层：`YAML`
- 执行边界：`Executor`
- 画像存储：`JSON`
- LLM 接口：`OpenRouter / OpenAI-compatible`
- 参考业务系统：`Spring Boot + Spring Security + Spring Data JPA`
- 数据：`H2`
- 审计日志：`AOP-based audit logging`

## 3. 组件和技术解释

### FastAPI

FastAPI 是 Python Web 框架，适合做：
- API 服务
- JSON 输入输出
- Pydantic 数据校验
- 快速原型

在这个项目里它主要承担：
- Agent API
- Chat orchestration
- 配置加载
- 执行调度
- 画像刷新

### React + Vite

React 是前端框架，Vite 是前端构建工具。  
在这里主要负责：
- 登录界面
- 聊天工作台
- 员工画像展示
- 建议参数按钮
- 写操作确认

### YAML

YAML 在这里不是普通配置文件，而是这个项目的核心设计之一。  
它用来定义：
- 系统身份
- 允许的操作
- 字段名称和类型
- 必填项
- 语义提示

你可以把它理解为：

`把系统操作能力抽象成一层“可配置的操作描述”，而不是把所有逻辑写死在 prompt 里。`

### Executor

Executor 是模型和真实业务系统之间的隔离边界。  
作用包括：
- 参数校验
- 解析目标接口
- 组装请求头
- 获取或刷新 JWT
- 执行真实 HTTP 请求

你要记住一句话：

`模型不能直接随意发 HTTP 请求，必须经过受控执行层。`

### Profile / 员工画像

画像层来源于真实审计日志，不是硬编码偏好。  
会提取：
- 常用操作
- 活跃时间
- 常见字段值
- 可读提示

### Spring Security

主要做：
- 登录认证
- 接口访问保护
- JWT 相关处理

### Spring Data JPA

主要做：
- 实体到数据库表的映射
- 常规数据访问
- 降低样板代码量

### AOP 审计日志

主要用于：
- 记录业务操作
- 供画像学习层读取
- 做可追溯性支持

你可以说：

“只要涉及真实操作，审计和可追踪性就很重要，所以我在参考业务系统里加了审计日志能力。”

## 4. 架构理解

这是项目文档里已经明确给出的主架构逻辑：

```text
Employee
  -> React Frontend
  -> FastAPI Agent API
     -> YAML Config
     -> Profile Store (JSON)
     -> LLM
     -> Executor
        -> Spring Boot Business System
           -> H2 Business Data + Audit Logs
```

如果面试官问“你设计这个架构的核心思想是什么”，你可以答：

核心思想是把 AI 放在一个受控的企业系统操作框架里，而不是让模型自由生成请求。也就是：
- 配置层描述允许的能力
- 编排层负责理解意图和补全参数
- 执行层负责安全调用
- 画像层负责提升后续建议质量

## 5. 面试官最可能问的问题

### Q1. Apprentice 和普通聊天机器人有什么不同？

回答：

它不是只做问答，而是把自然语言请求连接到真实企业系统操作。重点是“理解请求并执行受控操作”，而不是生成一段文本答案。

### Q2. 为什么要用 YAML 配置驱动？

回答：

因为我不希望每个业务操作都写死在 prompt 里。配置驱动意味着可以把系统能力、字段、语义和约束抽出来，让 Agent 能在更通用的框架里运行，也更方便迁移到第二个目标系统。

### Q3. 为什么需要 executor boundary？

回答：

因为模型本身不应该直接控制真实系统。Executor 是一个受控执行层，可以做参数校验、权限控制、token 管理和错误处理，这样系统才有工程上的可控性。

### Q4. 为什么做 profile / 员工画像？

回答：

因为很多企业系统操作是有重复模式的，比如某个员工高频做哪些操作、常填哪些字段、通常在什么时间段活跃。把这些信息结构化之后，可以帮助参数补全和操作建议更贴近真实使用习惯。

### Q5. 这个项目最大的技术难点是什么？

回答：

我觉得最大的难点不是前后端技术栈本身，而是如何在“模型能力”和“企业系统可控性”之间找平衡。也就是说，既要让 AI 有一定操作能力，又不能让它脱离边界，这就需要配置层、编排层、执行层和审计层一起工作。

### Q6. 这个项目为什么要有 AOP 审计日志？

回答：

因为我的目标不是只做一个 demo，而是要让系统未来具备可学习和可追踪能力。审计日志既能支持画像学习，也能支持后续排查和操作回溯。

### Q7. 为什么不用直接把后端接口描述塞给模型？

回答：

因为那样控制力太弱。我的设计更强调让模型只在“配置允许的操作集合”里选择，既减少错误，也提高安全性和稳定性。

### Q8. 这个项目现在做到什么程度？

回答：

目前是一个可以运行的原型，已经验证了登录、画像刷新、自然语言请求、参数补全、受控执行和结果摘要这条完整链路。重点是先验证产品价值和架构可行性，而不是一下子覆盖很多业务系统。
