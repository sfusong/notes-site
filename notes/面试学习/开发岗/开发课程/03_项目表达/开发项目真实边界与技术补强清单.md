# 开发项目真实边界与技术补强清单

这份文档用于控制面试风险：哪些项目可以强讲开发，哪些项目要讲交付和系统理解，哪些技术细节必须补到能追问。

统一定位：

`我不是传统纯后端工程师，也不是只做需求和交付。我在金融机构系统建设里承担过需求、联调、UAT、上线，也参与过部分后端、工具和自动化实现。我的优势是能把金融业务规则翻译成系统能力。`

## 项目边界总表

| 项目 | 最稳定位 | 可强讲 | 不要讲成 |
| --- | --- | --- | --- |
| 产品管理系统 | 供应商采购的正式统一平台 | 需求收敛、字段口径、供应商协作、UAT、上线 | 你主开发的系统 |
| 产品生命周期管理系统 | 正式平台上线前的自研辅助系统 | Spring Boot 后端、数据库、提醒规则、邮件/OA 对接 | 你独立完成全栈 |
| 说明书自动生成工具 | Python GUI 轻量工具 | 模板填充、字段映射、文件生成、快速落地 | 大型平台项目 |
| TA 自动化测试 | 申请/确认闭环自动化回归 | Jenkins、文件解析、数据库比对、批处理链路 | 完整测试平台主架构师 |
| TA 批前预警稽核 | 批前规则稽核和风险前置 | 规则口径、静态参数稽核、结果验证 | 全平台唯一开发 |
| TA 对接/理财登切换 | 金融系统联调和升级切换 | 字段口径、接口联调、上线窗口、问题收口 | 纯编码项目 |
| 广发授信变动查询 | 中后台查询功能开发 | 表结构、历史回填、查询、导出 | 复杂分布式系统 |

## 产品管理系统

### 可以讲

- 供应商采购的正式统一平台，目标是产品要素统一管理、销售文件生成、流程审批和下游集成。
- 你参与需求范围收敛、字段口径梳理、供应商调研、路演、POC/选型配合。
- 你牵头或参与 UAT、试运行、上线支持和业务培训。
- 审核供应商需求文档时看业务边界、字段口径、流程节点、角色权限、接口依赖和测试上线约束。

### 不要讲

- 不要说“这套系统前后端是我写的”。
- 不要把供应商弱化成执行人员。

### 面试安全话术

“产品管理系统是采购的正式长期平台，我不是底层开发角色，更多负责内部需求收口、字段和流程口径确认，以及和供应商、测试、业务共同推进 UAT、试运行和上线。”

## 产品生命周期管理系统

### 可以强讲

- 自研前后端系统，承接正式产品管理系统上线前的紧急业务需求。
- 后端 Spring Boot，前端基于内部 React 低代码/组件化平台。
- 你参与后端规则判断、数据库设计、数据中台联调、提醒结果落库、邮件/OA 接口对接。
- 目标盈提醒、分红提醒、到期提醒是最适合展开的功能。

### 必须补熟

| 追问 | 必答点 |
| --- | --- |
| 表怎么设计？ | 产品基础要素、个性化规则、工作日历、提醒记录/状态 |
| 目标盈怎么提醒？ | 读取产品参数和收益/净值数据，判断是否触发，落库并通知 |
| 为什么不用产品管理系统？ | 正式平台未上线，业务需求急，自研系统先承接过渡场景 |
| React 做了什么？ | 参与页面接口联动和展示，不包装成主前端 |

### 面试安全话术

“这条是我最适合讲开发参与度的项目。我主线在后端、数据库、规则和接口对接，前端有参与但不是唯一负责人。”

## 说明书自动生成工具

### 可以讲

- Python GUI 工具，面向业务同事快速生成说明书/销售文件。
- 核心是读取产品字段、映射模板占位符、填充生成文档，并做基础校验。
- 它是正式产品管理系统上线前的过渡工具，解决当时新产品发行效率问题。

### 不要讲

- 不要讲成复杂 Web 平台。
- 不要占太多篇幅，除非面试官追问 Python 或工具开发。

### 面试安全话术

“这条工具实现度比较高，但复杂度不如后面的系统项目。我会把它作为能快速把业务需求落成工具的补充案例。”

## TA 自动化测试

### 可以强讲

- 自动化对象是 TA 和代销之间的申请/确认闭环，不是 UI 自动化。
- 你参与自动化思路、技术选型、确认文件解析、数据库比对。
- 关键链路是：申请文件准备、SQL 数据准备、Jenkins 触发批处理、确认文件生成、文件解析、数据库比对、报告输出。

### 必须补熟

| 追问 | 必答点 |
| --- | --- |
| Jenkins 做什么？ | 串联环境准备、批处理触发、脚本执行、结果报告 |
| 比对哪些字段？ | 确认状态、交易账号、产品代码、份额、金额、日期、状态码 |
| 结果不一致怎么查？ | 先工具解析，再测试数据/环境，最后系统逻辑 |
| 文件解析难点？ | 定长/分隔符、字段类型、金额精度、日期格式、空值、状态码映射 |

### 面试安全话术

“我不是整套平台唯一主开发，但确认文件解析和数据库比对这类关键校验逻辑我参与实现过，所以我能讲清楚自动化闭环。”

## TA 批前预警稽核

### 可以讲

- 包含两类能力：恒生 AOP 的交易类批前预警，团队自研的静态参数稽核。
- 交易类预警看当前份额、新申购规模、募集规模、销售额度、持有限额等。
- 静态参数稽核看托管账号、销售商清算账号、费率、开放期、日期、产品状态等。
- 你参与需求讨论、规则清单、口径定义、系统化检查逻辑和验证。

### 必须补熟

| 追问 | 必答点 |
| --- | --- |
| 和监控区别？ | 监控看系统运行，稽核看业务规则能否安全进入跑批 |
| 为什么自研？ | 静态参数规则依赖内部业务口径，厂商标准能力不一定覆盖 |
| 误报怎么办？ | 查规则口径、数据质量、业务例外 |
| 漏报怎么办？ | 查数据源完整性、适用范围、结果展示过滤 |

### 面试安全话术

“这条项目我不会包装成自己独立开发完整平台。我更强调我参与了规则口径和自研静态参数稽核，把人工检查沉淀成系统规则。”

## TA 对接 / 理财登升级切换

### 可以讲

- 支持托管行、代销机构、理财登等外部接口升级和切换。
- 你做字段口径确认、接口规格梳理、联调推进、异常收口、上线窗口配合。
- 适合证明你熟悉金融机构真实系统交付，而不是只会本地写代码。

### 必须补熟

接口口径不一致时按四步答：

1. 收清下游真实需求。
2. 对齐字段来源、格式、枚举、时点。
3. 评估映射转换、上游改造还是下游适配。
4. 沉淀接口文档，联调验证并安排上线切换。

## 广发授信变动查询

### 可以讲

- 典型中后台查询功能开发。
- 你参与查询口径、表结构变更、历史数据回填、列表查询、统计和 CSV 导出。

### 必须补熟

- 查询条件如何映射 SQL。
- 为什么需要加索引或控制导出范围。
- 历史数据回填如何校验。
- CSV 导出如何处理字段、编码、金额和日期格式。

## 红线

面试里不要说：

- “这个项目就是我一个人写的，只是时间久忘了。”
- “同事写的，但我现在可以当成我写的讲。”
- “这块技术我没做过但应该差不多。”

可以说：

- “这部分我不是唯一负责人，但我参与了关键实现/验证。”
- “我主线在后端规则和接口联动，前端参与较少。”
- “这部分由供应商实现，我主要负责需求口径和验收，但我理解它的系统链路。”

## 最后收口

“我会把项目讲得真实一些：产品管理系统是供应商正式平台，我讲交付和系统理解；生命周期系统、TA 静态参数稽核、TA 自动化测试是我重点讲开发参与度的三条；说明书工具和广发查询作为补充。这样既不把边界讲过头，也能让面试官看到我确实具备开发岗需要的系统理解、编码参与和金融业务落地经验。”
