# 产品生命周期管理系统：高频追问

## 1. 这个系统解决了什么问题？

答：

`它解决的是特殊理财产品生命周期节点依赖人工记忆和线下协同的问题。目标盈、分红封闭式、定期分红这类产品在存续期会触发收益达标、分红准备、到期处理等事件，系统把这些规则转成可计算、可展示、可提醒的事件能力。`

关键词：

- 规则系统化
- 事件管理
- 协同提醒
- 降低漏节点风险

## 2. 为什么要自研？

答：

`当时正式产品管理系统还没有完全上线，但业务已经有特殊产品节点管理需求，所以先做一套范围更聚焦、上线更快的自研辅助系统。它不是替代正式平台，而是在正式平台上线前承接真实业务痛点。`

## 3. 你具体开发了什么？

答：

`我主做后端这一侧，包括业务规则判断、数据库设计、数据中台联调、提醒结果落库和邮件/OA 提醒接口；前端我也参与了接口联动和部分页面交互，但不是唯一前端负责人。`

如果继续追：

- 规则判断：目标盈、分红、到期等事件触发
- 数据库：基础要素、个性化参数、工作日历、事件、提醒记录
- 联调：数据中台、邮件、OA
- 前端：列表展示、状态展示、接口联动

## 4. 后端为什么用 Spring Boot？

答：

`Spring Boot 适合承接后端规则和接口服务。这个系统的核心不是页面，而是把产品类型、日期、收益阈值、分红计划这些规则统一放在后端判断，再通过 API 给前端展示和操作。`

## 5. React 在这里做什么？

答：

`React 主要承接业务交互：产品查询、事件列表、提醒状态、处理入口、表单展示，以及和后端 REST API 联动。它不是单纯做页面样式，而是让业务同事能看到和处理系统生成的生命周期事件。`

## 6. 表结构怎么设计？

答：

`我会按业务职责拆表：产品基础要素表存通用信息，产品个性化参数表存目标收益、分红规则这类特殊配置，工作日历表处理工作日和节假日，事件表记录触发出来的生命周期事件，提醒记录表记录发送渠道、接收人和发送状态。`

为什么这样拆：

- 基础要素和特殊规则分离，便于扩展产品类型
- 事件和提醒分离，便于追踪事件是否触发、提醒是否发送
- 工作日历独立，避免日期规则散落在代码里

## 7. 目标盈提醒怎么实现？

答：

`后端先读取产品基础信息和目标收益阈值，再从数据中台获取收益或净值数据，判断是否达到提醒条件。满足条件后生成生命周期事件，落提醒记录，并通过页面、邮件、OA 推送给相关业务角色。`

排查提醒错误：

1. 规则口径是否正确
2. 数据中台数据是否完整、最新
3. 产品状态是否满足触发条件
4. 边界值和比较符号是否写对
5. 提醒记录是否重复或漏发

## 8. 这个系统和产品管理系统是什么关系？

答：

`生命周期系统是自研辅助系统，聚焦特殊产品关键节点提醒。产品管理系统是供应商采购的正式统一平台，覆盖产品要素、说明书生成、审批流和下游集成。两者不是一个东西，生命周期系统是正式平台上线前的先行承接。`

## 9. 最大难点是什么？

答：

`最大难点不是页面开发，而是规则口径收敛。目标盈、分红、到期这类事件往往不是单一条件，要结合产品类型、日期、收益、状态、工作日历和业务口径。只有先把规则边界收清，后端逻辑才稳定。`

## 11. 一分钟收口版

`这条项目的核心是把特殊理财产品生命周期里的关键节点规则系统化。它不是正式产品管理系统，而是正式平台上线前的自研辅助系统。后端 Spring Boot 承接规则判断、事件生成、数据中台联调和提醒接口，前端 React 承接事件展示和状态交互。我主要参与后端规则、数据库设计、数据联调和邮件/OA 提醒。它最能体现我把金融业务规则落成系统能力的开发经历。`
