## 一句话定位

`React 不是你的主战场，但它能证明你参与过前后端分离项目里的页面交互和接口联动。`

主战项目是产品生命周期管理系统：后端 Spring Boot 负责规则、数据和接口，前端 React/内部低代码平台负责录入、查询、状态展示和提醒结果呈现。

## 30 秒讲法

“我不是专职前端，但在产品生命周期管理系统里参与过 React 相关页面和接口联动。React 在这个项目里主要承担产品信息录入、生命周期事件查看、提醒状态展示这些交互，后端通过 REST API 提供产品、事件、提醒记录等数据。我的主线还是后端规则和接口，但我理解前端如何通过组件化和状态驱动把后端处理结果变成业务可用页面。”

## 在产品生命周期系统里的分工

| 层次 | React 做什么 | 后端做什么 |
| --- | --- | --- |
| 表单录入 | 产品参数、提醒规则、筛选条件输入 | 参数校验、规则保存、数据落库 |
| 列表展示 | 生命周期事件、提醒记录、处理状态 | 查询数据、分页、状态计算 |
| 状态变化 | 已提醒、已处理、待确认等页面状态 | 状态流转、权限和业务规则 |
| 接口联动 | 调用 REST API，按返回结果刷新页面 | 提供查询、提交、更新、通知接口 |

## 基础问题怎么答

| 问题 | 答案 |
| --- | --- |
| React 核心是什么？ | 组件化和状态驱动。组件化让页面按业务模块拆分，状态驱动让页面随数据变化更新。 |
| props 和 state 区别？ | props 是外部传入组件的数据，state 是组件内部维护并驱动展示变化的数据。 |
| React 怎么和后端配合？ | 前端负责页面展示和用户交互，通过 REST API 查询或提交数据；后端负责规则、校验、数据处理和持久化。 |
| 为什么适合业务系统？ | 表单、列表、详情、状态展示很多，组件化和状态驱动更容易组织页面。 |
| 你是不是主要做前端？ | 不是，我定位偏后端/全栈参与，React 这块主要能讲清页面和接口联动。 |

## 最容易被追问的细节

### 页面数据怎么刷新？

“用户进入页面或切换筛选条件时，前端调用后端查询接口，拿到产品事件和提醒状态后更新页面状态。提交处理动作后，再调用更新接口，并重新拉取或局部更新列表。”

### 表单怎么校验？

“基础格式类校验可以在前端先做，比如必填、日期格式、数值范围；真正涉及业务口径的校验还是放在后端，比如产品状态是否允许修改、提醒规则是否符合产品类型。”

### 为什么业务校验不能只放前端？

“前端校验主要提升交互体验，不能作为最终规则来源。后端才是权威校验位置，否则绕过页面或接口重复调用时会有数据风险。”

## 1 分钟完整版

“React 对我来说不是单独的主战场，而是我在前后端分离项目里参与页面交互和接口联动的能力。比如产品生命周期管理系统里，后端 Spring Boot 负责生命周期规则、事件逻辑、提醒状态和数据接口，前端 React 则负责产品信息录入、事件查看、提醒结果展示和状态交互。我参与过这类页面和接口联动，能理解 props、state、组件化、REST API 调用这些基础。我的定位不是资深前端，而是偏后端/全栈参与，能把后端规则处理后的结果落到业务可用页面上。”
