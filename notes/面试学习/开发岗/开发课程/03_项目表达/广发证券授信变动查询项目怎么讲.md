# 广发证券授信变动查询项目怎么讲

一句话定位：

`典型金融中后台查询功能开发，重点是查询口径、表结构变更、历史数据回填、统计和 CSV 导出。`

这条不是主战项目，但适合证明你早期做过后端和数据库相关开发。

## 30 秒讲法

“广发证券这条经历是客户授信管理系统里的授信变动查询功能。我负责功能设计和后端开发，核心不是简单做一个查询页，而是先梳理风控人员需要看的授信变动口径，比如客户、调整前后授信值、变动时间、变动原因，再基于 PostgreSQL 做表结构变更和历史数据回填，最后支持列表查询、统计和 CSV 导出。这条项目比较能体现中后台开发基本功：业务查询口径、数据库结构和历史数据兼容要一起考虑。”

## 解决的问题

风控人员需要统一查看客户授信变化：

- 谁的授信发生变化。
- 什么时候变化。
- 调整前后值是多少。
- 调整原因和操作来源是什么。
- 是否能统计、导出、留档或进一步分析。

如果没有统一入口，这类信息容易散落在业务流程、日志或人工记录里，回溯成本高。

## 你可以讲的开发工作

| 工作 | 面试表达 |
| --- | --- |
| 查询口径梳理 | 明确筛选条件、展示字段、统计维度和导出字段 |
| 表结构变更 | 补充能表达授信变动前后值、变动原因、时间等信息的字段或记录结构 |
| 历史数据回填 | 让存量记录也能适配新查询口径，避免只支持新数据 |
| 后端接口 | 支持分页查询、条件过滤、统计汇总和 CSV 导出 |
| 数据校验 | 回填后核对记录数、关键字段和统计结果是否一致 |

## 高频追问

| 问题 | 答案 |
| --- | --- |
| 为什么要改表？ | 原结构不能稳定表达授信变动查询所需字段，改表是为了让查询、统计、导出有统一数据基础。 |
| 为什么要回填历史数据？ | 如果只支持新数据，风控无法统一回溯存量授信变动，查询结果会断层。 |
| 查询功能最容易踩什么坑？ | 查询口径不清、索引没覆盖高频条件、历史数据没兼容、导出字段和页面字段不一致。 |
| CSV 导出为什么值得讲？ | 中后台系统不只在线查看，还要支持业务分析和留档，导出字段口径必须和查询结果一致。 |
| 技术含量在哪里？ | 不在复杂架构，而在查询口径、表结构、历史兼容和导出能力一起落地。 |

## 如果被问性能

可以答：

“这类查询功能通常要关注高频过滤条件，比如客户编号、变动日期、状态、操作类型等。设计时会考虑索引覆盖、分页查询，以及导出量控制，避免一次性导出过大数据影响在线查询。”

## 如果被问回填怎么验证

按三层答：

1. 记录数校验：回填前后关键范围内记录数量是否符合预期。
2. 字段校验：抽样核对授信前值、后值、日期、客户编号等关键字段。
3. 统计校验：用新查询结果和原有口径或人工样本结果做对比。

## 1 分钟完整版

“广发证券授信变动查询是我早期做过的一条典型中后台开发。风控人员需要统一查询客户授信调整记录，包括调整前后值、变动时间、原因和统计导出。如果只是加一个页面是不够的，因为底层数据结构要能稳定表达这些变动信息，所以我参与了查询口径设计、PostgreSQL 表结构变更、历史数据回填，以及后端分页查询、统计和 CSV 导出。这个项目不算复杂平台，但很能体现后端开发基本功：先把业务口径定义清楚，再让数据库结构、存量数据和接口能力一致支撑它。”
