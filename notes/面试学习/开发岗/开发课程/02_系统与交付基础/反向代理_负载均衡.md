# 反向代理是什么，负载均衡是什么

这两个概念经常和 `Nginx` 一起出现，
也是开发岗面试里很常见的一组题。

如果你先只记一句话，可以记这个：

- `反向代理`：客户端不直接访问后端服务，而是先访问代理服务器，再由代理服务器转发请求
- `负载均衡`：把请求分散到多台后端机器上，避免某一台压力过大

---

## 1. 什么是代理

先把“代理”这个词理解一下。

代理就是：

`你不直接和目标对象打交道，而是先经过一个中间人。`

在网络里也是一样：

- 用户本来想访问某个服务
- 但他并不直接访问这个服务
- 而是先访问一个中间层
- 再由这个中间层去和真正的服务交互

这个中间层就是“代理”。

---

## 2. 反向代理是什么

`反向代理` 最简单的理解是：

`代理服务器站在服务端这一侧，对外统一接收请求，再把请求转发给后端服务。`

也就是说：

- 用户看到的是一个统一入口
- 但后面可能有一个或多个真实服务
- 用户通常并不直接知道真正处理请求的是哪一台服务

比如：

1. 浏览器访问某个域名
2. 请求先到 `Nginx`
3. `Nginx` 再把请求转发给后面的 Java 服务

这里 `Nginx` 就在做反向代理。

---

## 3. 为什么叫“反向”代理

这是很多人容易懵的地方。

你可以这样理解：

- `正向代理`：更像是替“客户端”做代理
- `反向代理`：更像是替“服务端”做代理

用户访问网站时，
如果前面放了一个 `Nginx`，
那用户其实访问到的是 `Nginx`，
而不是直接访问后端 Java 服务。

所以说，
这个代理是站在服务端前面的，
因此叫 `反向代理`。

你现在面试不一定要展开正向代理，
但至少要知道：

`反向代理是服务端入口层的代理。`

---

## 4. 反向代理能做什么

反向代理常见作用有：

- 统一入口
- 请求转发
- 隐藏后端真实服务地址
- 做 HTTPS 终止
- 做访问控制
- 做路由分发
- 为负载均衡做入口

所以它的本质作用不是“执行业务”，
而是：

`“先把请求接住，再决定怎么转给后端。”`

---

## 5. 用一个很具体的例子理解反向代理

假设你的系统里有一个后端服务跑在：

- `10.0.0.11:8080`

如果没有反向代理，
用户可能就要直接访问这个地址。

如果有了 `Nginx`：

- 用户访问的是 `https://example.com`
- `Nginx` 收到请求
- `Nginx` 再把请求转发给 `10.0.0.11:8080`

这时对于用户来说：

- 用户只知道统一域名
- 不需要知道后端机器的真实地址

这就是反向代理的价值之一。

---

## 6. 负载均衡是什么

`负载均衡` 最简单的理解是：

`当后面有多台服务时，把请求合理分摊到不同机器上。`

比如后面有 3 台 Java 服务：

- `10.0.0.11:8080`
- `10.0.0.12:8080`
- `10.0.0.13:8080`

如果所有请求都打到第一台，
那第一台可能很快就扛不住。

所以要有一个机制，
把请求分散开，
比如：

- 第 1 个请求给 A
- 第 2 个请求给 B
- 第 3 个请求给 C

这就是负载均衡。

---

## 7. 负载均衡为什么重要

它的作用主要有：

- 提高系统承载能力
- 避免单机压力过大
- 提高可用性
- 方便系统横向扩展

如果只有一台服务，
它挂了系统就挂了。

如果有多台服务并做了负载均衡：

- 某一台挂掉，其他机器还能接流量
- 后续要扩容时，也可以继续加机器

所以负载均衡和高可用、扩展性关系很大。

---

## 8. 反向代理和负载均衡是什么关系

这两个概念相关，但不是一回事。

你可以这样记：

- `反向代理` 解决的是“请求先由谁接住、怎么转发”
- `负载均衡` 解决的是“请求转发给哪一台后端机器”

很多时候它们会同时出现在 `Nginx` 身上：

1. `Nginx` 先作为反向代理接住所有请求
2. 再按负载均衡策略，把请求分发到不同后端服务

所以：

- 反向代理更偏“入口和转发”
- 负载均衡更偏“多实例分配”

---

## 9. 常见负载均衡策略

你不用背得很深，
但知道几个常见名字会比较好。

### 9.1 轮询

最常见、最容易理解。

就是请求按顺序分给不同机器：

- 第 1 个给 A
- 第 2 个给 B
- 第 3 个给 C
- 第 4 个再回到 A

### 9.2 加权轮询

如果有的机器性能更强，
就可以让它分到更多请求。

比如：

- A 权重高
- B 权重低

那 A 会拿到更多流量。

### 9.3 IP 哈希

根据客户端 IP 做分配，
让同一个客户端更容易落到同一台机器。

这个在某些需要会话一致性的场景里会用到。

---

## 10. 放到你的面试语境里怎么讲

如果面试官问：

“什么是反向代理？”

你可以答：

“反向代理就是由代理服务器统一接收客户端请求，再把请求转发给后端服务。客户端不直接访问后端机器，而是访问代理层。像 Nginx 在很多项目里就承担这个角色，它可以隐藏后端服务地址、做统一入口、做转发和 HTTPS 处理。” 

如果面试官问：

“什么是负载均衡？”

你可以答：

“负载均衡是在后端有多台服务时，把请求分散到不同机器上，避免单机压力过大，同时提升系统可用性和扩展性。像 Nginx 就可以在作为入口层的同时，把请求按轮询或其他策略分发到多台后端服务。” 

---

## 11. 更口语一点的讲法

你也可以这样讲：

“反向代理就是前面先放一个中间层来接请求，用户不直接打后端服务；负载均衡就是后面如果有多台服务，这个中间层再把请求分给不同机器。” 

---

## 12. 和你前面学过的 Nginx/Tomcat 怎么串起来

你可以把整个链路理解成这样：

1. 用户请求先到 `Nginx`
2. `Nginx` 作为反向代理统一接请求
3. 如果后面有多台 Java 服务，`Nginx` 再做负载均衡
4. 某一台后端服务接到请求后，在 `Tomcat` 或内嵌 `Tomcat` 中处理业务
5. 处理结果返回给用户

所以：

- `Nginx` 常承担反向代理和负载均衡
- `Tomcat` 常承担 Java 应用运行

---

## 13. 面试官可能继续追问什么

### 13.1 为什么需要反向代理

你可以答：

“我理解主要是为了做统一入口、隐藏后端服务、方便做 HTTPS、路由转发和后续扩展。” 

### 13.2 为什么需要负载均衡

你可以答：

“主要是为了分散压力、提高并发承载能力、提升高可用，并且方便后续横向扩容。” 

### 13.3 Nginx 为什么经常和负载均衡一起提

你可以答：

“因为 Nginx 常常既在前面做反向代理接流量，也负责把流量分发给多台后端服务，所以在实际项目里它经常同时承担这两个角色。” 

---

## 14. 你只要记住这 5 句话

1. `反向代理` 是前面先放一个代理层接请求
2. 客户端不直接访问后端服务
3. `负载均衡` 是把请求分配给多台后端机器
4. `Nginx` 往往同时承担反向代理和负载均衡
5. `Tomcat` 则负责真正运行 Java 应用

---

## 15. 一句总结

`反向代理解决“请求先到哪、怎么转”，负载均衡解决“后面多台服务时请求分给谁”；在很多 Java 系统里，Nginx 常同时承担这两层职责。`
