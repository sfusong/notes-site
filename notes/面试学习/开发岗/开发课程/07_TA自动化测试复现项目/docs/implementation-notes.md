# TA 自动化测试复现：实现复盘与面试记忆点

## 1. 这次复现真正做了什么

这次复现没有实现 TA 批处理。它实现的是 TA 回归测试流程里的自动化辅助部分：

1. 批前清理本批次数据。
2. 批前初始化产品、客户、交易申请和预期确认结果。
3. 生成代销商申请文件。
4. 把申请文件放到 TA inbound 目录。
5. 保留人工检查点，由测试人员进入 TA 图形化界面启动批处理。
6. 批后读取 TA outbound 目录下的确认文件。
7. 解析确认文件。
8. 根据批前埋下的数据读取预期确认结果。
9. 字段级比对确认文件是否符合预期。
10. 输出 Markdown 报告。

## 2. 这套工具链如何配合

Jenkins 负责把流程串起来。它不是业务逻辑本身，而是统一入口、参数入口和结果归档入口。

Shell 脚本负责承接 Jenkins 的每个 stage，让每一步看起来像真实 Linux 测试环境里的操作入口。

Java / Spring Boot CLI 负责文件生成、文件解析、预期与实际比对、报告输出。

SQL / H2 负责模拟 Oracle 里的造数和校验思路。真实项目中，SQL 是核心手段，因为批处理场景很多状态只能通过数据库高效准备和验证。

文件目录负责模拟 TA 和代销商之间的文件交互。`data/inbound` 是申请文件入口，`data/outbound` 是确认文件出口。

## 3. 最大设计难点

最大的难点不是 Java 代码，而是边界。

如果把 TA 批处理也写成本地 processor，项目会看起来更“完整”，但这会背离真实情况。真实项目里，TA 是被测系统，批处理由测试人员在图形化界面启动。我们的自动化工具只负责批前输入准备和批后输出校验。

这个边界一旦讲清楚，项目反而更可信：我们不是替代 TA，而是提高 TA 回归测试自动化水平。

## 4. 最大实现难点

第一，批次号要贯穿全流程。

`testBatchNo` 同时出现在申请数据、预期确认、实际确认快照、比对结果和自动化日志里。没有这个批次号，失败时就很难知道某条确认结果对应哪次回归。

第二，预期结果要从批前埋数推导出来。

比如成功申购场景里，产品必须可申购，客户风险必须有效，确认份额要由金额和净值计算出来。这样批后不是“看起来对”，而是能逐字段对账。

第三，确认文件不能只校验存在。

文件存在只能说明 TA 生成了产物，但不能说明 TA 处理正确。真正有价值的是校验 `confirmStatus`、`confirmAmount`、`confirmShares`、`confirmDate`、`reasonCode` 这些关键字段。

第四，人工检查点要显式建模。

Jenkinsfile 里的 `Manual TA Batch In GUI` stage 很重要。它告诉面试官：真实项目不是全无人值守，自动化和人工操作之间有清楚分工。

## 5. 面试时可以这样讲难点

“这个项目最难的不是写一个文件解析器，而是把 TA 回归测试的边界拆清楚。TA 批处理本身是被测系统，我们不能替代它；我们做的是批前把输入标准化，批后把结果校验标准化。为此我把流程拆成批前造数、申请文件生成、人工跑批检查点、确认文件解析、预期和实际字段级比对、报告输出几个模块。这里面比较关键的是批次号贯穿、预期结果来源和文件内容对账，否则失败时很难定位到底是造数错、文件错、TA 处理错，还是校验逻辑错。”

## 6. 哪些话可以说，哪些话不能说太满

可以说：

- 我参与的是 TA 自动化回归测试建设。
- 我熟悉的是申请文件进入、TA 跑批、确认文件输出、批后校验这条链路。
- 我主要做的是场景拆解、前后置条件定义、关键校验点设计、推动落地和验收。
- 技术上理解 Jenkins、Shell、SQL、Java 文件处理和数据库对账各自的职责。

不要说太满：

- 不要说我实现了 TA 批处理。
- 不要说全流程无人值守。
- 不要说确认文件由我们的工具生成。
- 不要说底层平台全部由我一个人主导开发。
