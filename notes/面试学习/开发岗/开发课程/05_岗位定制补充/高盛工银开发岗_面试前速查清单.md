## 2. 产品管理系统 / 生命周期系统

### 5. 产品管理系统和产品生命周期管理系统分别解决什么问题？
产品管理系统是供应商采购的统一正式平台，解决产品要素统一管理、说明书和销售文件自动生成、审批流和下游系统集成等问题。产品生命周期管理系统是正式平台上线前先承接业务需求的自研辅助系统，聚焦特殊产品的关键节点提醒和规则落地，比如目标盈、分红封闭式、定期分红产品。

### 6. 为什么一个采购，一个自研？
产品管理系统覆盖面大、流程长、涉及多个部门和下游系统，更适合采购成熟的长期平台能力。生命周期系统和说明书工具是因为业务需求来得快，正式平台短期内无法承接，所以先做了范围更聚焦、上线更快的自研工具，用来支持前期产品发行和运营需求。

### 7. 在产品管理系统里你到底做了什么？哪些是 vendor 做的？
我主要负责需求收敛、字段口径梳理、技术规格审核、SIT/UAT 组织、上线试运行和培训推进。供应商负责需求文档初稿、系统开发和 SIT 执行。我这边是在内部把业务需求转成明确系统需求，并推动业务、供应商、测试把系统真正落地。

### 8. 你怎么审核需求文档？
我主要看四个方面：第一，业务目标和系统边界是否清楚；第二，字段、流程、角色权限和口径是否明确；第三，接口依赖、上下游影响和数据来源是否写清楚；第四，性能、稳定性、测试、上线、历史数据迁移等约束是否覆盖。我的目标不是只看文档写没写，而是看这份需求能不能稳定交付。

### 9. 产品生命周期系统里最能证明你开发能力的功能是什么？
我会选目标盈产品的提醒功能。这个功能不是简单按固定日期提醒，而是要结合产品收益情况和业务规则判断是否接近目标收益，再提示业务关注。我主要参与了后端逻辑、数据中台联调、提醒结果落库，以及邮件和 OA 提醒接口的对接，这条功能比较能体现规则落地、数据联调和系统实现能力。

### 10. 如果目标盈提醒判断错了，你怎么排查？
我会按“规则、数据、系统”三层排查。先确认业务规则和口径有没有理解偏，比如看哪个收益指标、达到什么条件触发；再看上游数据是否完整、最新、口径一致；最后才查后端实现，比如条件判断、边界值处理、提醒状态和落库逻辑。不会一上来就直接改代码。

## 3. TA 自动化测试 / 批前预警

### 11. TA 自动化测试工具到底解决了什么问题？
它主要解决的是 TA 对接和版本迭代中的测试效率和稳定性问题。原来很多关键场景依赖外部代销机构联调或者人工验证，效率低，也不利于稳定做回归。这个工具把申请文件生成、环境准备、批处理、确认文件解析和数据库比对串成自动化闭环，让回归验证标准化。

### 12. 你在 TA 自动化测试里具体做了什么？
我是这个工具的需求提出方和部分核心功能的开发参与者。我参与了技术选型，并自己写了确认文件解析和数据库结果比对这部分代码，所以对 TA 的相关数据库表和确认口径有比较直接的理解。

### 13. 技术上怎么落地？
整体工作流主要通过 Jenkins Pipeline 统一触发，底层实现以 Java 为主。流程上包括自动生成代销申请文件、做数据准备和清理、触发批处理、获取系统生成的确认文件、解析确认文件内容、再和数据库结果做比对，最后输出测试结果。

### 14. 自动化结果不一致时你怎么判断问题在哪？
我会先查工具本身，比如解析规则、字段类型、格式处理、前后缀和枚举映射；再查测试数据和环境，比如申请文件内容、数据清理、前置状态和批处理准备；最后才查系统本身，比如批处理逻辑、确认口径或代码改造引入的问题。

### 15. TA 批前预警平台里有哪些类型的检查？
可以分成两类。第一类是交易类批前预警，更偏基于当前份额和新申购规模情况做检查，比如募集规模、销售额度、持有限额、产品成立条件，这部分更对应恒生 AOP。第二类是静态参数稽核，更偏检查关键参数有没有漏设或错设，比如托管账号、清算账号、费率、开放期、日期、产品状态等，这部分更偏团队自研。

### 16. 为什么静态参数稽核值得自研？
因为很多关键参数问题不一定被厂商标准预警能力完整覆盖，但对后续批处理和清算影响很大。自研的价值在于把内部真实高频、关键、贴业务口径的风险点沉淀成自己的检查项，让问题在批前就暴露出来。

## 4. 开发基础题

### 17. Spring Boot 是什么？你在项目里怎么用？
Spring Boot 是企业后端常见的 Java 框架，用来快速组织接口、业务逻辑、数据库访问和配置管理。在我的项目里，它主要承接后端服务能力，比如产品生命周期系统里的业务规则判断、提醒逻辑、接口提供和数据读写。

### 18. 一个典型的 Spring Boot 后端请求流程是怎样的？
请求先根据路径和映射关系进入对应的 Controller。Controller 接收参数并做基础校验，然后把请求分发到 Service。Service 负责业务逻辑处理，如果涉及数据库读写，再调用 DAO 或 Repository。处理完成后结果返回给 Controller，由 Controller 以统一结构返回给前端。

### 19. React 在你项目里怎么用？
React 在我的项目里主要负责页面交互、表单录入、列表展示、状态展示以及和后端 REST API 的联动。比如产品生命周期系统里，前端要把事件提醒、状态变化和产品信息通过组件化页面展示出来，并支持和后端接口联动。

### 20. SQL / 数据库设计你做过什么？
我做过查询功能相关的 SQL 和数据库设计，也做过业务系统里的基础表设计。比如广发证券授信变动查询项目里，我参与了表结构变更、历史数据回填、查询和导出功能；在产品生命周期系统里，我做过产品基础要素表、个性化要素表和工作日历相关表的设计。

### 21. MySQL 和 Redis 分别适合做什么？
MySQL 适合存结构化、需要长期持久化的核心业务数据，比如产品、交易、用户、状态记录。Redis 更适合做缓存和高频访问场景，主要价值是提升性能、减轻数据库压力，也常用于会话、计数器和一些临时状态。

### 22. 你平时怎么用 Git？
我平时最常用的是 `git clone`、`git pull`、`git checkout -b`、`git status`、`git add`、`git commit -m`、`git push`、`git branch`、`git merge`、`git log` 和 `git diff`。实际工作里更常见的是拉代码、切分支、提交本地改动、推送远程以及后续合并协作。

### 23. Linux 你常用来做什么？
我在自动化测试和发布链路里会接触 Linux 环境，主要用于查目录、看日志、跑脚本和做基础排查。常用命令包括 `ls`、`cd`、`pwd`、`cat`、`less`、`tail`、`grep`、`ps`、`top`、`chmod`。比如会用 `tail` 和 `grep` 看日志，用 `ps` 和 `top` 查进程状态，用 `chmod` 处理脚本权限。

### 24. 你怎么理解 CI/CD？
CI/CD 是把代码提交、构建、测试、制品生成和发布串成一条自动化链路。对我接触的场景来说，更接近持续交付而不是持续部署，也就是通过 Jenkins Pipeline 这类流程入口，把版本构建、验证和上线动作标准化，但最终生产发布通常仍然需要人工确认和窗口控制。

## 5. 交付、联调、上线题

### 25. 你怎么理解接口联调？
接口联调不只是技术协议调通，而是要把字段口径、业务时序、异常处理和系统间交互真正跑通。尤其在金融系统里，接口对接往往还涉及文件交互和批处理时点，所以联调的核心是让业务链路真正跑通。

### 26. 上下游字段口径不一致，你怎么处理？
我一般按四步处理：先收清下游真实需求，明确字段定义、格式和取值限制；再和上游做字段和口径对齐；如果不一致，再评估是通过字段映射、转换规则解决，还是由上游或下游改造；最后把统一口径和映射关系沉淀回文档，再进入联调和验证。

### 27. 你怎么理解 UAT？
UAT 是业务验收，不只是技术测试。我的理解是要先和业务一起把真实流程、关键节点、异常情况和边界场景收清楚，再把它们转成测试案例。除了主流程，还要覆盖异常流程、边界条件、角色权限差异和上下游影响，最后通过业务确认、测试执行、缺陷收口和测试纪要作为上线前置材料。

### 28. 上线切换时你最关注什么？
最关注时点控制、影响范围和回滚预案。特别是在接口升级切换或批处理链路场景里，我会重点看新旧链路如何衔接、核心流程是否已经充分验证、出问题后能否及时切回旧方案，以及相关方是否都已经准备好。

### 29. 如果上线前发现关键问题，但业务又想按时上线，你怎么判断？
我会先判断问题是否影响核心链路和数据正确性。如果影响交易、确认、批处理、关键审批或核心接口，那通常不能带病上线；如果问题不影响主链路，可以评估是否隔离、降级或拆分上线。我的原则是先看风险，再看时间，不会为了赶节点把高风险问题强行带到生产。

### 31. 你最大的短板是什么？

### 32. 你面对变化快、需求紧的时候怎么处理？
我通常先收边界，明确这次到底要解决什么问题，哪些是必须做，哪些可以后置；再找依赖和风险点，明确业务、开发、供应商和测试分别需要什么；最后按优先级分阶段推进，确保先把核心能力做出来，再逐步补齐。我的经验是变化快时更需要先把边界收清。

### 33. 你怎么体现主人翁精神？
我在项目里不会只看自己手上的一段任务，而会关注整条链路最后能不能真正落地。比如需求有没有收清、联调有没有跑通、UAT 有没有覆盖关键风险点、上线切换有没有回滚预案，这些我都会主动多看一步。我比较习惯站在“系统最后能不能顺利上线和稳定运行”的角度去推进事情。
