## 二、Docker / 容器化

## 1. Docker 最短定义

`Docker 是一种容器化技术，用来把应用和它依赖的运行环境一起打包，并在不同环境里以比较一致的方式运行。`

## 2. 容器和虚拟机的区别

- `虚拟机`：每个实例都有一套完整 Guest OS
- `容器`：共享宿主机内核，在操作系统层做隔离

所以：

- 虚拟机更重
- 容器更轻
- 容器启动更快、资源占用更少

## 3. Docker 的原理要怎么讲

最稳的说法：

`Docker 的核心思想是把应用和依赖一起封装到镜像里，然后通过容器把镜像跑起来。容器不是像虚拟机那样带一整套独立操作系统，而是共享宿主机内核，通过 Linux 的 namespace 和 cgroups 来实现进程、网络、文件系统和资源的隔离与控制。`

## 4. 面试官如果提到“虚拟化、IO”

你可以这样接：

`我理解 Docker 更偏操作系统层面的轻量虚拟化，而不是硬件层虚拟化。因为它共享宿主机内核，所以在启动速度和资源利用率上通常更高效。至于 IO，我理解容器本身并不是天然就让业务 IO 变快，而是通过更轻量的运行方式和更稳定的一致环境，减少部署和运行层面的额外开销。`

## 5. Docker 为什么对开发有价值

- 环境一致性
- 部署更方便
- 减少“本地能跑、线上跑不起来”
- 更适合做测试、交付和扩缩容

## 6. 镜像和容器是什么关系

- `镜像`：应用和运行依赖的静态模板
- `容器`：镜像运行起来之后的实例

最简单理解：

`镜像像打包好的应用环境，容器像真正跑起来的进程环境。`

## 7. Docker 的 IO 怎么理解

如果面试官继续追问 IO，你不用讲很深，抓住这几层：

- 容器里的文件系统通常基于镜像层和可写层
- 容器删除后，可写层里的数据通常也会丢
- 持久化数据一般要通过 volume 或挂载目录处理
- 容器共享宿主机内核，文件和网络 IO 最终还是落到宿主机资源上

最稳说法：

`我理解容器 IO 不是完全独立的一套硬件 IO，而是通过容器文件系统、挂载卷和宿主机内核能力来完成。真正要持久化的数据，一般不会只放在容器可写层里，而会用 volume 或外部存储。`

## 8. Docker 的网络怎么理解

你只要先知道：

- 容器有自己的网络命名空间
- 可以通过端口映射把容器服务暴露出来
- 容器之间也可以通过容器网络通信

最稳说法：

`Docker 会给容器提供隔离的网络环境，应用在容器里监听端口，再通过端口映射或容器网络和外部或其他服务通信。`

## 9. 你最稳的一句话答法

`Docker 的核心价值是把应用和依赖一起打包，并通过容器化方式在不同环境里稳定运行。它不是完整虚拟一套操作系统，而是通过共享宿主机内核和操作系统层隔离机制实现轻量运行，所以通常比传统虚拟机更轻、更快。`

## 三、Linux

## 1. Linux 是什么

`Linux 是企业后端系统里非常常见的服务器操作系统。`

## 2. 为什么开发要懂 Linux

- 服务常跑在 Linux 上
- 日志常在 Linux 上查
- 脚本常在 Linux 上执行
- 排障常绕不开 Linux

## 3. 常用命令你至少会讲

- `pwd`：看当前目录
- `ls`：看目录文件
- `cd`：切目录
- `cat` / `less`：看文件
- `tail -f`：跟日志
- `grep`：查关键字
- `ps` / `top`：看进程和资源
- `chmod`：改权限

## 4. 最稳答法

`我在自动化测试和发布链路里会接触 Linux 环境，主要用于查目录、看日志、跑脚本和做基础排查。常用命令包括 ls、cd、pwd、cat、less、tail、grep、ps、top 和 chmod。`

## 五、CI/CD、Jenkins、DevOps

## 1. CI/CD 是什么

`CI/CD 是把代码提交、构建、测试、制品生成和发布串成一条自动化链路。`

## 2. CI 和 CD 的区别

- `CI`：持续集成，更偏代码合并、构建、自动化测试
- `CD`：持续交付 / 持续部署，更偏把可发布版本稳定往后推

## 3. 你现在最稳的定位

`我接触到的更偏持续交付，而不是完全自动持续部署。`

## 4. Jenkins Pipeline 是什么

`Jenkins Pipeline 是把构建、测试、脚本执行、环境准备和发布动作串成标准流程的一种方式。`

## 5. DevOps 怎么最简单讲

`DevOps 不是单一工具，而是开发和运维协作的一种工程方式，强调把构建、测试、发布和运行维护通过自动化和流程标准化连接起来。`

## 6. 你最稳的一句话答法

`我对 CI/CD 的理解是把代码提交、构建、测试、制品生成和发布串成一条自动化链路。对我接触的场景来说，更接近持续交付，Jenkins Pipeline 是一个很常见的统一流程入口。`

## 六、IaC / Ansible / Terraform

## 1. IaC 是什么

`IaC = Infrastructure as Code，基础设施即代码。`

## 2. 核心价值

- 标准化
- 可复现
- 自动化
- 减少人工误差

## 3. Ansible 和 Terraform 怎么区分

- `Ansible`：更偏配置和批量执行
- `Terraform`：更偏资源定义和编排

## 4. 最稳答法

`IaC 的核心是通过代码或配置文件来标准化管理环境，而不是完全依赖人工操作。Ansible 更偏配置和执行，Terraform 更偏资源定义和编排。`

## 七、Spring Boot / Spring MVC / REST API

## 1. Spring Boot 是什么

`Spring Boot 是企业后端常见的 Java 框架，用来快速组织接口、业务逻辑、数据库访问和配置管理。`

## 2. Spring MVC 是什么

`Spring MVC 是 Spring 生态里处理 Web 请求的一套方式。`

## 3. REST API 是什么

`REST API 是后端通过 HTTP 对外暴露业务能力的方式。`

## 4. 后端请求链路怎么讲

1. 请求进入 `Controller`
2. `Controller` 接参数并做基础校验
3. `Service` 处理业务逻辑
4. `Repository / DAO` 访问数据库
5. 结果返回前端

## 5. 为什么业务逻辑放 Service

`因为 Controller 更适合做请求入口和参数处理，业务逻辑放 Service 更利于职责分离、复用和测试。`

## 6. 最稳答法

`Spring Boot 更像快速组织后端应用的框架，Spring MVC 更聚焦请求如何进入 Controller、如何接参数、如何返回结果。对我来说，一个典型后端系统最核心的链路就是请求进入 Controller，由 Service 处理业务逻辑，再通过 Repository 访问数据库，最后返回结果。`

## 八、Java OOP / 设计模式

## 1. OOP 是什么

`OOP 是面向对象编程，核心是把业务对象、状态和行为组织成类和对象。`

## 2. 四大特性

- `封装`：隐藏内部细节，对外提供稳定接口
- `继承`：复用共性，但不要滥用
- `多态`：同一个接口可以有不同实现
- `抽象`：提取共同模型，屏蔽不重要细节

## 3. 你最该会的设计模式

- `策略模式`：不同规则、不同算法、不同处理器
- `模板方法`：流程固定，部分步骤变化
- `工厂模式`：根据类型创建不同实现
- `责任链模式`：多个校验节点按顺序处理
- `适配器模式`：对接外部系统时做格式转换

## 4. 最稳答法

`我理解 OOP 不只是背封装、继承、多态，而是把业务规则抽象成可维护的对象模型。比如 TA 静态参数稽核里，不同稽核项可以抽象成不同规则策略；产品生命周期提醒里，不同提醒类型可以抽象成不同规则实现。这样主流程稳定，新增规则时主要做扩展，不需要大改原有代码。`

## 九、MySQL / Redis

## 1. MySQL 是什么

`MySQL 是关系型数据库，适合存结构化、长期持久化的核心业务数据。`

## 2. Redis 是什么

`Redis 是高性能内存型键值存储，常见用途是缓存。`

## 3. 为什么系统里常同时用它们

`MySQL 负责存核心数据，Redis 负责高频访问和性能优化。`

## 4. 索引是什么

`索引是帮助数据库加速查询的辅助结构，类似目录。`

## 5. 事务是什么

`事务的核心是一组操作要么都成功，要么都失败。`

## 6. 缓存高频概念

- 缓存穿透
- 缓存击穿
- 缓存雪崩

## 7. 最稳答法

`MySQL 更适合保存结构化、需要长期持久化的核心业务数据，Redis 更适合做缓存和高频访问场景下的轻量存储。很多系统同时用它们，是因为核心数据最终还是落在数据库里，而 Redis 主要用来减轻数据库压力、提升响应性能。`

## 十、数据库设计 / UML / 异常恢复

## 1. 数据库设计先看什么

`先看业务对象、业务关系和查询场景，再设计表、字段、主键、索引和状态。`

## 2. 索引怎么讲

`索引是加速查询的辅助结构，但会增加写入和维护成本，所以要围绕高频查询条件设计。`

## 3. 冗余字段怎么讲

`合理冗余可以用于查询效率和历史快照，尤其金融系统需要保留当时口径，便于追溯。`

## 4. 异常恢复怎么讲

`批次类任务要记录状态和日志，失败后能定位步骤；可恢复异常做有限重试，数据错误需要人工修正；重跑要保证幂等，避免重复数据和状态污染。`

## 5. UML 现场怎么讲

`不要画复杂图，先画清组件关系、数据流和关键类抽象。产品生命周期系统可以画前端、Spring Boot 后端、数据库、数据中台、邮件/OA；TA 稽核可以画 AuditRule 接口和多个具体规则实现。`

## 6. 最稳答法

`我做数据库设计会从业务对象和查询场景出发，同时考虑金融系统里的可追溯和异常恢复。比如稽核系统需要批次表、稽核结果表和明细表，既能看汇总，也能追到具体参数问题；重跑时要通过批次号和状态控制幂等。`

## 十一、Tomcat / Nginx / jar / war / JDK / JRE / JVM

## 1. Tomcat 和 Nginx 区别

- `Tomcat`：更偏 Java Web 应用运行容器
- `Nginx`：更偏 Web 服务器、反向代理、负载均衡器

## 2. 常见架构

- 前面 `Nginx`
- 后面 `Tomcat` 或内嵌 `Tomcat` 的 Spring Boot 服务

## 3. jar 和 war

- `jar`：更常见于 Spring Boot，通常内嵌 Tomcat，可直接 `java -jar`
- `war`：更常见于传统 Java Web 项目，部署到外部 Tomcat

## 4. JDK / JRE / JVM

- `JDK`：开发工具包，包含编译器、JRE 等
- `JRE`：运行 Java 程序所需环境
- `JVM`：真正执行 Java 字节码的虚拟机

## 5. 最稳答法

`Tomcat 更像 Java 应用运行层，Nginx 更像流量入口层；前者负责让 Java 应用跑起来，后者负责把请求接入、转发和分流。Spring Boot 场景下通常是 jar 包直接运行，内部带了内嵌 Tomcat。`

## 6. 反向代理和负载均衡

- `反向代理`：客户端请求先到代理服务器，再由代理转发给后端服务
- `负载均衡`：把请求分发到多台后端机器，避免单点压力过大

最稳答法：

`Nginx 常作为反向代理放在应用前面，对外提供统一入口，再把请求转发给后端服务。如果后面有多台服务实例，Nginx 也可以做负载均衡，把请求分摊到不同实例上。`

## 7. HTTPS 终止和会话保持

- `HTTPS 终止`：HTTPS 加解密在 Nginx 或负载均衡层完成，后端服务不用每个实例都直接处理证书
- `会话保持`：让同一个用户的一段会话尽量落到同一台后端服务上

最稳答法：

`HTTPS 终止通常放在入口层，比如 Nginx 或负载均衡层，这样证书和加解密可以统一管理。会话保持则是为了解决多实例后端场景下，同一个用户请求被分到不同实例时状态不一致的问题。`

## 十五、面试前最后一定要记住的 16 句话

1. Docker 是轻量容器化，不是完整虚拟一套 OS。
2. 容器共享宿主机内核，核心隔离机制是 namespace 和 cgroups。
3. Linux 对开发的重要性在于运行环境、日志、脚本和排障。
4. Git 是版本控制和团队协作工具。
5. CI/CD 是自动化构建、测试、制品和发布链路。
6. DevOps 是开发和运维协作的工程方式，不是单一工具。
7. IaC 是基础设施即代码，强调标准化和可复现。
8. Spring Boot 是企业后端常见 Java 框架。
9. REST API 是后端对外暴露业务能力的方式。
10. OOP 的核心是封装、继承、多态和抽象。
11. 策略模式适合规则很多、后续要扩展的场景。
12. MySQL 存核心结构化数据，Redis 做缓存和高频访问。
13. 数据库设计要从业务对象和查询场景出发。
14. 批次类任务要考虑状态、日志、重试和幂等。
15. Tomcat 跑 Java 应用，Nginx 做入口和反向代理。
16. 你不需要把这些讲很深，但一定不能完全空白。
17. 集合管数据，异常管失败，并发管共享，JVM 管运行。
18. 接口定契约，事务保一致，幂等防重复，日志便追溯，监控看运行。
