## 2. 产品生命周期管理系统

### Q2：请按“业务问题 -> 系统方案 -> 架构/数据流 -> 你做的开发工作 -> 测试验收 -> 价值”讲产品生命周期管理系统

### 你的当前回答问题

- 业务背景真实，但结构偏散，面试官不容易抓住架构主线。
- 技术工作说得不够具体，需要明确后端模块、表设计、跑批、数据联调、提醒落库、邮件/OA。
- 不要说“开发比较快”“比较简单”，会削弱项目含金量。
- 测试验收要讲成正常、边界、异常场景，而不是只说 SIT/UAT。

### 回答思路

固定六段：

1. 业务问题：产品数量多，特殊产品节点依赖人工台账和人脑记忆。
2. 系统方案：自研 PLM，承接正式产品管理系统上线前的特殊产品节点管理。
3. 架构/数据流：TA/数据中台基础数据 + PLM 补充要素 + 行情/工作日历 + 跑批规则 + 提醒记录 + 页面/邮件/OA。
4. 我的工作：后端规则、数据库、数据中台联调、跑批、提醒接口。
5. 测试验收：正常、边界、异常、业务 UAT。
6. 价值：降低遗漏和操作风险。

### 更稳版本

“产品生命周期管理系统解决的是产品存续期关键节点依赖人工台账和人工记忆的问题。当时产品数量已经比较多，像定期分红、目标盈这类特殊产品，在存续过程中会有分红测算、收益达标、提前到期等关键事件。如果完全靠业务同事手工登记和提醒，容易遗漏，也会带来操作风险。

系统方案上，我们做了一套自研的 PLM 系统。它不替代正式的产品管理系统，也不替代 TA，而是作为产品存续期补充要素和事件提醒的平台。产品基础信息，比如产品代码、名称、状态、成立日、到期日，主要从 TA 或数据中台同步进来；业务在 PLM 中维护定期分红、目标盈这类特殊产品的补充要素；行情和收益数据也通过数据中台进入系统。系统每天跑批，根据产品基础信息、补充要素、工作日历和行情数据做规则判断，生成提醒事件和提醒记录，再通过前端页面、邮件或 OA 推送给业务和投资经理。

技术上，后端使用 Java Spring Boot，数据库用 MySQL，前端基于内部 React 低代码平台。我主要负责后端这一侧，包括表结构设计、补充要素和提醒记录的存储、分红/目标盈规则判断、每日跑批逻辑、数据中台联调，以及提醒结果落库和邮件/OA 接口对接。前端我参与接口联动，但主要由同事负责。

测试验收方面，我们会按正常场景、边界场景和异常场景来测。比如定期分红是否在正确工作日窗口提醒，目标盈是否连续达标后触发，产品终止或行情缺失时是否不误触发。SIT 后再和业务一起做 UAT，确认提醒时间、提醒内容和处理流程符合业务口径。

这个项目的价值是把产品存续期管理从人工台账和人脑记忆，转成系统化、可追踪、可提醒的流程，降低遗漏和操作风险。”

### 记忆钩子

`基础数据 -> 补充要素 -> 行情/日历 -> 跑批规则 -> 提醒记录 -> 页面/邮件/OA`

### Q2.1：定期分红产品要生成提醒，需要哪些表或数据？后端跑批如何判断今天是否提醒？

### 你的当前回答问题

- 表结构说得偏泛，只说产品表、行情表、结果表、消息表。
- 定期分红最核心的是 `分红参数表`、`工作日历表`、`分红日程/提醒窗口表`。
- 不要说“定期分红比较简单”，可以说“它更偏日期和规则计算”。
- 跑批逻辑要讲出防重复提醒。

### 回答思路

数据分五类：

1. 产品基础信息。
2. 分红补充参数。
3. 产品工作日历。
4. 分红日程/提醒窗口。
5. 提醒日志。

跑批判断：

`读取有效产品 -> 读取分红参数 -> 计算分红测算日/基准日/提醒窗口 -> 判断批次日是否落在窗口 -> 检查是否重复提醒 -> 生成提醒记录`

### 更稳版本

“定期分红提醒大概需要几类数据。第一类是产品基础信息，比如产品代码、产品名称、产品状态、成立日、到期日、投资经理等，这部分主要来自 TA 或数据中台。第二类是分红补充参数，比如是否定期分红、分红频率、首次分红间隔、测算前提醒天数、测算后监控天数等，这部分由业务在 PLM 里维护。第三类是产品工作日历，用来判断提醒日是否需要按工作日顺延。第四类是分红日程或提醒结果表，用来记录计算出的分红测算日、分红基准日、提醒窗口和实际提醒状态。最后还有提醒日志表，记录批次号、提醒类型、产品、提醒内容、发送状态等。

跑批判断时，系统会按批次日读取有效产品和分红参数，先基于成立日、分红频率和工作日历计算分红测算日、分红基准日，以及测算前提醒窗口。然后判断今天这个批次日是否落在提醒窗口内，如果落在窗口内，并且这个产品当天没有重复生成过同类提醒，就生成提醒记录，并通过页面、邮件或 OA 输出。后续如果需要监控分红方案是否已设置，还会去对比 TA 或数据中台里的分红方案数据，如果测算后仍未设置，也会生成异常提醒。”

### 记忆钩子

`产品基础信息 + 分红参数 + 工作日历 + 分红日程/提醒记录 + 提醒日志`

### Q2.2：目标盈提醒的业务逻辑是什么？如果系统该提醒但没有提醒，你怎么排查？

### 你的当前回答问题

- 目标盈核心逻辑基本讲清了：目标收益率、观察开始日、连续 N 日、成立以来年化收益率。
- 排查时一开始就看接口文件/提醒表，顺序不够工程化。
- 不要太早说“必须翻代码”，应先确认数据、口径、日志，再看规则实现。

### 回答思路

业务逻辑：

`目标收益率 + 观察期开始日 + 连续达标天数 + 成立以来年化收益率`

排查链：

`产品状态 -> 目标盈参数 -> 工作日历 -> 行情收益 -> 规则结果 -> 提醒记录 -> 输出接口`

### 更稳版本

“目标盈产品的核心逻辑是，业务在 PLM 里维护目标止盈收益率、观察期开始日和连续达标天数。系统每天同步产品基础信息、产品工作日历和行情收益数据。当产品进入观察期后，跑批会判断最近 N 个产品工作日的成立以来年化收益率是否都大于等于目标收益率。如果连续达标，就生成目标盈提醒，推送给投资经理或相关业务人员。

如果系统该提醒但没有提醒，我会按链路排查。第一先看产品基础状态，比如产品是否有效、是否已经终止、批次日是否是产品工作日。第二看目标盈参数配置，比如目标收益率、观察期开始日、连续达标天数是否维护正确。第三看行情数据是否完整，最近 N 个工作日的成立以来年化收益率是否都同步进来了。第四看规则结果表或目标盈状态表，确认系统是否判断 today_hit 和 need_take_profit。第五再看提醒记录和输出链路，比如提醒记录是否落库、投管文件或邮件是否生成。如果前面数据和口径都没问题，但系统结果不对，再进一步查规则实现和日志，看是否存在比较符号、日期窗口、工作日判断或重复提醒过滤的问题。”

### 记忆钩子

`产品状态 -> 参数 -> 日历 -> 行情 -> 规则结果 -> 提醒记录 -> 输出`

### Q2.3：前端查询某个产品的生命周期提醒列表，请求进入 Spring Boot 后端后会经过哪些层？每层做什么？

### 你的当前回答问题

- Controller、Service、DAO/Mapper 分层讲到了，基础合格。
- 还需要补参数校验、DTO/VO 组装、统一返回、统一异常处理。

### 回答思路

固定链路：

`Controller 接参校验 -> Service 业务逻辑 -> Mapper/DAO 查库 -> Service 组装 DTO/VO -> Controller 统一返回 -> ExceptionHandler 处理异常`

### 更稳版本

“如果前端查询某个产品的生命周期提醒列表，请求首先会进入 Controller 层。Controller 主要负责接收 HTTP 请求、解析参数，比如产品代码、提醒类型、日期范围、分页参数，并做基础参数校验。校验通过后，会调用 Service 层。

Service 层负责业务逻辑，比如判断查询范围是否合理、是否需要过滤已关闭提醒、是否需要关联产品基础信息和提醒状态。如果只是查询，它会调用 Repository 或 Mapper 层去查提醒记录表、生命周期事件表和产品基础信息表。

DAO/Mapper 层负责和数据库交互，执行具体 SQL 或 ORM 查询。查到数据后会返回给 Service，Service 再组装成前端需要的 DTO 或 VO，比如产品名称、提醒类型、触发日期、处理状态、提醒内容。最后 Controller 返回统一格式的响应。

如果中间出现参数错误、产品不存在或数据库异常，会通过统一异常处理返回错误码和错误信息，而不是直接把底层异常抛给前端。”

### 记忆钩子

`Controller -> Service -> Mapper -> DTO/VO -> Unified Response -> ExceptionHandler`

## 3. TA 自动化测试

### Q3：请按“业务问题 -> 系统方案 -> 架构/数据流 -> 你做的开发工作 -> 测试验收 -> 价值”讲 TA 自动化测试项目

### 你的当前回答问题

- TA 业务闭环讲得真实：申请文件、批处理、确认文件、返回代销。
- 系统方案和数据流需要更清楚地说成 Jenkins、SQL/脚本、Java、TA 批处理、确认文件、expected/actual。
- “都是我负责的”和“不是独立开发”之间边界有点摇摆。
- 不要主动说后来没维护、去做别的项目了。
- 注意术语是 `理财登`，不是 `理财灯`。

### 回答思路

固定数据流：

`Jenkins 参数化触发 -> SQL/脚本准备数据 -> Java 生成申请文件 -> TA 跑批 -> Java 解析确认文件 -> expected/actual 比对 -> 输出报告`

### 更稳版本

“TA 自动化测试项目解决的是 TA 系统版本升级和代销联调时回归测试成本高的问题。TA 最核心的链路是代销商生成申请文件，TA 读取申请文件后进行批处理，生成确认结果和确认文件，再回传给代销。像申购、赎回、开户、销户这些核心场景，每次版本升级后都要回归，人工造数、放文件、跑批、查库、核对确认文件会比较耗时，也依赖测试人员经验。

系统方案上，我们没有把它做成普通 UI 自动化，而是把批前准备和批后校验自动化。Jenkins 作为统一调度入口，传入测试场景、业务日期、销售商、批次号等参数；前置脚本或 SQL 准备测试数据；Java 程序生成代销申请文件并投递到 TA 输入目录；TA 批处理仍然由真实系统执行；批后工具读取 TA 输出的确认文件，解析成结构化记录，再和数据库里的 expected result 做字段级比对，最后输出校验报告。

我参与了方案设计、技术选型和关键实现，主要包括申请文件生成、确认文件解析、字段级比对、数据库结果对账，以及部分脚本编排。测试验收时，我们会覆盖正常成功场景、拒绝场景和异常文件场景，比如确认状态、金额、份额、日期、原因码是否符合预期。这个项目的价值是把 TA 核心回归从人工经验流程变成可重复、可追踪的自动化验证闭环，降低回归成本，也减少外部代销联调依赖。”

### 记忆钩子

`Jenkins -> SQL/脚本 -> Apply 文件 -> TA 批处理 -> Confirm 文件 -> Parser -> Comparator -> Report`

### Q3.1：expected result 和 actual result 分别是什么？确认文件解析后按哪些字段比对？如果不一致怎么判断原因？

### 回答思路

核心结构：

`跑批前预期 -> 跑批后实际 -> 按 key 对齐 -> 字段级比对 -> 分层排查`

expected result：

- 跑批前根据测试场景、产品状态、客户状态、交易申请和业务规则准备的预期确认结果。

actual result：

- TA 真实批处理后生成的确认文件内容，经工具解析成结构化记录。

比对字段：

| 字段 | 作用 |
|---|---|
| `request_no` / 申请编号 | 对齐 expected 和 actual |
| `seller_code` / 销售商代码 | 区分代销渠道 |
| `product_code` / 产品代码 | 确认交易产品 |
| `trade_type` / 交易类型 | 区分申购、赎回、开户等 |
| `confirm_status` / 确认状态 | 成功、失败、部分确认 |
| `confirm_amount` / 确认金额 | 校验资金确认 |
| `confirm_shares` / 确认份额 | 校验份额登记 |
| `confirm_date` / 确认日期 | 校验批次和业务日期 |
| `reason_code` / 原因码 | 校验失败场景 |

### 更稳版本

“在这个项目里，expected result 指的是我们在跑批前根据测试场景、产品状态、客户状态、交易申请和业务规则预先生成或查询出来的预期确认结果。比如某个申购申请，在产品开放、客户风险有效、金额符合要求的情况下，预期应该确认成功；如果产品关闭或客户风险过期，预期应该失败，并带有对应原因码。

actual result 指的是 TA 真实批处理完成后生成的确认文件内容。自动化工具会读取 TA 输出目录里的确认文件，把每一行确认记录解析成结构化对象，比如申请编号、客户号、产品代码、交易类型、确认状态、确认金额、确认份额、确认日期、原因码等。

比对时一般会先用申请编号，或者申请编号加销售商代码、产品代码作为 key，把 expected 和 actual 对齐。然后逐字段比对确认状态、确认金额、确认份额、确认日期、交易类型、原因码这些关键字段。金额和份额字段要特别注意精度和四舍五入规则，日期要注意格式转换，状态和原因码要注意枚举映射。

如果比对不一致，我不会直接判断是 TA 系统 bug，会按三层排查。第一先看工具本身，比如确认文件解析是否错位、字段长度是否变化、金额精度和日期格式是否处理正确。第二看测试数据和环境，比如申请文件里的批次号、业务日期、产品状态、客户风险有效期、净值数据是否准备正确，expected result 是不是按正确场景生成。第三再看 TA 批处理本身，比如批处理节点是否成功，数据库确认结果是否和确认文件一致，日志里是否有异常。如果工具和数据都确认没问题，再把它归为 TA 处理逻辑或文件生成逻辑的问题。”

### 最短背诵版

“expected result 是跑批前根据测试场景和业务规则准备出来的预期确认结果，actual result 是 TA 跑批后确认文件里解析出来的实际确认结果。工具会用申请编号、销售商、产品代码这些字段把两边记录对齐，再比对确认状态、金额、份额、确认日期、交易类型和原因码。如果不一致，我会先查解析和比对逻辑，再查测试数据和环境，最后再看 TA 批处理和确认文件生成逻辑，避免把工具问题或造数问题误判成系统 bug。”

### 记忆钩子

`expected = 跑批前预期；actual = 确认文件实际；key 对齐；字段比对；工具 -> 数据 -> TA`

### Q3.2：为什么用 Jenkins 做 TA 自动化测试？Pipeline 有哪些 stage？失败后如何追踪？

### 你的当前回答问题

- 已经说出 Jenkins 是 workflow 编排平台，这点正确。
- Pipeline stage 也基本说到了：环境清理、申请文件生成、人工 TA 批处理断点、确认文件解析和比对。
- 需要补充 `参数化`、`执行记录`、`制品归档`、`权限和环境一致性`，这是 Jenkins 相比本地脚本的重要价值。
- “为什么不本地跑”不能只说“严谨性不好”，要讲标准化、可追溯、环境一致、多人协作。
- 失败排查要讲清楚：stage 状态、日志、输入文件、输出文件、报告、数据库批次号。

### 回答思路

Jenkins 在这条链路里负责：

- 参数化入口：场景、业务日期、销售商、批次号、环境。
- 流程编排：按 stage 串起环境准备、文件生成、人工断点、确认校验。
- 执行留痕：每次构建有日志、参数、执行人、结果。
- 制品归档：申请文件、确认文件、比对报告、日志可追溯。
- 团队协作：测试、开发、运维看到同一套执行结果。

典型 stage：

1. `Checkout / Build`
2. `Prepare Environment`
3. `Generate Apply File`
4. `Manual TA Batch In GUI`
5. `Validate Confirmation`
6. `Archive Reports`

### 更稳版本

“我们用 Jenkins 的核心原因不是因为业务逻辑在 Jenkins 里，而是它适合做回归流程的统一编排和留痕。TA 自动化测试不是只跑一个 Java 程序，而是要把环境准备、申请文件生成、人工启动 TA 批处理、确认文件解析、数据库比对和报告归档串成一个标准流程。

一条 Pipeline 大概会分成几个 stage。第一是 checkout 和 build，保证本次用的是当前版本代码；第二是 prepare environment，通过脚本或 SQL 清理历史测试数据、准备产品、客户、申请和 expected result；第三是 generate apply file，由 Java 程序按场景生成申请文件并放到 TA 输入目录；第四是 manual TA batch，这里会保留一个人工检查点，让测试人员在 TA 图形化界面启动真实批处理；第五是 validate confirmation，读取 TA 输出的确认文件，解析后和 expected result 做字段级比对；最后归档申请文件、确认文件、比对报告和日志。

如果只让测试人员在本地跑 Java 或脚本，也能跑通单次流程，但问题是每个人的环境、参数、执行顺序可能不一致，执行记录也不容易追溯。Jenkins 可以统一入口、统一参数、统一执行顺序，并保留每次构建日志和报告，更适合版本升级后的回归验证。

如果某个 stage 失败，比如确认文件校验失败，我会先看 Jenkins 上失败的是哪个 stage，再看该 stage 的控制台日志和归档报告。确认校验失败时，我会根据批次号去看本次申请文件、确认文件、expected/actual 差异报告，再进一步判断是确认文件没生成、文件格式变化、expected 数据准备错，还是 TA 批处理结果确实不符合预期。这样失败结果是可追踪、可复盘的，而不是只停留在‘脚本失败了’。”

### 最短背诵版

“Jenkins 在这里负责流程编排和留痕，不负责 TA 业务逻辑。Pipeline 会串起 build、环境准备、申请文件生成、人工 TA 批处理断点、确认文件解析比对和报告归档。相比本地跑脚本，Jenkins 的价值是参数统一、顺序统一、日志可追踪、报告可归档。失败时先看失败 stage，再看日志、申请文件、确认文件和 expected/actual 报告，按批次号定位问题。”

### 记忆钩子

`参数化入口 -> stage 编排 -> 人工断点 -> 报告归档 -> 失败可追踪`

### Q3.3：确认文件解析模块怎么设计？

面试官追问点：

1. 输入和输出是什么？
2. 核心类或方法怎么拆？
3. 金额、日期、状态码如何转换？
4. 文件格式变化时怎么降低改动成本？

### 你的当前回答问题

- 你能想到输入是文件，但输出和模块拆分还没有形成固定表达。
- 这题要重点说出 `Parser -> Domain Object -> Comparator`，否则会显得只知道“读文件”。
- 不需要讲复杂设计模式，但要讲清边界：解析模块只负责把文件转成结构化记录，不负责判断 TA 业务是否正确。

### 回答思路

输入：

- 确认文件路径。
- 批次号、销售商、业务日期等上下文参数。

输出：

- `List<ActualConfirmation>` 这类结构化确认记录。
- 必要时输出解析错误信息或异常行号。

核心拆分：

| 类/模块 | 作用 |
|---|---|
| `ConfirmationFileLocator` | 按日期、销售商、批次号定位确认文件 |
| `ConfirmationFileParser` | 逐行读取文件并解析字段 |
| `ActualConfirmation` | 承载解析后的实际确认结果 |
| `ConfirmationComparator` | 将 actual 和 expected 做字段级比对 |
| `MarkdownReportGenerator` | 输出差异报告 |

字段转换：

- 金额/份额：转 `BigDecimal`，注意精度、补零、舍入。
- 日期：转 `LocalDate`，统一格式如 `yyyyMMdd`。
- 状态码/原因码：维护映射关系，不在业务代码里散落硬编码。
- 空值和空格：trim，并区分空字符串和真正缺失。

格式变化应对：

- 把文件字段位置、分隔符、日期格式、状态码映射配置化或集中管理。
- Parser 只负责格式解析，Comparator 只负责业务比对，避免耦合。
- 增加样例文件和单元测试，文件格式变更时先改解析规则，再跑回归。

### 更稳版本

“确认文件解析模块的输入一般是 TA 输出目录里的确认文件路径，以及本次测试批次号、业务日期、销售商等上下文参数。输出不是简单的字符串，而是一组结构化的 actual result，比如 `ActualConfirmation` 列表，每条记录包含申请编号、确认状态、确认金额、确认份额、确认日期、原因码和来源文件名。

模块上我会拆成几层。第一层是文件定位，比如根据业务日期、销售商或批次号找到本次确认文件；第二层是 parser，负责逐行读取文件，跳过表头或空行，按分隔符或定长规则拆字段；第三层是 domain object，把每行转换成结构化确认对象；第四层是 comparator，用申请编号等 key 把 actual 和 expected 对齐并逐字段比对；最后由 report generator 输出差异报告。

字段转换上，金额和份额会用 `BigDecimal`，避免用 double 带来精度问题；日期会统一转换成 `LocalDate`，比如按 `yyyyMMdd` 解析；状态码、原因码会维护映射关系，避免在代码里散落硬编码；每个字段解析时也要处理 trim、空值、补零和格式异常。

如果确认文件格式发生变化，比如字段顺序、分隔符、日期格式或状态码变了，我会尽量把这些格式规则集中在 parser 或配置里，而不是写死在比对逻辑里。这样文件格式变化时主要改解析层，后面的 expected/actual 比对和报告输出可以尽量复用。同时会保留样例文件和单元测试，确保格式调整后不会把正常场景解析坏。”

### 最短背诵版

“确认文件解析模块输入是确认文件路径和批次上下文，输出是结构化的 `ActualConfirmation` 列表。设计上会拆成文件定位、文件解析、领域对象、字段比对和报告输出几层。解析时金额/份额用 BigDecimal，日期转 LocalDate，状态码和原因码做统一映射。如果文件格式变了，尽量只改 parser 或配置，不影响 comparator 和报告层。”

### 记忆钩子

`File -> Parser -> ActualConfirmation -> Comparator -> Report`

### Q3.4：为什么自动化回归前要清理环境和准备数据？如果环境没清干净会怎样？怎么设计？

### 你的当前回答问题

- 你抓住了核心词：`幂等性` 和 `控制变量`，方向很好。
- 需要注意发音和表达：是 `幂等`，不是 `密等`。
- 回答还需要更具体：清理什么、准备什么、怎么隔离批次、怎么避免误删数据。
- 你提到了 TA 系统日期，这很重要，可以扩展成 `业务日期一致性`。

### 回答思路

为什么要清理：

- 保证同一个场景重复执行时结果一致。
- 避免上一次测试残留影响本次 expected/actual。
- 降低问题定位难度。

如果没清干净：

- 上一批申请或确认结果被本次误读。
- inbound/outbound 文件目录里有旧文件。
- 数据库里有旧的申请、确认、对账结果。
- TA 系统业务日期、产品状态、客户状态和申请文件日期不一致。
- 最终把环境问题误判成 TA 系统 bug。

怎么设计：

1. 用 `test_batch_no` 隔离每次测试。
2. 清理本批次历史数据：申请、expected、actual、compare_result、run_log。
3. 清理本批次文件：inbound 申请文件、outbound 确认文件、报告。
4. 准备确定性数据：产品、客户、风险有效期、净值、申请数据、预期确认结果。
5. 校验业务日期：申请文件日期、TA 系统日期、批次日期保持一致。
6. 记录环境准备日志，失败时直接中断，不进入后续跑批。

### 更稳版本

“自动化回归前清理环境和准备数据，核心是为了保证测试的幂等性和可复现性。也就是说，同一个测试场景在同一套前置条件下重复执行，应该得到稳定结果。TA 回归测试链路比较长，涉及申请文件、数据库状态、业务日期、产品状态、客户状态、确认文件和比对报告，如果前置环境不干净，很容易把上一次测试残留带进本次结果里。

如果环境没有清理干净，可能会出现几类问题。比如数据库里残留了上一批交易申请或 expected result，导致本次比对时取错数据；TA 输出目录里残留旧确认文件，工具误解析了旧文件；产品状态、客户风险有效期或净值数据不是本次场景需要的状态；还有一种常见问题是业务日期不一致，比如申请文件日期、TA 系统日期和测试批次日期对不上。这些都会导致自动化报告失败，但失败原因未必是 TA 系统 bug，而可能只是环境或造数问题。

设计上，我会首先用 test_batch_no 隔离每次测试。环境准备阶段按批次清理本次相关的申请数据、expected result、actual snapshot、compare result、run log，以及 inbound/outbound 目录里的本批次文件和报告。然后再写入确定性的测试数据，包括产品、客户、风险有效期、净值、交易申请和预期确认结果。同时会校验业务日期，确保申请文件、TA 系统日期和本次批次日期一致。最后环境准备阶段也要有日志，如果前置数据准备失败，就让 pipeline 在 prepare stage 失败，不继续进入跑批和校验。”

### 最短背诵版

“清理环境和准备数据是为了保证自动化回归的幂等性。否则旧申请、旧确认文件、旧 expected result、旧比对结果或业务日期不一致，都可能污染本次校验。设计上我会用 test_batch_no 做批次隔离，清理本批次数据库记录和文件目录，再准备产品、客户、净值、交易申请和 expected result，并校验申请日期、TA 系统日期和批次日期一致。”

### 记忆钩子

`幂等性 -> 批次隔离 -> 清库清文件 -> 固定造数 -> 日期一致 -> 日志可追踪`

### Q3.5：expected result 生成逻辑怎么设计？放 Java 还是 SQL？多场景怎么组织？

面试官追问点：

1. expected result 基于哪些输入数据生成？
2. 逻辑放 Java 还是 SQL，为什么？
3. 多场景如何组织，比如申购成功、产品关闭、客户风险过期？

### 你的当前回答问题

- 你提到了交易申请、产品维度、销售商维度、客户持有限制等输入，这是对的。
- 不能说 expected result “经过 TA 系统本身的校验逻辑生成”。因为 expected 的价值是独立于被测 TA 结果，用来验证 TA，而不是用 TA 验证 TA。
- Java vs SQL 的理由还没讲出来。
- 多场景配置可以理解成：不同测试场景对应不同前置数据和预期结果。

### 回答思路

expected result 的输入：

- 交易申请：申请编号、交易类型、金额、业务日期、销售商、客户、产品。
- 产品状态：是否开放、是否可申购、净值、产品工作日。
- 客户状态：风险等级、风险测评是否过期、账户状态。
- 销售商/额度/交易控制：销售商是否允许交易、额度、限额。
- 场景配置：本次测试希望覆盖成功、产品关闭、风险过期等哪类规则。

Java 还是 SQL：

- SQL 适合做数据准备、清理、查询和批量插入。
- Java 适合做场景编排、规则判断、金额/份额计算、可读性和单元测试。
- 如果规则非常简单，可以 SQL 生成；如果规则有分支和扩展，Java 更好维护。

多场景组织：

- 用 `scenarioId` 区分场景。
- 每个场景包含产品、客户、交易申请、expected result。
- 规则可以拆成小 validator 或策略：产品状态校验、客户风险校验、额度校验、金额份额计算。
- expected 和 actual 分开存储，按 `test_batch_no + request_no` 对齐。

### 更稳版本

“expected result 不是从 TA 跑批结果里反推出来的，而是在跑批前基于测试场景和业务规则准备出来的预期确认结果。它的输入包括交易申请数据，比如申请编号、交易类型、金额、客户、产品、销售商和业务日期；也包括产品维度数据，比如产品是否开放、是否允许申购、净值、产品工作日；还包括客户维度数据，比如风险测评是否有效、客户状态；如果场景更复杂，还会考虑销售商额度、持有限额和交易控制参数。

这段逻辑放 Java 还是 SQL，要看复杂度。SQL 适合做数据清理、批量插入和查询，比如清理本批次数据、插入产品和客户前置数据。但 expected result 的业务判断如果涉及比较多分支，比如产品关闭、客户风险过期、申购成功后按净值计算份额，我更倾向放在 Java 里。Java 的可读性、可测试性更好，也方便用单元测试覆盖不同场景。SQL 可以作为数据准备和落库手段，Java 负责场景编排和规则生成。

如果要支持多个场景，我会用 scenarioId 来组织，比如 `PURCHASE_SUCCESS`、`PRODUCT_CLOSED`、`RISK_EXPIRED`。每个场景会定义一组确定性的产品、客户、交易申请和 expected confirmation。申购成功场景里，产品是 OPEN、客户风险有效，expected status 是 SUCCESS，份额按金额除以净值计算；产品关闭场景里，产品状态是 CLOSED，expected status 是 FAILED，原因码是 PRODUCT_CLOSED；客户风险过期场景里，客户风险有效期早于业务日期，expected status 是 FAILED，原因码是 RISK_EXPIRED。

实现上可以把场景数据构造和规则判断集中在一个 scenario definition 或 rule service 里，最后把 expected result 落到 expected_confirmation 表。后续 TA 跑批生成 actual 后，再按批次号和申请编号做字段级比对。”

### 最短背诵版

“expected result 是跑批前根据交易申请、产品状态、客户状态、销售商/额度和业务规则生成的预期确认结果，不能从 TA 跑批结果反推。SQL 更适合清理和准备数据，Java 更适合写场景规则和金额份额计算，因为可读、可测试、好扩展。多场景用 scenarioId 管理，每个场景定义产品、客户、申请和 expected confirmation，最后按批次号和申请编号与 actual 比对。”

### 记忆钩子

`expected 独立于 TA -> 申请/产品/客户/额度 -> Java 规则 + SQL 落库 -> scenarioId -> expected_confirmation`

## 4. TA 批前预警 / 静态参数稽核

### Q4.1：请按“业务问题 -> 系统方案 -> 架构/数据流 -> 你做的开发/技术工作 -> 测试验收 -> 价值”讲 TA 批前预警 / 静态参数稽核

### 你的当前回答问题

- 业务背景讲得真实：跑批中才发现交易超限或参数漏配，处理很被动。
- 已经区分出两类问题：交易类预警和静态参数稽核。
- 术语要稳定：说 `静态参数稽核`，不要说成“静态参数集合”。
- 不要只说“TA 会卡批”，要补清楚系统方案、数据流、规则落地、异常结果闭环。
- “预清算”这个词要谨慎，可以说 `批前检查` 或 `批前预校验`，避免让面试官误以为你们真的提前做清算。

### 回答思路

固定六段：

1. 业务问题：跑批中才发现交易规模、额度、账号、费率、开放期等问题，处理窗口很紧。
2. 系统方案：把检查前移到跑批前，分为交易类批前预警和静态参数稽核。
3. 架构/数据流：TA/产品/账户/费率/销售商数据 -> 规则执行 -> 异常结果表 -> 页面/报表/通知 -> 业务修正/复核。
4. 我的工作：规则清单、口径定义、检查项拆分、静态参数稽核落地、验证。
5. 测试验收：正常数据不误报、异常数据能报、历史问题回放、业务确认。
6. 价值：风险前置、减少夜间跑批中断、降低人工排查成本。

### 更稳版本

“TA 批前预警和静态参数稽核解决的是 TA 夜间跑批前很多风险发现太晚的问题。TA 跑批时如果发现交易规模、销售额度、持有限额、产品状态或者关键参数配置有问题，就可能导致卡批，业务和科技需要在夜间临时判断和处理。还有一类问题是静态参数漏配或错配，比如新产品的托管账号、销售商清算账号、费率、开放期、赎回资金账号等没有维护完整，白天没发现，晚上跑批时才暴露，处理窗口非常紧。

我在这条项目里主要参与需求讨论、稽核规则清单梳理、交易类预警和静态参数稽核的边界拆分、字段口径定义，以及自研静态参数稽核检查项的落地验证。技术上我重点关注的是把人工检查项转成可执行规则，比如明确适用范围、字段来源、判断条件、异常结果如何落表和展示。

测试验收时，我们会用正常数据验证不会误报，用缺账号、缺费率、开放期异常、产品状态冲突等模拟数据验证能报出来，也会用历史跑批中出现过的问题做回放，确认规则能提前识别。最终价值是把原来跑批中才发现的问题前置到跑批前，减少夜间临时处理和卡批风险，也让运营控制从人工经验检查变成系统化、可追踪的规则检查。”

### 最短背诵版

“TA 批前预警和静态参数稽核的核心是把跑批中才暴露的问题前置到跑批前。它分两类：交易类批前预警看份额、申购规模、募集规模、销售额度、持有限额；静态参数稽核看账号、费率、开放期、日期、产品状态、销售商配置。数据从 TA、产品、账户、费率、销售商等来源进入规则执行层，异常结果落表后通过页面或报表给业务处理。我主要参与规则清单、口径定义、检查项系统化和验证，价值是降低卡批和夜间排查成本。”

### 记忆钩子

`跑批中暴露太晚 -> 批前检查 -> 交易预警 + 静态稽核 -> 异常落表 -> 业务修正闭环`

### Q4.2：如何设计“销售商清算账号缺失”的静态参数稽核规则？

面试官追问点：

1. 这条规则检查哪些数据？
2. 判断条件是什么？
3. 异常结果表记录哪些字段？
4. 怎么避免误报？

### 你的当前回答问题

- 你已经抓住核心：用 `产品-销售商关系表` 和 `销售商清算账号表` 做关联校验。
- 需要补充规则适用范围，否则容易误报。例如只检查有效产品、有效销售关系、需要清算账号的销售商。
- SQL 思路要讲成 `left join + is null`，但要强调是示意，不需要背真实表名。
- 异常结果表要说清楚记录什么，方便业务定位和修正。

### 回答思路

检查数据：

- 产品基础信息：产品代码、产品状态、产品类型、是否可销售/存续。
- 产品销售商关系：产品关联了哪些销售商，关系是否有效。
- 销售商清算账号信息：销售商代码、账号类型、账号状态、生效日期、失效日期。
- 必要时还有交易控制或销售状态。

判断条件：

- 产品处于需要检查的状态，比如已成立、可销售、存续或开放。
- 产品和销售商关系有效。
- 该销售商/产品关系需要有清算账号。
- 在账号表里找不到有效清算账号，或者账号状态无效、已过期。

### SQL 思路示意

```sql
select
    ps.product_code,
    p.product_name,
    ps.seller_code,
    ps.seller_name,
    'SELLER_CLEARING_ACCOUNT_MISSING' as rule_code,
    '销售商清算账号缺失' as rule_name,
    '产品已关联销售商，但未维护有效清算账号' as exception_reason
from product_seller_relation ps
join product_info p
    on p.product_code = ps.product_code
left join seller_clearing_account a
    on a.seller_code = ps.seller_code
   and a.account_type = 'CLEARING'
   and a.account_status = 'VALID'
   and a.effective_date <= :batch_date
   and (a.expire_date is null or a.expire_date >= :batch_date)
where p.product_status in ('OPEN', 'ACTIVE', 'IN_OPERATION')
  and ps.relation_status = 'VALID'
  and ps.effective_date <= :batch_date
  and (ps.expire_date is null or ps.expire_date >= :batch_date)
  and a.seller_code is null;
```

如果账号是按 `产品 + 销售商` 维度维护，则 join 条件里再加：

```sql
and a.product_code = ps.product_code
```

### 异常结果表字段

| 字段 | 作用 |
|---|---|
| `batch_no` / `check_date` | 本次稽核批次和日期 |
| `rule_code` / `rule_name` | 哪条规则命中 |
| `product_code` / `product_name` | 方便定位产品 |
| `seller_code` / `seller_name` | 方便定位销售商 |
| `missing_field` | 缺失字段，如清算账号 |
| `exception_reason` | 异常原因 |
| `severity` | 风险等级 |
| `source_table` | 来源表，方便科技排查 |
| `handle_status` | 待处理、已修正、已豁免 |

### 怎么避免误报

- 只检查有效产品和有效销售关系，不检查已终止、发行失败、未启用销售商关系。
- 校验账号生效日期、失效日期和账号状态，避免把未生效或已失效账号当成有效。
- 区分账号维护维度：有些账号按销售商维护，有些按产品+销售商维护，不要 join 错层级。
- 区分账号类型：清算账号、赎回资金账号、电子账户不是一回事。
- 对业务确认的特殊场景设置豁免或白名单，并记录原因和有效期。

### 更稳版本

“这条规则的核心是检查：产品已经关联了某个销售商，并且这段产品-销售商关系是有效的，但系统里没有维护有效的销售商清算账号。

数据上至少需要三类表。第一是产品基础信息，判断产品是不是处于可销售、存续或需要跑批检查的状态；第二是产品和销售商关系表，确认这个产品关联了哪些销售商，以及关系是否有效；第三是销售商清算账号表，检查对应销售商是否维护了有效清算账号，包括账号类型、账号状态、生效日期和失效日期。

SQL 思路上一般是从产品销售商关系表出发，关联产品表过滤有效产品，再 left join 销售商清算账号表。如果 left join 后账号为空，就说明这个产品关联的销售商缺少有效清算账号。如果账号是按产品和销售商维度维护，就 join 产品代码和销售商代码；如果账号是销售商层面统一维护，就只按销售商代码和账号类型关联。

异常结果表里要记录本次检查批次、规则编码、产品代码、产品名称、销售商代码、销售商名称、缺失字段、异常原因、风险等级和处理状态。这样业务看到后能知道到底是哪只产品、哪个销售商、缺了什么账号。

为了避免误报，需要控制规则适用范围。比如已终止产品、发行失败产品、无效销售关系不检查；账号状态必须有效，且生效日期和失效日期覆盖当前批次日；还要区分清算账号、赎回资金账号、电子账户等账号类型。如果业务有特殊豁免，也要有白名单和有效期，而不是简单忽略。”

### 最短背诵版

“销售商清算账号缺失规则，本质是用产品销售商关系表 left join 销售商清算账号表。先过滤有效产品和有效销售关系，再查是否存在有效清算账号；如果不存在，就把产品、销售商、缺失字段、规则编码、异常原因和批次写入异常结果表。避免误报的关键是控制适用范围、账号状态、生效失效日期、账号类型和业务豁免。”

### 记忆钩子

`产品-销售商关系 -> left join 清算账号 -> is null -> 异常落表 -> 范围/状态/日期/类型防误报`

### Q4.3：规则上线后业务反馈误报很多，怎么处理？

### 你的当前回答问题

- 你能想到先查数据、再和业务确认口径、再调整查询条件，方向正确。
- “已经产生的误报在表里删掉”不建议现场说。金融系统更重视审计留痕，应该改成 `标记为误报/已豁免/已关闭`，并记录原因。
- 需要把误报原因分类讲清：数据问题、口径问题、规则适用范围问题、特殊豁免场景。

### 回答思路

定位误报原因：

1. 抽样看误报记录，确认产品、销售商、账号、规则命中原因。
2. 查源数据：产品状态、销售关系、账号状态、生效日期、失效日期。
3. 查规则条件：是否把不该检查的产品或销售关系纳入了。
4. 查业务例外：是否存在特殊销售商、特殊产品、统一账号、暂不需要账号的场景。

重新确认口径：

- 哪些产品状态要检查。
- 哪些销售关系要检查。
- 清算账号是销售商维度还是产品+销售商维度。
- 哪些账号类型算有效。
- 特殊场景是否允许豁免。

技术调整：

- 收窄 where 条件。
- 增加账号状态和有效期判断。
- 区分账号类型。
- 增加白名单/豁免表。
- 增加规则版本号，方便追踪调整前后结果。

历史误报处理：

- 不直接物理删除。
- 标记为误报、已关闭或已豁免。
- 记录处理人、处理时间、处理原因和规则版本。

### 更稳版本

“如果业务反馈误报很多，我不会直接先改代码，而是先抽样看误报记录，确认这些记录为什么被规则命中。比如看产品状态、销售商关系、账号表里是否真的没有有效清算账号，以及异常结果里记录的当前值和缺失字段是否准确。

然后我会从几类原因排查。第一是源数据问题，比如账号其实已经维护了，但账号状态、生效日期、失效日期不符合规则条件；第二是规则适用范围过宽，比如把已终止产品、无效销售关系、还未启用的销售商关系也纳入检查；第三是业务口径问题，比如有些销售商使用统一清算账号，不是按产品维度维护；第四是特殊豁免场景，比如某些产品或销售商暂时不需要这类账号。

和业务重新确认口径时，我会把误报样本拿出来一起过，明确哪些产品状态需要检查、哪些销售关系算有效、账号是按销售商维度还是产品+销售商维度维护、哪些账号类型算清算账号，以及是否需要白名单或豁免机制。

技术调整上，可以收窄 SQL 的 where 条件，补充账号状态、生效日期、失效日期、账号类型判断；如果确实存在业务例外，就增加白名单或豁免表，并设置有效期和原因。规则调整后用误报样本和正常样本重新跑一遍，确认误报下降，同时没有引入漏报。

对于已经产生的误报结果，我不会直接物理删除，而是把它们标记为误报、已关闭或已豁免，记录处理人、处理时间、处理原因和规则版本。这样既能避免业务继续处理无效预警，也保留了稽核和规则调整的审计痕迹。”

### 最短背诵版

“误报多时，我会先抽样看异常记录和源数据，再区分是数据问题、规则范围过宽、业务口径不一致，还是特殊豁免场景。和业务确认产品状态、销售关系、账号维度、账号类型和豁免口径后，再调整 SQL 条件或增加白名单。已经产生的误报不直接删除，而是标记为误报/已关闭/已豁免，并记录原因和处理人，保留审计痕迹。”

### 记忆钩子

`抽样 -> 查源数据 -> 对业务口径 -> 改规则/白名单 -> 标记关闭不物理删除`

### Q4.4：如果销售商清算账号缺失导致跑批卡住，但稽核规则没有提前报出来，怎么排查漏报？

### 你的当前回答问题

- “和误报一样的原则”方向没错，但现场不够具体。
- 漏报比误报更严重，因为它会让业务误以为批前检查通过。
- 需要按 `数据覆盖 -> 规则范围 -> 执行链路 -> 结果展示` 来排查。

### 回答思路

漏报排查四层：

1. 数据是否取全：产品、销售商关系、账号数据是否都进入稽核范围。
2. 规则范围是否过窄：产品状态、销售关系状态、账号维度、账号类型是否过滤错。
3. 规则是否执行：当天批次是否跑了这条规则，是否执行失败但没有告警。
4. 结果是否丢失：异常是否落表、是否被过滤、是否在页面/报表/通知展示。

修复方式：

- 补齐数据源或关联条件。
- 扩大规则适用范围。
- 增加执行日志和规则失败告警。
- 用历史问题回放验证修复有效。
- 增加回归用例，防止再次漏报。

### 更稳版本

“漏报和误报一样都要回到数据、口径和规则链路排查，但我会更重视漏报，因为漏报意味着系统没有把真实风险提前暴露出来。

我会先看数据是否取全。比如这只产品是否在产品基础表里，产品-销售商关系是否进入了稽核数据范围，销售商账号表是不是同步完整。如果源数据没有进入规则范围，规则本身再正确也报不出来。

第二看规则适用范围是否过窄。比如规则只检查了某几类产品状态，但这只产品实际处于另一个也应该检查的状态；或者只按销售商维度查账号，但实际账号是按产品+销售商维度维护；也可能账号类型判断不对，把赎回资金账号、清算账号、电子账户口径混淆了。

第三看规则当天是否正常执行。比如这条稽核规则有没有被调度起来，执行日志里是否成功，SQL 是否报错，是否因为某个异常导致整条规则没有产出结果。

第四看结果链路是否丢失。也就是规则其实识别出了异常，但异常结果没有正确落表，或者落表后被页面过滤、报表过滤、通知过滤掉了。

修复上，我会先根据这次漏报定位原因。如果是数据源问题，就补同步或修关联条件；如果是规则范围问题，就调整产品状态、销售关系、账号类型等条件；如果是执行链路问题，就增加规则执行日志和失败告警；如果是展示问题，就修正结果过滤逻辑。修完以后要用这次真实漏报案例做历史回放，确认规则能报出来，同时加入回归用例，避免后续再次漏报。”

### 最短背诵版

“漏报比误报更危险。我会按四层查：第一数据是否取全，第二规则范围是否过窄，第三规则当天是否执行成功，第四异常结果是否在落表或展示环节丢失。修复后用这次真实案例做历史回放，并加入回归用例，确认以后同类问题能提前报出来。”

### 记忆钩子

`漏报四层：数据没进来 -> 规则没覆盖 -> 执行没跑成 -> 结果没展示`

## 5. 需求、测试、验收和协作流程

### Q5.1：如果业务提出新增一条静态参数稽核规则，从需求提出到上线验收怎么走？

### 你的当前回答问题

- 你有很强的项目推进视角，知道要拉齐业务、产品 owner、开发/供应商、测试。
- 需要从开发岗角度补充：规则口径、数据来源、异常结果、测试案例、上线验证。
- 不要把重点都放在排期和每日跟进上，要讲清“需求如何变成可开发、可测试、可验收的系统规则”。
- “需求范围定好就不要变动”可以说成“控制需求边界和变更流程”，更成熟。

### 回答思路

按阶段讲：

1. 需求提出：业务说明风险场景、历史问题、希望检查什么。
2. 需求澄清：确认适用范围、字段来源、判断条件、异常文案、处理责任人。
3. 方案设计：数据源、规则逻辑、异常结果表、展示/通知、豁免机制。
4. 开发实现：SQL/后端规则、调度、落表、页面/报表。
5. SIT：开发/测试用正常、异常、边界数据验证。
6. UAT：业务确认规则口径、异常文案、处理流程。
7. 上线：发布、跑首批、观察结果、问题回收。
8. 验收闭环：测试报告、业务确认、上线记录、后续优化项。

### 更稳版本

“如果业务提出新增一条静态参数稽核规则，我会先把它从业务描述转成可开发、可测试、可验收的规则。

第一步是需求提出和澄清。业务需要说明这条规则解决什么风险，比如历史上是否因为某个参数漏配导致跑批失败。我们会一起确认适用范围，比如检查哪些产品状态、哪些销售关系、哪些账号类型；也要确认字段来源、判断条件、异常文案、处理责任人和是否允许豁免。

第二步是方案设计。技术侧会评估需要哪些数据源，是查 TA 表、产品表、销售商表还是账号表；规则用 SQL、后端代码还是配置化方式实现；异常结果落到哪张表，页面或报表怎么展示，是否需要通知，是否需要白名单或豁免机制。

第三步是开发实现。开发会实现规则查询或规则服务，把命中的异常写入异常结果表，同时记录批次、规则编码、产品、销售商、异常字段、异常原因和处理状态。如果涉及页面，还要增加查询、筛选和处理入口。

第四步是 SIT。测试会准备正常数据、异常数据和边界数据。比如正常产品不应该报，缺账号产品应该报，已终止产品不应该报，特殊豁免产品不应该报。SIT 重点验证规则逻辑、落表结果和页面展示。

第五步是 UAT。业务根据真实口径确认规则是否准确、异常文案是否看得懂、处理流程是否可执行。如果 UAT 发现误报或漏报，就回到口径和规则条件调整。

最后是上线和验收。上线后会关注首批执行结果，抽样看异常是否符合预期，必要时和业务一起做首日观察。验收材料包括测试报告、业务确认、上线记录和后续优化项。整个过程里最关键的是把业务风险点转成明确的数据字段、判断条件和验收标准。”

### 最短背诵版

“新增稽核规则时，我会先和业务确认风险场景、适用范围、字段来源、判断条件和异常文案；技术上设计数据源、规则逻辑、异常结果表和展示/通知方式；开发后用正常、异常、边界和豁免场景做 SIT；再让业务 UAT 确认口径和处理流程；上线后观察首批结果并形成验收记录。核心是把业务风险点转成可开发、可测试、可验收的规则。”

### 记忆钩子

`风险场景 -> 规则口径 -> 技术方案 -> SIT 样例 -> UAT 口径 -> 上线观察`

### Q5.2：业务需求很模糊，只说“希望系统提前发现产品参数配置问题”，你怎么推进？

### 更稳版本

“如果业务一开始提得比较模糊，比如只说希望系统提前发现产品参数配置问题，我不会直接进入开发，也不会简单把需求退回去。我会先帮助业务把这个需求结构化。

具体做法是，先和业务一起还原历史问题：过去到底哪些参数配置问题导致过跑批失败或清算异常，比如账号缺失、费率缺失、开放期设置错误、产品状态和交易控制不一致等。然后我会把这些问题拆成几个检查对象，比如产品基础信息、销售商关系、账户信息、费率参数、开放期、日期和状态。

接下来我会把模糊需求变成一个问题清单，类似填空题，让业务逐项确认：这条规则检查哪些产品状态，字段来源是哪张表或哪个系统，什么情况下算异常，异常结果给谁处理，是否允许豁免，UAT 时用什么样例证明规则正确。

如果业务暂时说不清所有规则，我会建议先做 MVP，优先选择历史上真实发生过、风险高、口径清楚的几条规则先落地。对于还没有确认清楚的部分，记录为待确认项或后续优化项，不放进当前开发范围。这样既能推进项目，也能避免因为需求口径不清导致后续误报、漏报或者反复返工。”

### 最短背诵版

“业务需求模糊时，我会先帮业务把需求结构化，而不是直接开发。先复盘历史问题，再拆成产品、销售商、账户、费率、开放期、日期、状态这些检查对象；然后用问题清单确认字段来源、异常标准、适用范围、处理人和验收样例。口径清楚的高风险规则先做 MVP，没确认清楚的记录为待确认或后续优化，不进入当前开发范围。”

### 记忆钩子

## 6. 技术基础

### Q6.1：Docker 容器化的基本原理是什么？它和传统虚拟机有什么区别？

面试官追问点：

1. Docker 解决什么问题？
2. 容器和虚拟机区别是什么？
3. 为什么容器启动更快、资源更轻？
4. Java/Spring Boot 项目里 Docker 怎么用？

### 你的当前回答问题

- 你已经说出了核心：Docker 解决环境一致性和可迁移性；容器共享宿主机内核；比虚拟机更轻；Spring Boot 可以通过 Dockerfile 做镜像，再 `docker run` 启动容器。
- 需要补准底层关键词：`namespace` 做隔离，`cgroups` 做资源限制，镜像分层/Union FS 做文件系统复用。
- “容器关闭后就没有了”要说得更精确：容器可写层默认随容器生命周期变化，持久化数据要放 volume 或外部存储。
- 不要说“反正它可以挂载”，现场换成 `volume bind mount` 或 `volume`。

### 回答思路

Docker 解决：

- 环境一致性。
- 依赖打包。
- 部署可复制。
- 本地、测试、生产环境差异降低。

容器 vs 虚拟机：

| 对比 | 虚拟机 | Docker 容器 |
|---|---|---|
| 隔离层 | 虚拟化硬件 + Guest OS | 操作系统层隔离 |
| 内核 | 每个 VM 有自己的 OS 内核 | 共享宿主机内核 |
| 启动速度 | 慢，像启动一台机器 | 快，像启动一个进程 |
| 资源占用 | 更重 | 更轻 |

基本原理：

- `namespace`：隔离进程、网络、文件系统、用户等视图。
- `cgroups`：限制和统计 CPU、内存、IO 等资源。
- `image layers` / 联合文件系统：镜像分层复用，容器启动时加一层可写层。
- `volume`：持久化数据，不依赖容器可写层。

### 更稳版本

“Docker 主要解决的是应用运行环境一致性和部署可复制的问题。以前一个 Java 服务在本地能跑，到测试或生产环境可能因为 JDK 版本、系统依赖、配置路径不同出问题。Docker 可以把应用、运行时和依赖打包成镜像，让它在不同环境里用相对一致的方式启动。

容器和传统虚拟机最大的区别是隔离层不同。虚拟机是在硬件虚拟化之上跑一套完整的 Guest OS，每个虚拟机都有自己的操作系统内核，所以更重、启动更慢。Docker 容器是在操作系统层做隔离，多个容器共享宿主机内核，通过 namespace 隔离进程、网络、文件系统等视图，通过 cgroups 限制 CPU、内存、IO 等资源，所以更轻，启动也更快，更像启动一个隔离的进程。

Docker 镜像本身是分层的，基础镜像、运行环境、应用 jar 包可以一层层叠加。容器启动时会在镜像只读层之上加一层可写层。容器删除后，这个可写层的数据通常也会丢失，所以真正需要持久化的数据，比如数据库文件、日志或上传文件，一般会放到 volume 或外部存储里。

在 Java/Spring Boot 项目里，常见做法是先用 Maven 或 Gradle 把项目打成 jar，再写 Dockerfile，指定基础 JDK/JRE 镜像，把 jar 拷进去，暴露端口，设置启动命令。构建镜像后，通过 docker run 或在 CI/CD 里部署启动容器。这样测试环境和生产环境可以用同一套镜像，减少环境差异导致的问题。”

### 最短背诵版

“Docker 解决的是应用和依赖打包、环境一致性和部署可复制。虚拟机是硬件虚拟化，每个 VM 有完整 Guest OS；容器是操作系统层隔离，共享宿主机内核，通过 namespace 做隔离，通过 cgroups 做资源限制，所以更轻、启动更快。Spring Boot 项目里通常先打 jar，再写 Dockerfile 构建镜像，运行时启动成容器；需要持久化的数据放 volume 或外部存储。”

### 记忆钩子

`环境一致性 -> 共享内核 -> namespace 隔离 -> cgroups 限制 -> 镜像分层 -> volume 持久化`

### Q6.2：Spring Boot 里一个 HTTP 请求从前端进来，到返回结果经历哪些步骤？

面试官追问点：

1. Controller、Service、Mapper/DAO 各自负责什么？
2. 参数校验一般放在哪里？
3. 事务一般放在哪里？
4. 为什么不建议 Controller 直接写 SQL 或复杂业务逻辑？

### 你的当前回答问题

- Controller、Service、Mapper/DAO 的基本职责答对了。
- 参数校验放 Controller 层是合理的，但要补充：简单格式校验在 Controller/DTO，业务校验在 Service。
- 事务一般放 `Service` 层，不是 DAO 层。因为一个业务操作可能调用多个 DAO，需要在业务边界统一提交或回滚。
- Controller 不写 SQL 的理由可以讲成：职责分离、可维护、可测试、复用性和事务边界清晰。

### 回答思路

请求链路：

`HTTP Request -> Controller -> DTO 参数校验 -> Service 业务逻辑/事务 -> Mapper/DAO 数据访问 -> Service 组装结果 -> Controller 统一响应 -> ExceptionHandler 统一异常`

分层职责：

| 层 | 职责 |
|---|---|
| Controller | 接收请求、解析参数、基础校验、调用 Service、返回统一响应 |
| Service | 业务逻辑、业务校验、事务边界、调用多个 DAO/外部接口 |
| Mapper/DAO | 数据库访问，执行 SQL 或 ORM 操作 |
| DTO/VO | 请求参数和返回结果模型 |

### 更稳版本

“一个 HTTP 请求进来以后，通常先进入 Controller。Controller 负责接收请求、解析参数，比如路径参数、query 参数或 request body，并做基础参数校验，比如必填、格式、长度、分页范围等。校验通过后，Controller 调用 Service 层。

Service 层负责业务逻辑和业务校验。比如查询生命周期提醒列表时，Service 会判断产品是否存在、查询范围是否合理、是否需要过滤已关闭记录，也可能组合多个 Mapper 查询结果。事务一般也放在 Service 层，因为一个业务操作可能涉及多张表或多个 DAO，需要在同一个业务边界里统一提交或回滚。

Mapper 或 DAO 层负责和数据库交互，执行具体 SQL 或 ORM 操作。它不应该承载复杂业务规则，只负责把数据查出来、写进去或更新状态。

查询结果返回后，Service 会组装成前端需要的 DTO 或 VO，再由 Controller 返回统一响应格式。如果过程中出现参数错误、业务异常或数据库异常，通常会通过统一异常处理，比如 `ControllerAdvice`，返回标准错误码和错误信息。

不建议在 Controller 里直接写 SQL 或复杂业务逻辑，是因为这样会导致职责混乱，代码难复用、难测试，也不利于事务控制和后续维护。分层以后，Controller 轻，Service 管业务，DAO 管数据，结构会清楚很多。”

### 最短背诵版

“请求进来先进 Controller，负责接参、基础校验和调用 Service；Service 负责业务逻辑、业务校验和事务边界；Mapper/DAO 负责数据库访问。简单格式校验放 Controller/DTO，业务校验放 Service，事务通常也放 Service。Controller 不直接写 SQL，是为了职责清晰、方便维护测试，也避免事务和业务逻辑混乱。”

### 记忆钩子

`Controller 接请求 -> Service 管业务和事务 -> DAO 查库 -> DTO/VO 返回 -> 统一异常`

### Q6.3：提醒记录表如何设计索引？查询变慢怎么排查？

题目：

`reminder_record(id, product_code, reminder_type, reminder_date, status, created_at)`

前端经常按：

`product_code + reminder_type + reminder_date 范围`

查询提醒列表。

### 回答思路

核心索引：

```sql
create index idx_reminder_product_type_date
on reminder_record(product_code, reminder_type, reminder_date);
```

为什么：

- `product_code`：通常筛选产品，区分度较高。
- `reminder_type`：同一个产品下再按提醒类型过滤。
- `reminder_date`：最后做范围查询。

注意：

- 联合索引遵循最左前缀原则。
- 等值条件放前面，范围条件放后面。
- 如果范围字段后面还有字段，后面的字段通常不能继续充分利用索引做过滤。

如果查询还经常按 `status` 过滤：

```sql
create index idx_reminder_product_type_status_date
on reminder_record(product_code, reminder_type, status, reminder_date);
```

前提是 status 确实经常作为查询条件，否则别盲目加。

查询变慢排查：

1. 看 SQL 是否命中索引：用 `explain`。
2. 看 where 条件是否符合索引顺序。
3. 看是否对索引字段用了函数或隐式类型转换。
4. 看返回数据量是否太大，是否需要分页。
5. 看是否 `select *`，能否只查必要字段。
6. 看数据量是否增长，需要归档、分区或冷热数据拆分。
7. 看索引是否过多导致写入慢。

### 更稳版本

“如果前端经常按 product_code、reminder_type 和 reminder_date 范围查询，我会优先考虑建一个联合索引，比如 `(product_code, reminder_type, reminder_date)`。原因是 product_code 和 reminder_type 通常是等值过滤，reminder_date 是范围查询。联合索引里一般把等值条件放前面，范围条件放后面，这样能更好利用索引。

如果业务还经常按 status 过滤，比如只查待处理提醒，那可以评估建 `(product_code, reminder_type, status, reminder_date)`。但索引不是越多越好，索引会占空间，也会影响 insert 和 update，所以要根据真实查询频率和数据分布决定。

如果后续查询变慢，我会先用 explain 看执行计划，确认是否走了预期索引，有没有全表扫描。然后看 SQL 条件是否符合联合索引的最左前缀，是否对 product_code 或 reminder_date 这类索引字段做了函数处理，是否存在类型不一致导致隐式转换。再看返回数据量是不是太大，是否需要分页、限制日期范围或者只查必要字段。如果数据量持续增长，还可以考虑按日期归档历史数据，或者做分区、冷热数据拆分。”

### 最短背诵版

“这类查询我会建联合索引 `(product_code, reminder_type, reminder_date)`，因为前两个是等值过滤，日期是范围查询。联合索引要注意最左前缀和等值在前、范围在后。如果还经常按 status 查，可以评估把 status 放到日期前。查询慢时先用 explain 看有没有走索引，再查索引顺序、函数/隐式转换、返回数据量、分页、select 字段和历史数据归档。”

### 记忆钩子

`等值在前 -> 范围在后 -> explain -> 最左前缀 -> 分页/归档`

### Q6.4：哪些情况下 SQL 建了索引但没走索引，或者索引失效？

### 最好记的 6 种场景

#### 1. 联合索引不满足最左前缀

索引：

```sql
(product_code, reminder_type, reminder_date)
```

能较好命中：

```sql
where product_code = ?
  and reminder_type = ?
```

不一定能很好命中：

```sql
where reminder_type = ?
```

因为没有从最左边的 `product_code` 开始用。

#### 2. 对索引字段做函数或计算

不推荐：

```sql
where date_format(reminder_date, '%Y-%m') = '2026-05'
```

更推荐：

```sql
where reminder_date >= '2026-05-01'
  and reminder_date < '2026-06-01'
```

原因：对索引字段做函数后，数据库可能无法直接利用原索引顺序。

#### 3. 字段类型不一致，发生隐式转换

如果 `product_code` 是字符串，但 SQL 写成：

```sql
where product_code = 1001
```

数据库可能做隐式类型转换，影响索引使用。

更稳：

```sql
where product_code = '1001'
```

#### 4. `like` 前面带 `%`

可能无法有效走普通 B+Tree 索引：

```sql
where product_name like '%目标盈'
```

相对更容易利用索引：

```sql
where product_name like '目标盈%'
```

#### 5. 范围条件后面的列利用不充分

索引：

```sql
(product_code, reminder_date, status)
```

SQL：

```sql
where product_code = ?
  and reminder_date between ? and ?
  and status = ?
```

`reminder_date` 是范围条件，后面的 `status` 可能不能充分利用索引过滤。

所以常见设计是把等值条件放前面，范围条件放后面：

```sql
(product_code, status, reminder_date)
```

#### 6. 返回数据量太大，优化器认为全表扫描更划算

即使有索引，如果条件区分度很低，比如查出全表很大比例数据，数据库可能认为走索引再回表不如直接全表扫描。

例子：

```sql
where status = 'ACTIVE'
```

如果大部分记录都是 `ACTIVE`，单独给 `status` 建索引效果可能很一般。

### 更稳版本

“索引不是建了就一定会走。常见情况有几类。第一是联合索引没有满足最左前缀，比如索引是 product_code、reminder_type、reminder_date，但查询只按 reminder_type 查，就不一定能很好利用这个索引。第二是对索引字段做函数或计算，比如对 reminder_date 做 date_format，这会影响数据库利用原索引顺序。第三是字段类型不一致导致隐式转换，比如字符串字段用数字去比较。第四是 like 查询前面带百分号，比如 `%目标盈`，普通 B+Tree 索引很难利用。第五是联合索引里范围条件后面的列可能利用不充分，所以一般等值条件放前面，范围条件放后面。第六是如果返回数据量太大，优化器可能认为全表扫描比走索引更划算。

所以排查时我会用 explain 看执行计划，确认有没有走预期索引，再看 SQL 条件、字段类型、函数计算、like 写法和返回数据量。”

### 最短背诵版

“索引失效常见几种：联合索引不满足最左前缀；对索引字段做函数或计算；字段类型不一致导致隐式转换；like 前面带 `%`；范围条件后面的列利用不充分；以及返回数据量太大，优化器认为全表扫描更划算。排查时用 explain 看执行计划。”

### 记忆钩子

`最左前缀、函数计算、类型转换、前置%、范围后列、大量数据`

### Q6.5：Java 里 HashMap 的基本原理是什么？项目里有没有类似场景？

面试官追问点：

1. HashMap 怎么存 key-value？
2. 为什么查找通常比较快？
3. hash 冲突怎么办？
4. 为什么 equals 和 hashCode 要一起重写？
5. 项目里有没有用 Map 做数据对齐或比对？

### 回答思路

HashMap 原理：

- 底层可以理解成 `数组 + 链表/红黑树`。
- put 时先算 key 的 hash，再定位数组下标。
- 如果下标为空，直接放。
- 如果下标已有元素，说明 hash 冲突，再用 equals 比较 key。
- Java 8 后，同一个桶里链表太长时会转红黑树，提高极端情况下查询效率。

为什么快：

- 正常情况下通过 hash 可以直接定位到桶，平均时间复杂度接近 O(1)。
- 但如果 hash 冲突严重，性能会下降。

equals 和 hashCode：

- hashCode 决定对象放在哪个桶。
- equals 决定两个 key 是否真正相等。
- 如果两个对象 equals 相等，它们的 hashCode 必须相等，否则 HashMap 可能找不到原来的 key。

项目场景：

- TA 自动化测试 expected/actual 比对。
- 用 `requestNo` 或 `requestNo + productCode + sellerCode` 作为 key。
- 把 expected result 先放进 Map。
- 遍历 actual result 时按 key 找对应 expected，再逐字段比对。

### 更稳版本

“HashMap 是 Java 里常用的 key-value 数据结构。底层可以简单理解为数组加链表或红黑树。put 一个元素时，会先根据 key 计算 hash 值，再根据 hash 定位到数组里的某个桶。如果这个桶为空，就直接放进去；如果桶里已经有元素，就说明发生了 hash 冲突，这时会继续用 equals 判断 key 是否真正相等。如果是同一个 key，就覆盖 value；如果不是，就挂到同一个桶的链表或树结构里。Java 8 以后，如果同一个桶里的链表太长，会转成红黑树，避免极端冲突时查询退化太严重。

HashMap 查找通常比较快，是因为正常情况下可以通过 hash 直接定位到桶，平均复杂度接近 O(1)。但如果 hash 分布很差，大量 key 落到同一个桶里，性能就会下降。

重写 equals 时通常也要重写 hashCode，是因为 HashMap 先用 hashCode 定位桶，再用 equals 判断 key 是否相等。如果两个对象 equals 相等，但 hashCode 不相等，它们可能被放到不同桶里，后续用相等对象去 get 时就找不到原来的值。

在 TA 自动化测试里，expected/actual 比对就很适合用 Map。比如把 expected confirmation 按申请编号，或者申请编号加销售商、产品代码组成 key，先放到 Map 里。然后遍历确认文件解析出的 actual result，用同样的 key 去 Map 里找 expected result，再逐字段比对确认状态、金额、份额、日期和原因码。这样比双层循环更清晰，效率也更高。”

### 最短背诵版

“HashMap 底层可以理解成数组加链表/红黑树。put 时先算 key 的 hash 定位桶，冲突时再用 equals 判断 key 是否相等。正常情况下查找接近 O(1)，冲突严重会变慢。equals 相等的对象 hashCode 必须相等，否则 HashMap 可能找不到 key。项目里 expected/actual 比对可以用 Map，把 expected 按 requestNo 做 key 存起来，再遍历 actual 去查对应 expected 做字段比对。”

### 记忆钩子

`hash 定位桶 -> equals 判相等 -> 冲突链表/红黑树 -> expected Map 对齐 actual`

### Q6.6：手写逻辑：expectedList 和 actualList 按 requestNo 对齐，并判断 confirmStatus 是否一致

题目：

- `expectedList`：预期确认结果列表。
- `actualList`：实际确认结果列表。
- 每个对象都有 `requestNo` 和 `confirmStatus`。
- 判断每个 expected 是否都能在 actual 中找到同 requestNo 的记录，并且 confirmStatus 一致。

### 回答思路

不要双层循环。

用 `HashMap`：

1. 先遍历 `actualList`，用 `requestNo` 做 key，把 actual 放进 Map。
2. 再遍历 `expectedList`。
3. 用 expected 的 `requestNo` 去 Map 里找 actual。
4. 找不到：记录 missing。
5. 找到了但 status 不一致：记录 mismatch。
6. 都一致：通过。

时间复杂度：

- 建 Map：O(n)
- 遍历 expected：O(m)
- 总体：O(n + m)

### Java 伪代码

```java
Map<String, ActualConfirm> actualMap = new HashMap<>();

for (ActualConfirm actual : actualList) {
    actualMap.put(actual.getRequestNo(), actual);
}

List<String> errors = new ArrayList<>();

for (ExpectedConfirm expected : expectedList) {
    ActualConfirm actual = actualMap.get(expected.getRequestNo());

    if (actual == null) {
        errors.add("Missing actual result, requestNo=" + expected.getRequestNo());
        continue;
    }

    if (!Objects.equals(expected.getConfirmStatus(), actual.getConfirmStatus())) {
        errors.add(
            "Status mismatch, requestNo=" + expected.getRequestNo()
            + ", expected=" + expected.getConfirmStatus()
            + ", actual=" + actual.getConfirmStatus()
        );
    }
}

boolean passed = errors.isEmpty();
```

### 更完整一点的版本

如果还要检查 actual 里是否存在 expected 没有的多余记录，可以再加一步：

```java
Set<String> expectedKeys = expectedList.stream()
    .map(ExpectedConfirm::getRequestNo)
    .collect(Collectors.toSet());

for (ActualConfirm actual : actualList) {
    if (!expectedKeys.contains(actual.getRequestNo())) {
        errors.add("Unexpected actual result, requestNo=" + actual.getRequestNo());
    }
}
```

### 面试口述版

“我会先把 actualList 转成一个 Map，key 是 requestNo，value 是 actual 对象。然后遍历 expectedList，用 expected 的 requestNo 去 Map 里查对应 actual。如果查不到，说明 TA 没有生成对应确认结果；如果查到了，就比较 confirmStatus。如果状态不一致，就记录 expected 和 actual 的差异。这样比双层循环效率高，时间复杂度是 O(n + m)。如果还要严谨一点，也会反向检查 actual 里是否有 expected 没有的多余记录。”

### 最短背诵版

“先把 actualList 按 requestNo 放进 HashMap，再遍历 expectedList。找不到 actual 就是 missing；找到了但 confirmStatus 不一致就是 mismatch；最后看 errors 是否为空。复杂度 O(n + m)，比双层循环更适合 expected/actual 对账。”

### 记忆钩子

`actual 建 Map -> expected 查 Map -> missing / mismatch / unexpected`

### Q7.5：讲一次你和业务方或其他团队意见不一致的经历，你怎么处理？

- 素材很好：产品管理系统里业务希望加很多硬校验，技术侧担心上线初期影响生产流程。
- “硬校验 vs 软校验”是一个非常适合讲协作和风险平衡的例子。
- 不要讲太散，不要同时展开自动清算复核等第二个例子。
- 不要说“技术开发能力差”，可以说“上线初期需要控制系统拦截风险”。

### 回答思路

用产品管理系统做主故事：

1. 背景：产品管理系统上线前，业务希望用很多硬校验降低操作风险。
2. 冲突：业务希望强拦截；技术担心规则口径未完全验证时影响生产流程。
3. 动作：承认业务风险诉求，把校验分级，关键风险硬校验，提示类先软校验。
4. 协作：和业务一起确认哪些必须拦截、哪些先提示；UAT 观察误报漏报。
5. 结果：既控制风险，又避免上线初期流程被过度阻断。
6. 复盘：好的协作不是简单说不，而是提供替代方案。

### 更稳版本

我的处理方式不是直接否定业务需求，而是先认可他们想降低操作风险的目标，然后和他们一起把校验分级。对于监管、合同关键字段、必填要素这类明确不能错的内容，我们支持做硬校验；对于一些口径还需要观察、或者更多是提醒性质的内容，我建议先做软校验，比如明显弹窗提示、要求业务二次确认、记录确认日志，但不直接阻断提交。

## 8. Coding 现场沟通

### Q8.1：拿到算法题后，如果没有马上想到最优解，你会怎么和面试官沟通？

### 你的当前回答问题

- 你已经说到输入输出、可行思路、写代码、关键变量和结论，方向正确。
- 需要补充边界条件、样例验证、复杂度分析。
- “给出结论”可以说得更具体：跑一个样例，说明时间复杂度和空间复杂度。
- 如果没想到最优解，先讲 brute force 或可行解，再主动说可以优化。

### 回答思路

现场 coding 沟通顺序：

1. 复述题意，确认输入输出。
2. 确认边界：空数组、重复元素、负数、长度限制、是否有序。
3. 先讲可行思路，不沉默。
4. 如果有优化空间，再讲优化思路。
5. 写代码时说明关键变量和数据结构。
6. 用样例跑一遍。
7. 说时间复杂度和空间复杂度。
8. 如果没写完，也要清楚说明剩余部分思路。

### 更稳版本

“如果我拿到题目后没有马上想到最优解，我会先保持沟通，不会直接沉默写代码。第一步我会复述题意，确认输入和输出，比如输入是否可能为空、是否有重复元素、是否有序、数据范围大概是多少。第二步我会先讲一个可行解，即使不是最优，也先让面试官知道我能完整解决问题。第三步再看这个可行解的瓶颈在哪里，能不能用 HashMap、双指针、栈、二分等方式优化。

写代码时，我会边写边说明关键变量，比如 Map 存什么、指针代表什么、循环不变量是什么。写完后用题目样例或我自己构造的边界样例跑一遍，确认结果。最后我会说明时间复杂度和空间复杂度。如果过程中发现代码有问题，我会及时解释并修正，而不是硬写到底。”

### 最短背诵版

“我会先确认输入输出和边界条件，再讲一个可行思路；如果不是最优，再说明怎么优化。写代码时解释关键变量和数据结构，写完后跑样例，最后说时间和空间复杂度。即使一开始没想到最优解，也会保持和面试官沟通，先保证有正确解，再逐步优化。”

### 记忆钩子

`题意 -> 边界 -> 可行解 -> 优化 -> 代码变量 -> 样例 -> 复杂度`

### Q9.1：你平时怎么使用 Git 进行团队协作？

面试官追问点：

1. 常用 Git 命令有哪些？
2. 一般提交流程是什么？
3. 多人改同一个文件产生冲突怎么处理？
4. 怎么保证提交质量？

### 你的当前回答问题

- 你知道企业内部代码平台、远端仓库、本地修改、add/commit/push/pull 的基本流程。
- 口误需要修正：接手已有项目一般是 `git clone`，不是 `git push`。
- `git add` 是把文件放到 `暂存区`。
- 冲突处理不能只说“选择如何合并”，要讲清查看冲突标记、理解两边逻辑、手动合并、测试、再提交。
- 建议补充分支协作：从主干/开发分支拉 feature branch，提交 MR/PR，code review 后合并。

### 回答思路

常用命令：

- `git clone`
- `git pull`
- `git status`
- `git checkout` / `git switch`
- `git branch`
- `git add`
- `git commit`
- `git push`
- `git log`
- `git diff`
- `git merge`

标准流程：

1. 从远端 clone 项目或 pull 最新代码。
2. 基于开发分支创建 feature branch。
3. 本地开发，频繁 status/diff。
4. add + commit，写清 commit message。
5. push 到远端分支。
6. 提 MR/PR，经过 review 和 CI 后合并。

冲突处理：

1. pull/merge 时发现 conflict。
2. 打开冲突文件，看 `<<<<<<<`, `=======`, `>>>>>>>`。
3. 理解 current 和 incoming 两边逻辑。
4. 手动保留正确代码或整合两边逻辑。
5. 本地编译/测试。
6. `git add` 冲突文件，重新 commit。

### 更稳版本

“我们内部也有类似 GitLab 的代码仓库平台。平时如果接手已有项目，我会先用 `git clone` 把远端仓库拉到本地；如果已经有本地仓库，在开发前会先 `git pull` 同步当前分支最新代码。开发时一般会基于开发分支拉一个 feature branch，在本地修改后用 `git status` 看变更，用 `git diff` 检查具体改动，再 `git add` 放到暂存区，`git commit` 生成本地提交，最后 `git push` 到远端分支，提交 merge request 或 pull request，等 code review 和 CI 通过后再合并。

常用命令包括 `clone`、`pull`、`branch`、`checkout` 或 `switch`、`status`、`diff`、`add`、`commit`、`push`、`log`、`merge`。

如果多人修改同一个文件产生冲突，我不会直接覆盖。会先打开冲突文件，看 Git 标出来的冲突区域，理解当前分支和对方分支各自改了什么，再手动合并成正确逻辑。合并后会本地编译、跑测试或至少跑相关用例，确认没问题后再 `git add` 冲突文件并提交。

为了保证提交质量，我会尽量做到每次 commit 聚焦一个主题，commit message 写清楚；提交前看 diff，避免提交临时文件、调试代码或无关改动；重要功能会补测试或至少跑本地验证；涉及公共模块时会更依赖 code review，避免影响其他人。”

### 最短背诵版

“接手项目先 `git clone`，日常开发前 `git pull`，新需求一般拉 feature branch。修改后用 `git status` 和 `git diff` 检查，再 `git add`、`git commit`、`git push`，提交 MR/PR 等 review 和 CI。冲突时看冲突标记，理解两边逻辑，手动合并，跑测试，再 add 和 commit。提交质量靠小步提交、清晰 message、提交前看 diff、跑测试和 code review。”

### 记忆钩子

`clone/pull -> branch -> status/diff -> add/commit/push -> MR -> conflict 手动合并 -> test`

### Q9.2：Spring Boot 服务部署到 Linux 后访问不了，你会怎么排查？

面试官追问点：

1. 服务进程是否启动？
2. 端口是否监听？
3. 日志里是否有异常？
4. 配置文件、环境变量、启动参数是否正确？
5. 服务器资源、磁盘、网络、防火墙是否正常？
6. 如果前面有 Nginx，如何判断是应用问题还是代理问题？

### 回答思路

排查顺序可以记成：

`进程 -> 端口 -> 日志 -> 配置 -> 资源 -> 网络/代理 -> 依赖`

第一步，看服务有没有起来。

- `ps -ef | grep java`
- `jps -l`
- `systemctl status xxx`

如果进程不存在，说明服务启动失败，要直接看启动日志和部署脚本。

第二步，看端口有没有监听。

- `ss -lntp | grep 8080`
- `netstat -lntp | grep 8080`
- `lsof -i:8080`

如果进程在但端口没监听，可能是应用启动过程中报错、端口配置不对、profile 不对，或者服务还没启动完成。

第三步，本机 curl 验证应用是否可访问。

- `curl http://127.0.0.1:8080/health`
- `curl http://127.0.0.1:8080/接口路径`

如果本机能通、外部不通，优先看网络、防火墙、安全组、Nginx 转发。如果本机也不通，优先看应用本身。

第四步，看日志。

- `tail -f app.log`
- `grep -i "error" app.log`
- `grep -i "exception" app.log`

重点看端口占用、数据库连接失败、Redis 连接失败、配置文件缺失、权限不足、SQL 报错、依赖服务不可用。

第五步，看配置。

- 是否用了正确的 `spring.profiles.active`
- 端口 `server.port` 是否正确
- 数据库、Redis、第三方接口地址是否是当前环境地址
- 启动参数、环境变量、配置中心是否生效

第六步，看资源。

- `df -h` 看磁盘是否满了
- `free -m` 看内存
- `top` 看 CPU 和进程状态
- `du -sh *` 看日志是否打满磁盘

第七步，如果前面有 Nginx 或网关，判断是代理问题还是应用问题。

- 先在服务器本机 curl 应用端口。
- 如果应用端口通，再看 Nginx 配置、upstream、access log、error log。
- 如果 Nginx 报 502，常见原因是后端服务没起来、端口错、upstream 地址错、连接超时。

### 更稳版本

“如果 Spring Boot 服务部署到 Linux 后访问不了，我会按从底层到上层的顺序排查。

首先确认服务进程是否存在，比如用 `ps -ef | grep java`、`jps -l`，如果是 systemd 管理的服务，也会看 `systemctl status`。如果进程都没有，说明服务启动失败，就直接去看启动日志和部署脚本。

第二步看端口是否监听，比如用 `ss -lntp | grep 8080` 或 `lsof -i:8080`。如果进程存在但端口没监听，可能是应用启动过程中报错、端口配置不对、profile 不对，或者依赖连接失败导致启动没有完成。

第三步我会在服务器本机用 `curl 127.0.0.1:端口` 测一下。如果本机能访问但外部访问不了，那更可能是网络、防火墙、Nginx 或网关转发问题；如果本机也访问不了，就继续看应用日志。

日志方面，我会看 `tail -f` 和 `grep error/exception`，重点关注端口占用、数据库连接失败、Redis 连接失败、配置文件缺失、权限不足、SQL 报错、依赖系统不可用等问题。

然后再检查配置和资源，比如当前启用的 `profile` 是否正确，数据库和 Redis 地址是否是当前环境，端口配置是否正确；同时用 `df -h`、`free -m`、`top` 看磁盘、内存、CPU 是否异常。

如果服务前面挂了 Nginx，我会先区分是应用问题还是代理问题：应用本机端口能通，再查 Nginx 的 upstream 配置和 access/error log；如果出现 502，一般会重点查后端服务是否存活、端口是否配置错、是否连接超时。”

### 最短背诵版

“我会按 `进程、端口、日志、配置、资源、网络/代理` 的顺序排查。先用 `ps` 或 `jps` 看 Java 进程是否存在，再用 `ss` 或 `lsof` 看端口是否监听，然后在服务器本机 `curl 127.0.0.1:端口` 判断应用本身是否可访问。接着看日志里的 error 和 exception，排查端口占用、数据库/Redis 连接失败、配置文件错误、权限问题。再看 profile、端口、数据库地址等配置，以及磁盘、内存、CPU。若本机能通但外部不通，就看 Nginx、网关、防火墙和网络转发。”

### 记忆钩子

`ps 看进程 -> ss 看端口 -> curl 本机 -> tail 看日志 -> profile 看配置 -> df/free/top 看资源 -> Nginx/网络看外部访问`

### Q9.3：你怎么理解 CI/CD？Spring Boot 项目的基本 Pipeline 应该有哪些阶段？

面试官追问点：

1. CI/CD 分别是什么？
2. Continuous Delivery 和 Continuous Deployment 有什么区别？
3. 一个后端项目从提交代码到部署，一般有哪些阶段？
4. Jenkins 在你的 TA 自动化测试项目里承担什么角色？
5. 怎么保证上线质量？

### 你的当前回答问题

- 你抓到了 CI/CD 是自动化链路、Pipeline 编排，这是核心方向。
- 术语需要修正：CI 是 `Continuous Integration`，不是单纯的 integration；CD 是 `Continuous Delivery` 或 `Continuous Deployment`。
- 金融机构里通常不会强调“自动上生产”，更稳的说法是：自动完成构建、测试、打包、部署到测试环境，生产发布一般仍有审批、变更窗口和人工确认。
- Jenkins 在 TA 自动化测试里不要说成完整生产发布 CI/CD，更准确是“用 Jenkins Pipeline 做自动化任务编排和回归测试流程固化”。
- 上线质量不只靠 SIT/UAT，还可以补充静态检查、单元测试、构建产物管理、灰度/回滚、上线后冒烟验证。

### 回答思路

CI：

- 持续集成。
- 开发提交代码后，系统自动拉代码、编译、运行单元测试、做基础检查。
- 目标是尽早发现代码集成问题。

CD：

- Continuous Delivery：持续交付，自动构建和部署到测试/预生产环境，生产发布通常保留人工审批。
- Continuous Deployment：持续部署，测试通过后自动部署到生产。
- 金融机构通常更偏持续交付，因为有变更审批、发布窗口、回退预案和合规要求。

Spring Boot Pipeline 常见阶段：

1. 拉取代码：从 Git 仓库 checkout。
2. 编译构建：Maven/Gradle build。
3. 单元测试：运行 JUnit 等测试。
4. 静态检查：代码规范、安全扫描、依赖漏洞扫描。
5. 打包制品：生成 jar/war 或 Docker 镜像。
6. 制品归档：上传到制品库或镜像仓库。
7. 部署测试环境：部署到 SIT/UAT 环境。
8. 自动化测试/冒烟测试：确认关键接口和核心流程可用。
9. 审批发布：金融机构通常需要变更审批。
10. 生产部署：按发布窗口上线。
11. 上线后验证：日志、健康检查、核心功能冒烟。
12. 回滚预案：失败时回退到上一版本。

TA 自动化测试项目中的 Jenkins：

- 不是完整生产发布流水线。
- 主要承担流程编排：清理测试数据、生成申请文件、触发/等待 TA 批处理、解析确认文件、比对 expected/actual、输出结果。
- 价值是把原来人工反复做的回归测试步骤标准化、可重复化。

### 更稳版本

“我理解 CI/CD 是把代码从提交到构建、测试、交付、部署尽量自动化和标准化的一套流程。

CI 是 Continuous Integration，也就是持续集成。开发提交代码后，流水线会自动拉代码、编译、运行单元测试和基础检查，尽早发现代码集成问题。

CD 有两种含义。一种是 Continuous Delivery，持续交付，意思是代码通过自动化构建和测试后，可以自动部署到测试或预生产环境，但生产发布通常还需要人工审批。另一种是 Continuous Deployment，持续部署，测试通过后自动发布到生产。金融机构一般更偏持续交付，因为生产上线涉及变更审批、发布窗口、回退方案和合规要求。

如果是一个 Spring Boot 项目，我理解一条基本 Pipeline 会包括：从 Git 拉代码，Maven 或 Gradle 编译构建，执行单元测试和静态检查，打包成 jar/war 或 Docker 镜像，把制品上传到制品库，然后部署到测试环境，执行自动化测试或冒烟测试。生产发布前一般会经过审批，上线后再做健康检查、日志观察和核心功能验证，同时要准备回滚方案。

我在 TA 自动化测试项目里用 Jenkins 更多是做自动化流程编排，不是完整生产发布。比如通过 Pipeline 串起测试数据清理、申请文件生成、TA 批处理后的确认文件解析、expected/actual 比对和结果输出。它的价值是把原来依赖人工的回归测试流程固定下来，提高可重复性和测试效率。”

### 最短背诵版

“CI 是持续集成，代码提交后自动拉代码、编译、测试和检查，尽早发现集成问题。CD 可以是持续交付或持续部署，金融机构通常更偏持续交付，也就是自动构建和部署到测试/预生产，生产发布还需要审批。Spring Boot 的 Pipeline 一般包括拉代码、Maven 构建、单元测试、静态检查、打包 jar 或镜像、部署测试环境、冒烟测试、审批上线、上线后验证和回滚预案。我在 TA 自动化测试里用 Jenkins 主要做流程编排，把数据清理、申请文件生成、确认文件解析和结果比对串起来。”

### 记忆钩子

`CI = 拉代码 + 编译 + 测试；CD = 打包 + 部署 + 验证；金融机构 = 审批 + 窗口 + 回滚`

### Q9.4：上线后发现接口报错或核心功能异常，你会怎么处理？怎么理解回滚？

面试官追问点：

1. 上线后异常，第一反应是什么？
2. 什么时候选择回滚，什么时候选择 hotfix？
3. 回滚是不是只把上一个 jar 包重新部署？
4. 如果涉及数据库脚本或配置变更怎么办？
5. 如何确认回滚成功？
6. 金融机构里如何同步业务和技术相关方？

### 你的当前回答问题

- 你抓到了“排查”和“回滚”两个关键动作。
- 需要补充：先做影响范围评估，判断是否影响核心交易、客户、批处理、监管报送等关键链路。
- 需要补充：生产问题要先止血，再定位，不要陷入长时间排查导致影响扩大。
- 回滚不只是把上一个版本制品重新部署，还要关注配置、数据库脚本、缓存、定时任务、外部接口开关等是否兼容。
- 金融机构语境下，要同步业务、运维、测试、项目负责人，保留日志和现场，按应急/变更流程处理。

### 回答思路

上线后问题处理顺序：

1. 确认影响范围：哪些接口、哪些用户、哪些业务流程受影响。
2. 判断严重程度：是否影响核心交易、批处理、资金清算、客户操作。
3. 先止血：如果影响核心功能且短时间不能修复，优先回滚。
4. 同步相关方：业务、测试、运维、项目负责人，必要时启动应急流程。
5. 保留现场：日志、报错时间、请求参数、版本号、发布内容。
6. 执行回滚或 hotfix。
7. 回滚后验证：健康检查、核心接口、关键业务流程、日志。
8. 复盘：根因、测试缺口、发布检查项、监控告警改进。

回滚的理解：

- 回滚是把系统恢复到上一个稳定可用状态。
- 最简单是部署上一版本的制品，但完整回滚还要看：
  - 代码制品是否回退。
  - 配置是否回退。
  - 数据库结构或数据脚本是否兼容。
  - 缓存是否需要清理。
  - 定时任务、开关、外部接口是否需要恢复。
- 如果本次发布包含数据库不可逆变更，回滚会更复杂，所以发布前要尽量做兼容设计和回滚预案。

什么时候 hotfix：

- 问题影响小、原因明确、修复风险低，可以 hotfix。
- 问题影响核心链路、原因不明确、修复需要较长时间，优先回滚。

### 更稳版本

“如果上线后发现接口报错或核心功能异常，我会先做两件事：一是快速确认影响范围，二是判断是否需要先止血。

具体来说，我会先看这个问题影响的是单个接口还是核心业务链路，比如是否影响交易、清算、批处理、客户操作或业务日常处理。如果影响核心功能，并且短时间内不能明确修复，我会倾向于优先回滚，先恢复系统稳定性，再做根因排查。

同时我会同步业务、测试、运维和项目负责人，保留现场信息，比如报错时间、请求参数、版本号、日志、发布内容。技术上会看应用日志、Nginx 或网关日志、数据库日志，确认是代码问题、配置问题、数据问题还是依赖系统问题。

我理解回滚不是简单地重新发一个旧包，而是把系统恢复到上一个稳定可用状态。应用层面可以部署上一版本 jar/war 或镜像，但还要确认本次发布是否包含配置变更、数据库脚本、缓存、定时任务或外部接口开关。如果数据库结构有不可逆变更，回滚就会复杂很多，所以发布前应该准备回滚包、回滚步骤和验证清单，尽量做兼容性设计。

回滚完成后，我会做健康检查和核心功能冒烟验证，比如接口能否访问、核心流程能否跑通、日志是否还有异常，再同步业务确认生产恢复。事后还要复盘根因，补充测试用例、发布检查项或监控告警，避免同类问题再次发生。”

### 最短背诵版

“上线后出问题，我会先判断影响范围和严重程度。如果影响核心链路且短时间不能修复，优先回滚止血；同时同步业务、测试、运维和负责人，保留日志和现场。回滚不是简单发旧包，而是恢复到上一个稳定状态，要看代码制品、配置、数据库脚本、缓存、定时任务和外部接口开关是否都兼容。回滚后要做健康检查、核心功能冒烟和日志观察，确认恢复后再复盘根因，补测试和发布检查项。”

### 记忆钩子

`影响评估 -> 先止血 -> 同步相关方 -> 保留现场 -> 回滚/修复 -> 冒烟验证 -> 复盘改进`

### Q9.5：你怎么理解 Nginx 和 Tomcat？请求经过 Nginx 到 Spring Boot 时，它们各自承担什么角色？

面试官追问点：

1. Nginx 是什么，常见作用有哪些？
2. Tomcat 是什么？
3. Spring Boot 和 Tomcat 的关系是什么？
4. 一个请求经过 Nginx 到后端服务的链路是什么？
5. 502/504 一般是什么问题？

### 你的当前回答问题

- 你说对了 Nginx 的核心作用：反向代理。
- 你说对了 Tomcat 的核心作用：Java Web 应用运行容器。
- 建议补充 Nginx 的几个常见职责：反向代理、负载均衡、SSL 终止、静态资源、限流、统一入口。
- Spring Boot 通常内嵌 Tomcat，打成 jar 后可以直接运行；传统 Java Web 项目也可以打 war 部署到外部 Tomcat。
- 回答时最好讲请求链路，不要只讲定义。

### 回答思路

Nginx：

- 主要作为统一入口和反向代理。
- 客户端请求先到 Nginx。
- Nginx 根据域名、路径、upstream 配置转发到后端服务。
- 还可以做负载均衡、SSL 证书卸载、静态资源服务、限流、日志记录。

Tomcat：

- Java Web 容器 / Servlet 容器。
- 负责接收 HTTP 请求，创建/调度线程处理请求，把请求交给 Java Web 应用。
- Spring MVC 的 Controller 最终是在 Tomcat 容器里被调用。
- Spring Boot 默认可以内嵌 Tomcat，所以 jar 启动后本身就带一个 Web 容器。

请求链路：

1. 用户浏览器或前端发起 HTTP 请求。
2. 请求先到 Nginx。
3. Nginx 根据路由规则转发到某个后端 Spring Boot 实例。
4. Spring Boot 内嵌 Tomcat 接收请求。
5. Tomcat 把请求交给 Spring MVC 的 DispatcherServlet。
6. DispatcherServlet 分发到 Controller。
7. Controller 调 Service、DAO/Mapper，访问数据库或缓存。
8. 结果逐层返回，再由 Nginx 返回给客户端。

常见错误：

- 502 Bad Gateway：Nginx 找不到可用后端，后端服务没起来、端口错、连接被拒绝。
- 504 Gateway Timeout：Nginx 能连到后端，但后端处理太慢或超时。

### 更稳版本

“我理解 Nginx 更多是在系统入口层做反向代理和流量转发。客户端请求通常先到 Nginx，Nginx 根据域名、路径或者 upstream 配置，把请求转发到后端某个服务实例。它还可以做负载均衡、SSL 证书卸载、静态资源服务、限流和访问日志。

Tomcat 是 Java Web 容器，也可以理解为 Servlet 容器。它负责承载 Java Web 应用，接收 HTTP 请求，管理线程，把请求交给应用里的 Servlet 或 Spring MVC 处理。Spring Boot 默认通常是内嵌 Tomcat，所以我们把 Spring Boot 项目打成 jar 运行时，其实里面已经带了一个 Web 容器；传统方式也可以打成 war 包部署到外部 Tomcat。

如果讲一个完整请求链路，就是用户请求先到 Nginx，Nginx 按规则转发到后端 Spring Boot 实例，Spring Boot 内嵌 Tomcat 接收请求，再交给 Spring MVC 的 DispatcherServlet，最后路由到 Controller，Controller 调 Service，Service 再调 Mapper/DAO 访问数据库或缓存，结果再原路返回。

排障时，如果 Nginx 返回 502，我会优先看后端服务是否存活、端口是否正确、upstream 配置是否正确；如果是 504，更像是后端处理超时，需要看接口耗时、数据库慢查询或下游依赖超时。”

### 最短背诵版

“Nginx 是入口层，主要做反向代理、负载均衡、SSL、静态资源和限流；Tomcat 是 Java Web/Servlet 容器，负责承载 Java Web 应用并处理 HTTP 请求。Spring Boot 通常内嵌 Tomcat，所以 jar 启动后就能对外提供 Web 服务。请求链路是：浏览器到 Nginx，Nginx 转发到 Spring Boot，Tomcat 接收请求，DispatcherServlet 分发到 Controller，再调用 Service、DAO、数据库，最后返回。502 多是后端不可用或端口配置错，504 多是后端处理超时。”

### 记忆钩子

`Nginx = 入口/转发；Tomcat = Java Web 容器；链路 = Nginx -> Tomcat -> DispatcherServlet -> Controller -> Service -> DAO`

### Q9.6：你怎么理解 Redis？后端系统里 Redis 通常用来解决什么问题？缓存和数据库不一致怎么办？

面试官追问点：

2. Redis 为什么快？

4. 缓存和数据库不一致为什么会发生？
5. 更新数据时先更新数据库还是先更新缓存？
6. 如果删除缓存失败怎么办？

### 你的当前回答问题

- 你说对了 Redis 的核心使用方式：内存缓存、高频访问数据、减少数据库 IO、提升响应速度。
- 不建议说“正常不会出现缓存不一致”。缓存和数据库是两份数据，双写不是天然原子操作，不一致是常见风险。
- 需要补充：数据库通常是主数据源，缓存是加速层；金融系统里不能把缓存当最终可信数据源。
- 需要掌握一个最常用策略：Cache Aside，也就是读缓存、未命中读库并写缓存；写数据时先更新数据库，提交成功后删除缓存。
- 删除缓存失败时，要有重试、消息队列、定时修复、TTL 兜底。

### 回答思路

Redis 是什么：

- 基于内存的高性能 Key-Value 数据库。
- 常用作缓存，也可以用于分布式锁、计数器、限流、会话、排行榜等。
- 支持 String、Hash、List、Set、Sorted Set 等数据结构。

Redis 为什么快：

- 数据主要在内存中。
- 网络模型高效。
- 常见操作数据结构简单。
- 相比每次访问关系型数据库，少了磁盘 IO 和复杂 SQL 查询成本。

常见场景：

- 缓存高频读取数据，例如产品基础信息、参数配置、字典项。
- 分布式锁，防止并发任务重复执行。
- 限流计数，例如接口访问频率控制。
- 临时状态或会话信息。

缓存一致性：

- 数据库一般是 source of truth。
- Redis 是加速层，允许在部分场景下短暂最终一致。
- 如果业务要求强一致，核心交易判断尽量直接查数据库，或者减少缓存使用。

典型 Cache Aside 模式：

读流程：

1. 先查 Redis。
2. 命中则直接返回。
3. 未命中则查数据库。
4. 把数据库结果写入 Redis，并设置 TTL。

写流程：

1. 先更新数据库。
2. 数据库事务提交成功后删除缓存。
3. 下一次读请求缓存未命中，会从数据库读取最新值并重建缓存。

为什么不是直接更新缓存：

- 并发写入时，更新缓存容易出现旧值覆盖新值。
- 删除缓存更简单，下一次读库重建更可靠。

删除缓存失败怎么办：

- 设置缓存 TTL 兜底。
- 删除失败进入重试。
- 通过消息队列异步重试删除。
- 定时任务做缓存修复。
- 对一致性要求高的核心链路，避免依赖缓存结果做最终判断。

### 更稳版本

“我理解 Redis 是一个基于内存的高性能 Key-Value 存储，在后端系统里最常见的用途是做缓存。比如一些高频读取、变化不那么频繁的数据，如果每次都查数据库，会增加数据库压力和响应耗时；放到 Redis 后，可以减少数据库 IO，提高接口响应速度。

除了缓存，Redis 也可以用于分布式锁、计数器、限流、会话状态、临时数据等场景。它支持 String、Hash、List、Set、Sorted Set 等数据结构。

缓存和数据库不一致是需要重点关注的，因为数据库和缓存本质上是两份数据，更新不是天然原子操作。我理解数据库应该是最终可信的数据源，Redis 是加速层。在金融系统里，核心交易和资金类判断不能完全依赖缓存里的旧数据。

常见的缓存使用方式是 Cache Aside。读的时候先查 Redis，命中就返回；没命中就查数据库，然后把结果写入 Redis 并设置过期时间。写的时候一般先更新数据库，数据库事务提交成功后删除缓存，而不是直接更新缓存。这样下一次读请求发现缓存不存在，就会从数据库读最新数据并重建缓存。

如果删除缓存失败，就可能出现短暂不一致，所以需要有 TTL 兜底，也可以通过重试、消息队列异步删除、定时任务修复等方式降低风险。如果某个业务场景对一致性要求非常高，我会倾向于直接查数据库，或者在设计上避免使用缓存作为最终判断依据。”

### 最短背诵版

“Redis 是基于内存的高性能 Key-Value 存储，常用来做缓存，减少数据库 IO，提高响应速度，也可用于分布式锁、计数器、限流和临时状态。缓存和数据库是两份数据，所以不一致是常见风险。一般数据库是最终可信源，缓存只是加速层。常见 Cache Aside 模式是：读先查缓存，未命中查库并写缓存；写先更新数据库，提交成功后删除缓存，让下一次读从数据库重建。删除缓存失败可以靠 TTL、重试、消息队列或定时修复兜底；强一致场景尽量直接查数据库。”

### 记忆钩子

`Redis = 内存缓存；DB = 真相；读缓存 miss 查库；写库后删缓存；TTL/重试兜底`

### Q9.6.1：数据库已更新但缓存没更新，新查询命中旧缓存怎么办？能避免吗？

### 核心理解

这个场景不能简单说“不会发生”。数据库和 Redis 是两个系统，普通情况下没有天然的分布式事务，所以数据库更新成功、缓存删除失败时，确实可能有新查询命中旧缓存。

### 稳定回答

“这种情况确实可能发生，因为数据库和缓存不是一个原子事务。常见做法不是更新缓存，而是更新数据库成功后删除缓存，让下一次查询缓存 miss 后从数据库重建。

如果数据库更新成功但删除缓存失败，就可能短时间读到旧缓存，所以需要加 TTL、删除失败重试、消息队列异步补偿，或者定时任务修复缓存。

并发更复杂的场景可以用延迟双删，比如先删缓存、更新数据库、过一小段时间再删一次缓存，降低并发读把旧数据重新写入缓存的概率。

但如果是金融交易、额度校验、资金清算这类强一致要求高的场景，就不应该只依赖缓存，而应该以数据库或核心系统为准。”

### 最短背诵版

“能降低概率，但普通缓存架构很难绝对避免，因为 DB 和 Redis 不是原子事务。一般写库成功后删除缓存，不直接更新缓存；删除失败要有 TTL、重试、MQ 补偿或定时修复；高并发下可以延迟双删；强一致金融场景不只信缓存，以数据库或核心系统为准。”

### 记忆钩子

`不是原子事务 -> 写库后删缓存 -> 删除失败重试/TTL -> 并发用延迟双删 -> 强一致不用缓存做最终判断`

### Q9.7：你怎么理解数据库事务？ACID 是什么？PLM 中“保存补充要素”和“生成提醒配置”要么都成功要么都失败，怎么设计？

面试官追问点：

1. ACID 分别是什么？
2. 事务解决什么问题？
3. Spring Boot 项目里如何使用事务？
4. 事务应该放在哪一层？
5. 如果有数据库操作和外部接口调用，是否都能放在一个事务里？

### 你的当前回答问题

- 你说出了最核心的原子性：要么全部成功，要么全部失败。
- ACID 需要补全：Atomicity、Consistency、Isolation、Durability。
- 需要把答案从“概念”落到项目设计：在 Service 层用 `@Transactional` 包住多个数据库写操作。
- 金融场景里可以用“一笔交易涉及主表和关联表，要么都落库，要么都回滚”来类比，很好。
- 注意：数据库事务主要管同一个数据库里的操作，外部接口、消息、邮件等不能简单依赖数据库事务一起回滚。

### 回答思路

ACID：

- A Atomicity 原子性：事务内操作要么全部成功，要么全部失败。
- C Consistency 一致性：事务执行前后，数据要满足业务规则和约束。
- I Isolation 隔离性：并发事务之间相互隔离，避免互相干扰。
- D Durability 持久性：事务提交后，数据持久保存，即使系统故障也不应丢失。

事务解决的问题：

- 多张表、多条记录需要作为一个整体提交。
- 中间任一步失败，需要回滚前面已经执行的数据库操作。
- 避免出现主表写成功、明细表没写成功的半成品数据。

PLM 设计：

场景：

- 用户保存产品补充要素。
- 系统根据补充要素生成提醒配置。
- 两者强绑定：如果补充要素保存了，但提醒配置没生成，后续跑批会不完整；如果提醒配置生成了，但补充要素没保存，也会形成脏数据。

实现方式：

1. Controller 接收请求并做基础参数校验。
2. Service 层定义一个完整业务方法，例如 `saveProductLifecycleConfig()`。
3. 在 Service 方法上加 `@Transactional`。
4. 方法内部先保存产品补充要素表。
5. 再生成提醒规则/提醒配置表。
6. 如有任一步抛出异常，事务回滚，两张表都不落库。
7. 提交成功后，再做非事务动作，例如发通知、刷新缓存、记录操作日志。

为什么放在 Service 层：

- Controller 只负责接收请求和返回结果。
- Mapper/DAO 只负责单表数据访问。
- Service 才承载一个完整业务动作，跨多张表的事务边界应该放在 Service。

注意点：

- 默认情况下，Spring 对 RuntimeException 会回滚。
- 如果捕获异常后不再抛出，事务可能不会回滚。
- 如果涉及邮件、OA、外部接口调用，不建议直接放在数据库事务里一起依赖回滚；可以在事务提交后再异步处理，失败再补偿。

### 更稳版本

“我理解数据库事务是把一组数据库操作作为一个完整的业务单元来处理，要么全部成功提交，要么出现异常时全部回滚，避免出现半成功的数据状态。

ACID 里面，A 是 Atomicity 原子性，指事务内操作要么全成要么全失败；C 是 Consistency 一致性，指事务前后数据要满足业务规则和约束；I 是 Isolation 隔离性，指并发事务之间要有隔离，避免互相影响；D 是 Durability 持久性，指事务提交后数据要持久保存。

如果放到产品生命周期管理系统里，比如保存产品补充要素和生成提醒配置这两个动作，我会把它们设计成一个 Service 层的完整业务方法，在这个方法上加 `@Transactional`。Controller 只负责接收请求和做基础参数校验，Service 方法里面先保存产品补充要素表，再根据产品类型、观察日期、提醒阈值等信息生成提醒配置表。如果中间任何一步失败，比如提醒配置生成失败或者数据库插入异常，就抛出异常，让整个事务回滚，避免出现补充要素保存了但提醒配置缺失，或者提醒配置生成了但主数据不存在的情况。

事务边界我会放在 Service 层，因为这个层代表一个完整业务动作；Mapper/DAO 只负责单表操作，不适合作为完整业务事务边界。

另外，如果保存成功后还要发邮件、调用 OA 或刷新缓存，我不会简单把这些外部动作当成数据库事务的一部分。更稳的方式是数据库事务先提交成功，然后再做异步通知或缓存删除；如果外部动作失败，通过重试或补偿机制处理。”

### 最短背诵版

“事务就是一组数据库操作要么全部提交，要么失败回滚。ACID 是原子性、一致性、隔离性、持久性。在 PLM 里，保存产品补充要素和生成提醒配置应该放在同一个 Service 方法里，用 `@Transactional` 控制事务。方法里先写补充要素表，再写提醒配置表；任何一步失败就抛异常回滚，避免半成品数据。事务边界放 Service 层，因为它代表完整业务动作。邮件、OA、缓存这类外部动作不要强行放进数据库事务，提交后再异步处理或补偿。”

### 记忆钩子

`ACID = 原子/一致/隔离/持久；Service 加事务；多表同成败；外部调用靠补偿`

### Q9.8：你怎么理解 RESTful API？PLM 的“查询提醒列表”和“新增产品补充要素”接口怎么设计？

面试官追问点：

1. RESTful API 的核心思想是什么？
2. URL 应该用名词还是动词？
3. GET 和 POST 有什么区别？
4. 查询参数和请求体怎么放？
5. 返回结构如何设计？
6. 如何考虑幂等性和错误码？

### 你的当前回答问题

- 你抓到了 RESTful 是接口设计风格，围绕资源做增删改查。
- `GET` 用于查询、`POST` 用于新增，这个方向对。
- URL 建议使用名词资源，不要在路径里写 `add` 这类动词，动作由 HTTP Method 表达。
- 资源命名建议统一复数，例如 `/api/products/{productCode}/lifecycle-elements`、`/api/reminders`。
- 参数设计可以分为 path variable、query params、request body。

### 回答思路

RESTful 核心：

- 把系统里的对象抽象成资源。
- URL 表示资源。
- HTTP Method 表示动作。
- 常见方法：
  - `GET` 查询。
  - `POST` 新增。
  - `PUT` 全量更新。
  - `PATCH` 部分更新。
  - `DELETE` 删除。

查询产品提醒列表：

- 接口：`GET /api/reminders`
- Query 参数：
  - `productCode`：产品代码，可选。
  - `productName`：产品名称，可选。
  - `reminderType`：提醒类型，如目标盈、定期分红。
  - `status`：提醒状态，如 pending、sent、ignored。
  - `startDate` / `endDate`：提醒日期范围。
  - `page` / `pageSize`：分页。
- 返回：
  - `code`
  - `message`
  - `data.list`
  - `data.total`

新增产品补充要素：

- 接口：`POST /api/products/{productCode}/lifecycle-elements`
- Path 参数：
  - `productCode`
- Body：
  - `productType`：产品类型。
  - `targetYield`：目标收益率，目标盈产品使用。
  - `observationStartDate`：观察开始日。
  - `observationDays`：连续观察天数。
  - `dividendFrequency`：分红频率，定期分红产品使用。
  - `thresholdConfig`：阈值配置。
  - `effectiveDate`：生效日期。
  - `remark`：备注。
- 后端处理：
  - Controller 做基础参数校验。
  - Service 保存补充要素。
  - 根据要素生成提醒配置。
  - 使用事务保证同成败。

也可以设计为：

- `POST /api/lifecycle-elements`
- Body 中带 `productCode`。

如果以产品为主资源，更推荐：

- `POST /api/products/{productCode}/lifecycle-elements`

错误码：

- `400` 参数错误。

- `409` 重复配置或状态冲突。

“我理解 RESTful API 是一种面向资源的接口设计风格。URL 用来表示资源，HTTP Method 用来表示动作，所以一般路径里尽量用名词，不用 `add`、`query` 这种动词。比如 `GET` 表示查询，`POST` 表示新增，`PUT/PATCH` 表示更新，`DELETE` 表示删除。

如果参数不合法返回 400，产品不存在返回 404，如果已经存在同类配置或状态不允许新增，可以返回 409，系统异常返回 500。这样接口语义会比较清晰，也方便前后端和测试沟通。”

### 最短背诵版

“RESTful 是面向资源的接口风格，URL 表示资源，HTTP Method 表示动作，所以路径尽量用名词，不用 add/query 这类动词。查询提醒列表可以用 `GET /api/reminders`，查询条件放 query params，如 productCode、reminderType、status、date range、page/pageSize。新增产品补充要素可以用 `POST /api/products/{productCode}/lifecycle-elements`，productCode 放路径，目标收益率、观察日、分红频率、阈值等放 body。后端 Controller 校验参数，Service 保存要素并生成提醒配置，用事务保证同成败。”

### Q9.9：Java 异常处理怎么理解？什么时候 catch，什么时候 throw？事务回滚和统一异常返回怎么做？

面试官追问点：

1. Java 异常有哪些类型？
2. 什么时候应该 `catch`，什么时候应该继续 `throw`？
3. 为什么不能随便吞异常？
4. Spring 事务什么情况下会回滚？
5. 后端接口如何统一返回错误信息？

### 核心理解

异常处理的目标不是“把报错吃掉”，而是：

- 能恢复的异常，在当前层处理。
- 不能恢复的异常，向上抛给统一异常处理。
- 关键异常要记录日志，方便排查。
- 数据库写操作失败时，要让事务回滚，避免半成品数据。
- 对前端返回统一格式，不把堆栈和内部细节直接暴露出去。

### 回答思路

Java 异常分类：

- `Checked Exception`：编译期异常，代码必须显式处理，例如 `IOException`。
- `Unchecked Exception`：运行时异常，继承 `RuntimeException`，例如 `NullPointerException`、`IllegalArgumentException`。
- `Error`：JVM 级别严重问题，例如 `OutOfMemoryError`，一般业务代码不处理。

什么时候 catch：

- 当前层能够明确处理或补救。
- 需要转换异常类型。
- 需要记录日志后再抛出。
- 调用外部接口失败，需要做重试、降级或返回明确错误。

什么时候 throw：

- 当前层无法处理。
- 需要让上层统一处理。
- 需要触发事务回滚。
- 业务规则不满足，例如产品不存在、状态不允许、重复配置。

不应该做的事：

- `catch (Exception e) {}` 空 catch。
- catch 后只打印日志但不抛出，导致调用方以为成功。
- 把系统内部异常堆栈直接返回前端。

事务回滚：

- Spring 默认对 `RuntimeException` 和 `Error` 回滚。
- 对 checked exception 默认不回滚，除非配置 `rollbackFor = Exception.class`。
- 如果在事务方法里 catch 异常但不重新抛出，事务可能不会回滚。

统一异常返回：

- Controller 不到处写 try-catch。
- 用 `@ControllerAdvice` / `@RestControllerAdvice` 做全局异常处理。
- 对业务异常返回业务错误码和可读 message。
- 对系统异常记录详细日志，对前端返回通用错误信息。

### 更稳版本

“我理解异常处理不是简单地把异常 catch 掉，而是要看当前层有没有能力处理。

Java 里异常大致分为 checked exception、runtime exception 和 error。业务开发里更常见的是 runtime exception，比如参数非法、空指针、状态不合法等。Checked exception 比如 IO 异常，需要显式处理；Error 通常是 JVM 层面的严重问题，业务代码一般不处理。

什么时候 catch，我理解是当前层能够明确处理的时候，比如可以重试、降级、转换成业务异常，或者需要记录上下文日志后再抛出。如果当前层处理不了，就应该继续 throw 给上层或统一异常处理器。最忌讳的是空 catch，或者 catch 后只打印日志但不抛出，这会让调用方误以为流程成功。

在 Spring 项目里，如果一个 Service 方法涉及多张表写入，我会用 `@Transactional` 管事务。Spring 默认对 RuntimeException 和 Error 回滚；如果 catch 了异常但不重新抛出，事务可能不会回滚。所以数据库保存失败时，我一般会记录必要日志，然后抛出业务异常或运行时异常，让事务回滚，避免出现部分表写成功、部分表失败的半成品数据。

接口返回方面，我倾向于用统一异常处理，比如 `@RestControllerAdvice`。Controller 不到处写 try-catch。业务异常返回明确错误码和 message，比如参数不合法、产品不存在、重复配置；系统异常则记录详细日志，但对前端只返回通用错误信息，避免暴露堆栈和内部实现。”

### 最短背诵版

“异常处理不是把异常吃掉，而是当前层能处理就 catch，不能处理就 throw。能处理包括重试、降级、转换业务异常、记录上下文；不能处理就交给上层统一异常处理。Spring 事务默认对 RuntimeException 回滚，如果在事务里 catch 后不抛出，可能导致事务不回滚。数据库保存失败时应记录日志并抛出异常，让事务回滚。接口层可以用 `@RestControllerAdvice` 统一处理异常，业务异常返回错误码和 message，系统异常记录日志但不把堆栈暴露给前端。”

### 记忆钩子

`能处理才 catch；不能处理就 throw；事务异常要抛出；ControllerAdvice 统一返回`

### Q9.11：你怎么理解 JVM？Java 运行时内存分哪些区域？线上 OOM 怎么初步排查？

面试官追问点：

1. JVM 是什么？
2. Java 为什么能跨平台？
3. JVM 内存大概分哪些区域？
4. GC 主要回收什么？
5. OOM 可能是什么原因？
6. 线上 OOM 如何初步排查？

### 核心理解

JVM 可以理解为 Java 程序的运行环境。Java 代码先编译成字节码，JVM 负责加载字节码、解释或编译执行，并管理内存和垃圾回收。

短时间只需要记住：

- JVM 是 Java 程序运行的虚拟机。
- Java 跨平台靠 JVM：同一份字节码，不同操作系统上由不同 JVM 执行。
- 内存重点记三块：堆、栈、方法区。
- GC 主要回收堆里的对象。
- OOM 排查看日志、内存、dump、最近变更。

### 回答思路

JVM 是什么：

- Java Virtual Machine，Java 虚拟机。
- 负责运行 Java 字节码。
- 提供内存管理、类加载、垃圾回收、即时编译等能力。

为什么 Java 跨平台：

- Java 源码编译成 `.class` 字节码。
- 字节码不是直接给操作系统执行，而是由 JVM 执行。
- 不同平台有不同 JVM，所以 Java 代码可以“一次编译，到处运行”。

内存区域：

- 堆 Heap：存放对象实例，是 GC 重点管理区域。比如 `new Product()` 生成的对象主要在堆里。
- 栈 Stack：每个线程私有，存放方法调用、局部变量、方法参数。方法调用结束，对应栈帧出栈。
- 方法区 / 元空间 Metaspace：存放类信息、常量、静态变量等元数据。
- 程序计数器：记录当前线程执行到哪条字节码。
- 本地方法栈：支持 native 方法调用。

面试短时间重点讲：

- 堆：放对象。
- 栈：放方法调用和局部变量。
- 方法区/元空间：放类信息。

GC 是什么：

- Garbage Collection，垃圾回收。
- 主要回收堆中不再被引用的对象。
- 目标是释放内存，避免对象不断堆积。

OOM 常见原因：

- 一次性加载太多数据，比如大文件、大查询结果集。
- 集合对象持续增长没有释放。
- 缓存没有过期策略，越积越多。
- 线程过多或线程栈占用过大。
- 内存参数配置过小。
- 代码存在内存泄漏，例如静态集合长期持有对象引用。

OOM 初步排查：

1. 看错误日志，确认是 `Java heap space`、`Metaspace` 还是其他类型。
2. 看服务最近是否有版本发布、配置变更、大批量任务、数据量突增。
3. 看机器资源：`top`、`free -m`、容器内存限制。
4. 如果有 heap dump，用 MAT / jvisualvm 等工具分析大对象和引用链。
5. 看是否有大集合、大查询、缓存、批处理任务、文件读取没有分页/分批。
6. 临时止血可以重启或扩内存，但根因还是要定位对象为什么持续增长。

### 更稳版本

“我理解 JVM 是 Java 程序的运行环境。Java 源码会先编译成字节码，JVM 负责加载和执行这些字节码，同时提供内存管理、类加载、垃圾回收等能力。Java 能跨平台，也是因为同一份字节码可以在不同操作系统对应的 JVM 上执行。

JVM 内存区域如果简化理解，重点是堆、栈和方法区。堆主要存放对象实例，比如业务代码里 new 出来的对象，GC 主要也是回收堆里不再被引用的对象。栈是线程私有的，存放方法调用、局部变量和参数，方法执行完对应栈帧就出栈。方法区或者元空间主要存放类信息、常量、静态变量等元数据。

如果线上服务出现 OOM，我会先看日志里具体是哪类 OOM，比如 `Java heap space` 还是 `Metaspace`。然后结合最近是否有版本发布、批量任务、数据量突增或配置变更来判断方向。技术上会看机器内存、容器限制、GC 日志，如果有 heap dump，会用工具分析哪些对象占用最大、是谁在持有引用。

常见原因包括一次性查出太多数据、批处理没有分页、集合或缓存持续增长、缓存没有过期策略、线程数过多，或者内存参数设置过小。处理上可以先重启或扩内存止血，但根因还是要定位对象为什么没有释放，比如改成分页查询、限制缓存大小、补 TTL、优化批处理逻辑。”

### 最短背诵版

“JVM 是 Java 程序运行的虚拟机，负责执行字节码、管理内存和做垃圾回收。Java 跨平台靠的是同一份字节码在不同平台 JVM 上执行。内存重点记三块：堆放对象，栈放方法调用和局部变量，方法区/元空间放类信息。GC 主要回收堆中不再被引用的对象。线上 OOM 先看日志确认类型，再看最近发布、数据量、批处理、缓存和内存配置；有 dump 就分析大对象和引用链。常见原因是一次性查太多数据、缓存不释放、集合持续增长或内存配置过小。”

### 记忆钩子

`JVM 跑字节码；堆放对象；栈跑方法；元空间放类；GC 清堆；OOM 看日志/dump/最近变更`

### Q9.12：你怎么理解线程池？为什么不建议每来一个任务就 new Thread？PLM 批量计算提醒时怎么用？

面试官追问点：

1. 线程池是什么？
2. 为什么不建议无限制 `new Thread`？
3. 线程池有哪些核心参数？
4. PLM 批量跑批计算提醒时，为什么可以考虑线程池？
5. 金融系统里使用线程池要注意什么风险？

### 核心理解

线程池可以理解成“提前准备好一组工作线程”。任务来了以后，不是每次都临时创建新线程，而是交给线程池里的线程处理。

它解决的问题是：

- 减少频繁创建和销毁线程的成本。
- 控制并发数量，避免任务太多把 CPU、内存、数据库、下游接口打爆。
- 统一管理任务排队、超时、异常和拒绝策略。

### 回答思路

为什么不建议每个任务都 `new Thread`：

- 线程创建和销毁有成本。
- 线程太多会抢 CPU，导致上下文切换开销很大。
- 每个线程都占用内存，过多可能导致 OOM。
- 没有统一管理，任务失败、异常、排队、限流都不好控制。

线程池核心参数：

- `corePoolSize`：核心线程数，平时保留的工作线程数量。
- `maximumPoolSize`：最大线程数，高峰时最多能扩到多少。
- `workQueue`：任务队列，线程忙不过来时任务先排队。
- `keepAliveTime`：非核心线程空闲多久后回收。
- `RejectedExecutionHandler`：队列满、线程也满时的拒绝策略。

PLM 场景：

- 每天早上跑批扫描产品。
- 对每个产品计算是否触发目标盈、定期分红等提醒。
- 如果产品数量很多，完全串行可能慢。
- 可以按产品或产品批次拆成任务，提交给固定大小线程池并行计算。
- 计算结果写入提醒结果表。
- 通知发送可以再走单独线程池或异步任务，避免影响主跑批。

注意事项：

- 线程数不能太大，要根据机器资源、数据库连接池、下游系统承受能力设置。
- 数据库写入要考虑幂等，避免重复生成提醒。
- 每个任务要捕获并记录异常，不能因为单个产品失败导致整个批次无感失败。
- 跑批要有任务状态表或执行日志，方便失败重跑。
- 金融系统里宁愿可控地慢一点，也不要无限并发导致核心系统被打爆。

### 更稳版本

“我理解线程池就是提前创建和维护一组工作线程。任务来了以后，不是每次都 `new Thread`，而是把任务提交给线程池，由线程池复用已有线程处理。

不建议每来一个任务就创建新线程，是因为线程创建和销毁有成本，线程太多会带来 CPU 上下文切换和内存开销，严重时可能导致服务变慢甚至 OOM。而线程池可以控制并发数量，通过任务队列、最大线程数和拒绝策略来保护系统。

线程池常见核心参数包括核心线程数、最大线程数、任务队列、空闲线程存活时间和拒绝策略。简单理解就是：平时有多少工人，忙的时候最多扩到多少，忙不过来的任务排在哪里，排满了怎么办。

如果放到 PLM 的批量提醒计算场景，每天早上系统要扫描一批产品，判断目标盈、定期分红等提醒是否触发。如果产品数量较多，完全串行可能耗时较长，可以按产品或产品批次拆分任务，交给固定大小的线程池并行计算，计算结果写入提醒结果表。通知发送也可以单独异步处理，避免拖慢主跑批。

但金融系统里用线程池要很谨慎。线程数不能无限大，要考虑数据库连接池、下游系统和服务器资源承受能力。任务写库要做幂等，避免重复生成提醒；每个任务要记录异常和执行结果，方便排查和重跑。我会倾向于用有界队列和固定大小线程池，保证并发可控。”

### 最短背诵版

“线程池就是提前准备一组工作线程，任务来了复用这些线程处理，而不是每次 new Thread。这样可以减少线程创建销毁成本，并控制并发，避免线程太多导致 CPU 切换、内存上涨、数据库或下游系统被打爆。核心参数可以记成核心线程数、最大线程数、任务队列、空闲回收时间和拒绝策略。PLM 跑批时，可以把产品按批次拆成任务，用固定大小线程池并行计算提醒，再写入结果表；通知发送可以异步处理。但线程数要受控，任务要幂等，要记录异常，不能为了快影响系统稳定。”

### 记忆钩子

`线程池 = 固定工人；new Thread = 临时招人；核心参数 = 工人数/队列/拒绝；PLM = 分批计算 + 幂等 + 可控并发`

### Q9.13：你怎么理解日志？生产问题排查时日志应该怎么打、怎么看？

面试官追问点：

1. 日志有什么作用？
2. 常见日志级别有哪些？
3. 生产问题排查时如何通过日志定位？
4. 日志里应该记录什么，不应该记录什么？
5. 如何把一次请求的链路串起来？

### 核心理解

日志是生产排障最重要的依据之一。它不是为了“多打印”，而是为了在问题发生后能还原现场：什么时间、哪个请求、哪个用户/产品/批次、走到哪一步、失败原因是什么。

### 回答思路

日志作用：

- 记录系统运行状态。
- 记录关键业务流程节点。
- 记录异常和上下文。
- 帮助定位生产问题。
- 支撑审计和问题复盘。

常见级别：

- `DEBUG`：开发调试信息，生产一般不开或少开。
- `INFO`：关键流程节点，例如任务开始/结束、批次号、处理数量。
- `WARN`：非预期但系统还能继续，例如某个产品参数缺失、外部接口短暂失败。
- `ERROR`：明确失败或异常，需要关注，例如数据库写入失败、接口调用失败、批处理失败。

应该记录：

- 请求 ID / traceId。
- 关键业务主键：产品代码、交易申请编号、批次号、任务 ID。
- 入参摘要，不记录敏感明文。
- 关键流程节点。
- 异常堆栈。
- 耗时。
- 外部接口返回码和失败原因。

不应该记录：

- 客户身份证号、银行卡号、手机号等敏感信息明文。
- 密码、token、密钥。
- 过大的完整报文，避免日志爆炸。
- 只打印“失败了”但不带上下文。

生产排查链路：

1. 确认问题发生时间、接口、产品/批次/用户。
2. 用 traceId、requestId、产品代码、批次号搜索日志。
3. 看请求是否进入系统。
4. 看走到哪个业务节点失败。
5. 看异常堆栈和外部依赖返回。
6. 结合数据库、Nginx、应用日志、批处理日志交叉验证。

PLM 场景：

- 跑批开始：记录批次号、日期、产品数量。
- 每个产品失败：记录产品代码、提醒类型、失败原因。
- 跑批结束：记录成功数量、失败数量、耗时。
- 通知发送失败：记录提醒 ID、渠道、错误码。

### 更稳版本

“我理解日志是生产排障和问题复盘的重要依据。好的日志不是越多越好，而是要能在问题发生后还原现场，比如什么时间、哪个请求、哪个产品或批次、走到哪一步、为什么失败。

常见日志级别有 DEBUG、INFO、WARN、ERROR。DEBUG 主要用于开发调试，生产一般少开；INFO 用来记录关键流程节点，比如批处理开始和结束、处理数量；WARN 表示有异常情况但系统还能继续，比如某个非核心参数缺失；ERROR 表示明确失败，需要关注，比如数据库写入失败、外部接口调用失败。

生产排查时，我会先确认问题发生时间、接口、产品代码或批次号，然后用 traceId、requestId、产品代码、交易申请编号等关键字段去搜索日志，看请求是否进入系统，执行到了哪个业务节点，是否有异常堆栈或外部接口错误码。必要时会结合 Nginx 日志、应用日志、数据库记录和批处理日志一起看。

日志里应该记录关键业务主键、请求 ID、批次号、流程节点、耗时、异常堆栈和外部接口返回码。但不能记录敏感信息明文，比如身份证号、银行卡号、密码、token，也不能无节制打印大报文。

放到 PLM 跑批里，我会在跑批开始记录批次号、日期和待处理产品数量；每个产品计算失败时记录产品代码、提醒类型和失败原因；跑批结束记录成功数、失败数和耗时。这样如果某个产品没有提醒，就可以顺着日志定位是数据没进来、规则没命中，还是通知发送失败。”

### 最短背诵版

“日志的作用是还原现场，帮助生产排障。常见级别有 DEBUG、INFO、WARN、ERROR。INFO 记录关键流程节点，WARN 记录可继续的异常，ERROR 记录明确失败。排查时先确认时间、接口、产品代码或批次号，再用 traceId/requestId 搜日志，看请求是否进入系统、走到哪一步、哪里抛异常。日志应记录业务主键、批次号、耗时、异常堆栈和外部接口返回码，但不能记录身份证、银行卡、密码、token 等敏感信息明文。PLM 跑批要记录开始、结束、产品失败原因、成功失败数量，方便追踪。”

### 记忆钩子

`日志 = 还原现场；INFO 流程；WARN 风险；ERROR 失败；traceId 串链路；敏感信息不明文`

### Q9.14：你怎么理解 IaC、Ansible、Terraform？

面试官追问点：

1. IaC 是什么？
2. Ansible 和 Terraform 大概分别解决什么问题？
3. 它们和 CI/CD 有什么关系？
4. 如果你之前没深度用过，怎么回答才稳？

### 核心理解

IaC 是 Infrastructure as Code，基础设施即代码。它的思想是把服务器、网络、数据库、中间件、配置、部署环境等基础设施，用代码或配置文件来描述和管理，而不是完全依赖人工在页面上点点点。

短时间只需要记住：

- IaC：用代码管理环境和基础设施。
- Terraform：偏“创建和管理基础设施”，比如云主机、网络、数据库等资源。
- Ansible：偏“配置和部署自动化”，比如安装软件、修改配置、部署服务。
- 价值：标准化、可重复、可审计、减少人工误操作。

### 稳定回答

“我理解 IaC 是 Infrastructure as Code，也就是基础设施即代码。它的核心思想是把服务器、网络、中间件、部署环境和配置，用代码或配置文件的方式管理起来，而不是完全依赖人工操作。这样做的好处是环境可以标准化、可重复创建、可审计，也能减少人工配置差异和误操作。

Terraform 和 Ansible 都和 IaC 有关，但侧重点不一样。Terraform 更偏向基础设施资源的创建和管理，比如创建云服务器、网络、安全组、数据库实例等；Ansible 更偏向配置管理和自动化部署，比如在服务器上安装软件、修改配置文件、启动服务、批量执行部署脚本。

它们也可以和 CI/CD 结合。比如 Pipeline 里前面用 Maven 构建应用，用 Docker 生成镜像，后面可以通过 Ansible 去部署服务，或者通过 Terraform 管理底层环境资源。

我之前没有在生产环境里深度使用过 Terraform 或 Ansible，但我理解它们解决的是环境标准化和自动化交付问题。如果入职后团队有成熟规范，我会先学习现有 playbook、module 和发布流程，从小的配置修改、部署脚本和环境变更开始参与。”

### 最短背诵版

“IaC 是基础设施即代码，用代码或配置文件管理服务器、网络、中间件和环境配置，价值是标准化、可重复、可审计、减少人工误操作。Terraform 偏创建和管理基础设施资源，比如云主机、网络、数据库；Ansible 偏配置管理和自动化部署，比如安装软件、改配置、启动服务。它们可以接到 CI/CD 后半段，帮助完成环境准备和应用部署。我没有深度生产使用过，但理解它们解决的问题，入职后可以按团队规范学习现有脚本和流程。”

### 记忆钩子

`IaC = 环境也写成代码；Terraform 管资源；Ansible 管配置/部署；价值 = 标准化 + 可重复 + 可审计`

## 10. Apprentice 个人项目追问

### Q10.1：你简历里的 Apprentice 是什么项目？为什么做它？

### 回答定位

Apprentice 是加分项，不是主线项目。面试时不要讲成成熟上线系统，也不要讲成高自由度自动替代员工的 Agent。最稳定位是：

**一个从员工行为学习出发，逐步形成查询、建议、参数预填和受控执行能力的企业内部员工 AI 助手原型。**

### 稳定回答

“Apprentice 是我个人做的一个面向企业传统系统的内部员工 AI 助手原型。它的出发点不是简单做问答机器人，而是想解决员工在传统系统里大量重复操作、路径依赖经验和上下文切换成本高的问题。

我的思路是先通过零侵入的方式记录员工在系统里的操作行为，再基于这些行为形成员工画像和操作模式理解，然后在交互层提供查询、建议、参数预填这类低风险辅助能力。更后面的阶段，才是在权限、白名单接口、人工确认和审计机制都比较清楚的前提下，探索受控执行。

从技术结构上，它不是纯聊天产品，也不是一上来全自动 Agent。我更愿意把它定义成 Workflow 为主干、局部使用 Agent 能力做意图理解和工具调用的原型。整体流程是确定的，比如行为采集、画像学习、交互辅助、受控执行；但在具体交互时，需要结合用户意图、上下文和可用工具来生成建议或调用接口。

我做这个项目主要是因为这两年 AI 工具发展很快，我希望把它和我熟悉的企业内部系统、金融机构后台流程结合起来，验证 AI 能不能在可控边界内进入真实工作流，而不是只停留在问答层面。”

### 最短背诵版

“Apprentice 是我个人做的企业内部员工 AI 助手原型。它不是简单问答机器人，而是想解决传统系统里重复操作多、路径依赖经验强、上下文切换成本高的问题。整体路径是先零侵入记录操作，再做员工画像和行为学习，然后提供查询、建议、参数预填，最后在白名单接口、人工确认和审计机制下探索受控执行。我把它理解成 Workflow 为主干、局部用 Agent 做意图理解和工具调用的原型。”

### 记忆钩子

`不是问答机器人；先学行为；再做辅助；最后受控执行；Workflow 主干 + Agent 局部能力`

### Q10.2：这个项目有哪些技术点？你具体做了什么？

### 回答思路

可以讲四层：

1. 行为采集层：通过 AOP / 审计日志思路记录员工操作，不侵入业务主流程。
2. 配置层：用配置描述不同系统的接口、字段、权限、操作定义，尽量做到换系统不改核心 Agent 代码。
3. Agent 服务层：用 Python / FastAPI 实现对话入口、配置加载、画像读取、提示词构建、工具选择和受控执行。
4. 前端交互层：提供聊天式交互，让员工通过自然语言查询、获得建议或触发受控操作。

### 稳定回答

“技术上我主要想验证的是配置驱动的企业系统 AI 操作层能不能跑通，而不是为某一个具体系统写死一套助手。

第一层是行为采集。我参考企业 Java 系统常见的 AOP 和审计日志思路，设想在不侵入业务主流程的情况下记录员工操作，包括操作人、接口、参数摘要、时间和结果，用来支持后续画像学习。

第二层是配置驱动。不同业务系统的接口、字段、操作和权限都不一样，所以我不希望 Agent 代码里写死具体业务词汇，而是通过配置文件描述系统有哪些操作、每个操作需要哪些字段、怎么校验、怎么调用后端接口。

第三层是 Agent 服务。我用 Python / FastAPI 做了一个后端原型，负责加载配置、读取用户画像、构建受控提示词、做参数校验，并根据用户意图调用白名单工具或接口。

第四层是前端交互，提供一个聊天式入口，让用户通过自然语言表达需求，系统返回建议、参数预填或执行结果。

我在这个项目里主要做的是产品定义、架构设计和原型实现。它目前还是个人 PoC，不是生产上线项目，所以我不会把它讲成成熟系统；但它验证了一个方向：企业内部 AI 工具如果想真正有用，不能只懂知识，还要能理解工作流、权限边界和受控执行。”

### 最短背诵版

“技术上我按四层理解：AOP/审计日志做行为采集；配置层描述接口、字段、权限和操作；Python/FastAPI Agent 服务做配置加载、画像读取、提示词构建、参数校验和受控工具调用；前端提供聊天式交互。我主要做产品定义、架构设计和原型实现。它是个人 PoC，不是生产系统，重点验证配置驱动的企业 AI 操作层是否可行。”

### 记忆钩子

`采集层 -> 配置层 -> Agent 服务层 -> 前端交互层；个人 PoC，不夸大上线`

### Q10.3：如果放到金融机构里，这个项目最大的风险是什么？怎么控制？

### 稳定回答

“如果放到金融机构里，最大的风险不是模型能不能回答，而是权限、审计、误操作和合规边界。

所以我不会一上来让它自由执行高风险操作。第一阶段更适合做查询、建议、参数预填和草稿生成。真正涉及写操作或业务状态变更时，需要白名单接口、权限校验、人工确认、操作留痕和可回滚机制。

另外，模型输出不能直接作为最终业务判断。它可以辅助员工发现路径、整理参数、生成建议，但关键交易、额度、清算、风控类动作仍然要以核心系统规则和人工确认为准。

我对这个项目的理解是，AI 在金融后台系统里应该先做可控提效，而不是追求高自治替代。先建立信任，再逐步扩大能力边界。”

### 最短背诵版

“放到金融机构里，最大风险是权限、审计、误操作和合规。第一阶段只做查询、建议、参数预填、草稿生成；写操作必须走白名单接口、权限校验、人工确认和操作留痕。模型不能替代核心系统规则，交易、额度、清算、风控类判断仍以核心系统和人工确认为准。AI 应该先做可控提效，不是一上来高自治替代。”

### 记忆钩子

`金融 AI 风险 = 权限/审计/误操作/合规；先辅助，后受控执行`
