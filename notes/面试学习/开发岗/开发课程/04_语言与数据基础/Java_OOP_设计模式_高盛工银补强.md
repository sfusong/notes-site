# Java OOP 与设计模式高盛工银补强

这份只解决一个问题：

`如果 onsite 问 Java 面向对象、设计原则、设计模式，你能用自己的项目讲出来。`

---

## 一、先背一版总答案

如果面试官问：

`你怎么理解面向对象？`

可以答：

`我理解面向对象的核心，是把现实业务里的对象、规则和行为抽象成类和对象，通过封装控制内部状态，通过继承和接口复用共性，通过多态让不同实现可以被统一调用。放到我做过的项目里，比如 TA 自动化测试里的不同文件解析规则、静态参数稽核里的不同稽核项、产品生命周期系统里的不同提醒规则，其实都可以抽象成一组接口和不同实现，这样后续扩展新规则时，不需要大改主流程。`

这段很重要，因为它把 OOP 和你的项目挂上了。

---

## 二、面向对象四大特性

## 1. 封装

`封装是把数据和操作数据的方法放在一起，并通过访问控制隐藏内部细节。`

面试稳妥说法：

`封装的价值是降低外部调用方对内部实现的依赖。比如提醒规则怎么判断、稽核项怎么查询，可以封装在 Service 或 Rule 类里，Controller 只关心调用结果。`

## 2. 继承

`继承是子类复用父类的属性和行为。`

但要注意：

`继承不是越多越好，真实项目里更常用接口和组合来扩展能力。`

面试稳妥说法：

`继承适合表达明确的 is-a 关系，但如果只是复用能力，我会更倾向组合或接口，避免继承层级过深。`

## 3. 多态

`多态是同一个接口或父类引用，在运行时可以指向不同实现。`

项目挂钩：

- 不同稽核项统一实现 `AuditRule`
- 不同提醒类型统一实现 `ReminderRule`
- 不同文件格式统一实现 `FileParser`

面试稳妥说法：

`多态让主流程可以面向接口编程，不用关心具体是哪一种规则或解析器。后续增加新规则时，只要新增实现类并注册进去，主流程不需要大改。`

## 4. 抽象

`抽象是提取共同特征，屏蔽不重要细节。`

项目挂钩：

`TA 静态稽核虽然每个稽核项检查字段不同，但都可以抽象成：读取配置、执行检查、生成结果、输出明细。`

## 三、接口和抽象类

## 1. 接口

`接口定义能力和契约，不关心具体实现。`

适合：

- 多种实现
- 面向能力编程
- 规则扩展

## 2. 抽象类

`抽象类可以保留部分通用实现，同时把变化点留给子类。`

适合：

- 多个类有明显公共流程
- 只有部分步骤不同

## 3. 面试稳妥答法

`接口更像定义能力，抽象类更像沉淀公共骨架。如果是静态参数稽核这种规则很多、后续可能扩展的场景，我会优先用接口定义统一规则契约；如果每个规则都有相同的执行流程，只是个别步骤不同，也可以用抽象类承接公共模板。`

---

## 四、面向接口编程

最短定义：

`代码依赖接口，而不是直接依赖具体实现。`

为什么重要：

- 降低耦合
- 方便替换实现
- 方便单元测试
- 方便扩展新规则

项目答法：

`比如 TA 自动化测试里，申请文件、确认文件、数据库结果都要解析和比对。可以把解析能力抽象成 Parser 接口，把比对能力抽象成 Comparator 接口，主流程只依赖接口，这样后续不同代销商、不同文件格式可以各自扩展实现。`

## 五、SOLID 设计原则够用版

## 1. 单一职责

`一个类尽量只负责一类清晰职责。`

项目例子：

- Controller 负责接请求
- Service 负责业务流程
- DAO 负责数据库访问
- Rule 负责具体规则判断

## 2. 开闭原则

`对扩展开放，对修改关闭。`

项目例子：

`新增一个稽核项时，最好新增一个规则类，而不是反复修改一个巨大的 if-else。`

## 3. 里氏替换

`子类应该可以替换父类或接口，不破坏原有逻辑。`

面试一般不用展开太深。

## 4. 接口隔离

`接口不要设计得过大，调用方不应该被迫依赖用不到的方法。`

## 5. 依赖倒置

`高层模块依赖抽象，不直接依赖具体实现。`

项目例子：

`批处理校验主流程依赖 AuditRule 接口，而不是依赖每一个具体稽核规则类。`

## 六、你最该会的 5 个设计模式

## 1. 策略模式

最适合你的项目。

定义：

`把一组可替换的算法或规则封装成不同策略，运行时选择其中一种执行。`

项目挂钩：

- TA 静态参数稽核：不同稽核项是不同策略
- 产品生命周期提醒：目标盈、分红、到期是不同策略
- TA 自动化测试：不同文件格式解析是不同策略

面试答法：

`静态稽核项很多，如果全部写在一个方法里会很难维护。我会更倾向把每个稽核项抽象成一个规则策略，主流程负责调度，具体规则负责自己的查询和判断。`

## 2. 模板方法模式

定义：

`父类定义固定流程，子类实现其中变化的步骤。`

项目挂钩：

稽核流程可以抽象成：

1. 加载数据
2. 执行检查
3. 生成结果
4. 保存或展示

不同规则只实现第二步。

面试答法：

`如果多个稽核规则执行流程基本一致，只是检查逻辑不同，就可以用模板方法把公共流程收敛起来。`

## 3. 工厂模式

定义：

`把对象创建逻辑集中管理，调用方不用直接 new 具体实现。`

项目挂钩：

`根据稽核项编码、文件类型、提醒类型，创建对应的处理器。`

面试答法：

`如果规则类型比较多，我会考虑用工厂或注册表来根据类型获取对应实现，避免主流程里到处写 new 和 if-else。`

## 4. 责任链模式

定义：

`多个处理节点按顺序执行，每个节点处理自己负责的部分。`

项目挂钩：

批前检查可以按顺序执行：

1. 基础参数检查
2. 产品状态检查
3. 开放期检查
4. 账户配置检查
5. 费率规则检查

面试答法：

`如果检查项之间有明确顺序，或者需要一条链路逐步校验，可以用责任链。每个节点只负责自己的校验逻辑，整体流程更清晰。`

## 5. 适配器模式

定义：

`把不兼容的接口转换成系统内部统一接口。`

项目挂钩：

- 数据中台字段和系统内部字段不一致
- 不同代销商文件格式不一致
- 邮件/OA 接口入参格式不一致

面试答法：

`系统对接时外部接口格式不一定和内部模型一致，我会用适配层做字段转换和格式统一，避免外部变化直接影响核心业务逻辑。`

## 七、三个项目怎么挂 OOP

## 产品生命周期管理系统

你可以说：

`这个系统里不同产品事件的提醒规则不同，比如目标盈、分红、到期提醒。我理解可以把提醒规则抽象成统一接口，主流程负责加载产品和日历数据，具体规则负责判断是否触发提醒。这样后续增加新的产品事件，不需要改动整个提醒链路。`

## TA 静态参数稽核

你可以说：

`静态参数稽核天然适合规则化设计。不同稽核项可以抽象成规则对象，每个规则负责自己的查询条件、判断逻辑和结果输出。主流程只负责调度规则、汇总结果和展示预警。`

## TA 自动化测试

你可以说：

`自动化测试里有文件生成、批处理触发、确认文件解析、数据库比对等步骤。不同代销商、不同文件格式可以通过 Parser 或 Comparator 的不同实现来扩展，主流程保持稳定。`

## 八、容易被问的题和稳妥答案

## 1. 面向对象四大特性是什么

`封装、继承、多态、抽象。封装控制内部细节，继承复用共性，多态支持统一接口不同实现，抽象提取共同模型。`

## 2. 接口和抽象类有什么区别

`接口更强调能力契约，抽象类可以沉淀公共状态和公共流程。如果是多种规则、多种处理器扩展，我优先用接口；如果流程一致、部分步骤变化，可以用抽象类。`

## 3. 你项目里哪里能用设计模式

`TA 静态稽核可以用策略模式，每个稽核项是一个策略；如果执行流程一致，可以叠加模板方法；如果根据规则编码选择规则实现，可以用工厂或注册表。`

## 4. 为什么不能全写 if-else

`少量 if-else 没问题，但规则越来越多后，会导致主流程变得很长、难测试、难维护。规则对象化后，每个规则可以独立开发和测试，也更符合开闭原则。`

## 5. 什么是低耦合高内聚

`高内聚是一个模块内部职责集中，低耦合是模块之间依赖尽量少。比如 Controller、Service、DAO、Rule 分层清楚，就是为了让每一层关注自己的事情。`

---

## 九、最后背这 8 句话

1. 面向对象的核心是封装、继承、多态和抽象。
2. 封装让外部不用关心内部实现。
3. 多态让主流程可以面向接口编程。
4. 接口定义能力，抽象类沉淀公共流程。
5. 策略模式最适合规则很多、后续要扩展的场景。
6. 模板方法适合流程固定、部分步骤变化的场景。
7. 工厂模式适合根据类型创建不同处理器。
8. 你的 TA 稽核、TA 自动化测试、产品提醒规则，都可以用 OOP 的方式讲出技术含量。
