# Spring MVC / 请求流程 / 参数校验 / 返回结果怎么讲

这份材料的目标，是让你在开发岗面试里，能把：

- `Spring MVC 是什么`
- `一个 HTTP 请求进入后端后大概经历什么`
- `参数校验为什么重要`
- `返回结果为什么要统一`

讲得稳、讲得像做过接口开发的人。

---

## 一、Spring MVC 是什么

最简单理解：

`Spring MVC` 是 Spring 生态里处理 Web 请求的一套模型。

它主要解决的是：

- 浏览器或前端请求怎么进入后端
- 请求参数怎么接
- 业务逻辑怎么调用
- 结果怎么返回给前端

所以你可以把它理解成：

`后端处理 HTTP 请求的一套标准方式。`

---

## 二、Spring MVC 和 Spring Boot 是什么关系

最稳的理解是：

- `Spring Boot` 更像快速搭建后端应用的框架外壳
- `Spring MVC` 是里面处理 Web 请求的重要组成部分

所以你面试时可以这么说：

“Spring Boot 是快速组织后端应用的框架，而 Spring MVC 更聚焦 Web 请求处理，也就是接口请求如何进入 Controller、如何接参数、如何返回结果。”

---

## 三、一个请求进入后端后大概经历什么

这是非常值得讲顺的一条链路。

最典型的流程可以理解成：

1. 前端发起 HTTP 请求
2. 请求进入 Controller
3. Controller 接收参数并做基础校验
4. Controller 调用 Service
5. Service 执行业务逻辑
6. Service 通过 Repository / DAO 访问数据库
7. 结果返回到 Controller
8. Controller 再把结果返回给前端

你后面面试时，如果能把这条链路讲顺，会很像做过后端的人。

---

## 四、Controller 在做什么

最简单理解：

`Controller` 是接口入口层。

它主要做：

- 接收请求
- 获取参数
- 调用业务层
- 返回结果

它不应该做太多：

- 复杂业务逻辑
- 大量数据库处理
- 很重的规则判断

因为这些更适合放在 Service。

---

## 五、Service 在做什么

`Service` 更像业务逻辑层。

它主要做：

- 规则判断
- 流程编排
- 业务状态处理
- 调用多个 Repository 或外部接口

所以你可以把 Service 理解成：

`真正承接业务规则的地方。`

这点和你的产品生命周期管理系统特别好挂。

---

## 六、参数校验为什么重要

这是中后台接口开发里很基础但很值钱的一层。

因为后端不能默认前端传来的参数永远正确。

常见问题有：

- 必填字段缺失
- 类型不对
- 长度不对
- 格式不对
- 枚举值非法
- 时间范围不合理

所以参数校验本质上是在做：

`把错误尽量挡在接口入口。`

---

## 七、参数校验通常怎么理解

你现在不用讲很深的注解体系，先有这个认知就够：

- 基础必填和格式校验可以放在接口入参阶段
- 更复杂的业务规则校验通常在 Service 层做

也就是说：

- `参数是否合法`
- `业务上是否允许`

这两件事不完全是一回事。

这个区分很重要。

---

## 八、如果面试官问：为什么参数校验不能只靠前端

最稳答法：

“因为前端校验更多是用户体验层面的，而后端接口才是真正的业务边界。如果只依赖前端校验，接口仍然可能收到非法参数，所以后端必须做自己的参数校验和规则判断。”

这题很常见。

---

## 九、返回结果为什么常常要统一

最简单理解：

如果每个接口都各自返回一套乱七八糟的结构，前端会很难处理，调用方也很难统一判断成功失败。

所以很多企业项目会统一返回结构，比如：

- 是否成功
- 状态码
- 提示信息
- 业务数据

你不需要死背某种 JSON 模板，但要理解它的意义：

`让接口行为更一致，前后端更容易协作。`

---

## 十、如果面试官问：为什么要统一返回结构

你可以答：

“统一返回结构的价值主要在于让前后端协作更稳定，也让调用方更容易统一处理成功、失败和异常情况。这样接口风格一致，后续维护和排查问题也更方便。”

---

## 十一、异常怎么和接口返回连起来

你现在可以建立这个最基础的理解：

- 参数不合法，要返回明确错误
- 业务规则不满足，也要返回可理解结果
- 系统异常不能直接把底层报错原样暴露给前端

所以一个成熟接口设计通常会区分：

1. 参数问题
2. 业务问题
3. 系统问题

这个意识很重要。

---

## 十二、这课怎么和你的项目绑定

### 1. 产品生命周期管理系统

特别适合讲：

- 前端页面提交产品信息和查询条件
- Controller 接收请求
- Service 做分红、到期、提醒等规则判断
- Repository 做数据读写
- 返回给前端页面展示

### 2. 广发证券授信变动查询

特别适合讲：

- 查询条件接收
- 分页和结果返回
- 导出接口

### 3. 说明书生成工具

虽然不是 Spring MVC 主项目，但可以帮助你说明：

- 不是所有系统都一定要很重
- 但如果是标准业务系统，统一接口和请求流程就很重要

---

## 十三、如果面试官问：你怎么理解一个后端接口

你可以答：

“我理解一个后端接口不只是一个 URL，而是系统对外暴露业务能力的边界。它至少要解决几件事：请求怎么接、参数是否有效、业务规则怎么处理、数据库怎么访问，以及结果怎么稳定返回给前端或其他系统。”

这句很强。

---

## 十四、如果面试官问：你在接口开发里最关注什么

最稳答法：

“我会比较关注四件事：第一是入参和返回结构是否清晰；第二是参数校验和业务规则边界是否明确；第三是接口和业务逻辑是否分层清楚；第四是异常场景下前端或调用方能不能拿到可理解、可处理的结果。”

---

## 十五、面试可直接复述的一版总答案

“Spring MVC 可以理解为 Spring 生态里处理 Web 请求的一套标准方式。一个典型请求进入后端后，通常会先进入 Controller，由 Controller 接参数并做基础校验，再调用 Service 处理业务逻辑，Service 通过 Repository 访问数据库，最后把结果返回给前端。  
我理解接口开发的重点不只是把请求打通，而是要把参数校验、业务逻辑分层和返回结果组织清楚。前端校验更多解决交互体验问题，而后端接口才是真正的业务边界，所以后端必须自己做参数和规则判断。很多企业项目也会统一接口返回结构，这样前后端协作会更稳定，调用方处理成功失败和异常也更一致。”  

---

## 十六、你现在最该记住的 10 句话

1. Spring MVC 是 Spring 生态里处理 Web 请求的一套方式
2. Spring Boot 是应用框架，Spring MVC 更偏请求处理
3. 一个典型请求链路是 Controller -> Service -> Repository
4. Controller 更偏接口入口，Service 更偏业务逻辑
5. 参数校验是在接口入口尽早拦住问题
6. 前端校验不能替代后端校验
7. 参数是否合法和业务上是否允许不是一回事
8. 统一返回结构有利于前后端协作和问题排查
9. 接口开发的重点不只是打通，而是边界清晰、分层清晰、异常可控
10. 这节最好和产品生命周期管理系统、授信查询项目一起讲
