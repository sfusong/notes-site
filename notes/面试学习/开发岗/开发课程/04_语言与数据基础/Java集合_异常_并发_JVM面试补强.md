# Java 集合、异常、并发、JVM 面试补强

这份只补一件事：

`当面试官从 Java 面向对象继续往下追，你能顺着讲，不会突然断掉。`

它不是源码大全。你现在最需要的不是把 `HashMap` 每一行源码背下来，而是能把常见问题讲准，并且接回你的项目。

---

## 一、先记住这条主线

Java 基础题通常不是孤立问的。

面试官问 `集合`，其实在看你怎么组织数据。

面试官问 `异常`，其实在看你怎么处理失败。

面试官问 `并发`，其实在看你是否知道共享数据会出问题。

面试官问 `JVM`，其实在看你是否知道 Java 程序不是凭空运行的。

所以你的总回答可以是：

`我理解 Java 基础不只是语法，而是后端系统可靠性的底层语言能力。集合决定数据怎么组织，异常决定失败怎么暴露和恢复，并发决定多线程下数据是否安全，JVM 决定 Java 程序如何运行和管理内存。`

---

## 二、集合：先讲清 List、Set、Map

### 1. List

`List 是有序、可重复的集合。`

常见实现：

- `ArrayList`：底层更像动态数组，查询快，按下标访问方便
- `LinkedList`：底层更像链表，插入删除在某些场景下更灵活，但随机访问不如 ArrayList

面试稳妥答法：

`如果我只是存一批查询结果、页面列表、待处理记录，一般优先想到 ArrayList，因为它简单、查询方便，也符合大多数业务开发场景。`

### 2. Set

`Set 主要用于去重。`

常见实现：

- `HashSet`：基于哈希结构，常用于快速判断是否存在

项目挂钩：

`比如 TA 自动化测试里，如果要检查确认文件里是否有重复申请编号，或者批前稽核里要检查某类配置是否重复，就可以用 Set 的去重语义来表达。`

### 3. Map

`Map 存 key-value 键值对。`

常见实现：

- `HashMap`：最常用的键值映射结构

项目挂钩：

`比如把产品代码映射到产品信息，把申请编号映射到确认结果，把规则编码映射到具体规则处理器，都很适合用 Map 表达。`

---

## 三、HashMap：不用背很深，但要讲出风险

最短定义：

`HashMap 是基于哈希结构的 key-value 容器，适合按 key 快速查找 value。`

你至少要知道 4 件事：

- key 通过 `hashCode` 定位到桶
- 如果 key 相同或哈希冲突，需要继续比较 `equals`
- 所以重写 `equals` 时通常也要重写 `hashCode`
- `HashMap` 不是线程安全的

面试稳妥答法：

`HashMap 的核心价值是按 key 快速查找。它依赖 hashCode 和 equals 判断 key 的位置和相等性，所以如果自定义对象作为 key，重写 equals 时通常也要重写 hashCode。否则可能出现两个对象逻辑上相等，但在 Map 里找不到或重复存的问题。`

如果追问线程安全：

`普通 HashMap 不适合多个线程同时读写。如果确实有并发访问，需要考虑 ConcurrentHashMap，或者在业务层避免共享可变 Map。`

## 四、异常：不要只说 try-catch

异常的核心不是“报错”，而是：

`系统失败时，能不能被识别、被记录、被恢复或被明确返回。`

### 1. Checked Exception 和 Runtime Exception

- `Checked Exception`：编译期要求处理，通常代表外部条件导致的可预期失败
- `Runtime Exception`：运行时异常，常见于程序逻辑问题，比如空指针、数组越界、非法参数

面试稳妥答法：

`我理解异常处理要区分业务可预期失败和程序错误。比如参数不合法、数据不存在，可以转成明确的业务错误；如果是空指针、数组越界这类运行时问题，就说明代码健壮性或校验不足，需要修复。`

### 2. 项目里怎么讲异常

产品生命周期系统：

`如果外部数据中台返回异常，后端不能直接让页面白屏，而要记录日志、返回明确错误信息，并让业务知道是数据未到、接口失败还是规则判断失败。`

TA 自动化测试：

`如果确认文件解析失败，要能定位是文件格式问题、字段缺失、编码问题，还是 expected/actual 比对逻辑问题。`

TA 批前稽核：

`如果某条稽核规则失败，不能让整个批次信息消失，而是要记录具体规则、具体产品、具体异常原因。`

---

## 五、日志：异常的另一半

异常如果没有日志，就像案发现场没有监控。

你不用把日志框架讲很深，但要知道日志的价值：

- 记录关键入参
- 记录异常堆栈
- 记录批次号、产品代码、申请编号这类定位字段
- 支持排查和追溯

面试稳妥答法：

`我理解金融系统里日志特别重要，因为很多问题不是当场就能复现。日志里至少要保留批次号、业务主键、关键状态和异常原因，这样排查时才能从业务现象追到系统链路。`

---

## 六、并发：先守住线程安全

并发不用一上来讲高并发架构。

你先讲清：

`多个线程同时访问同一份可变数据，就可能产生线程安全问题。`

### 1. synchronized

`synchronized 可以让同一时间只有一个线程进入某段临界区。`

优点：

- 简单
- 语义清楚

代价：

- 可能降低并发性能
- 锁范围设计不好会影响吞吐

### 2. volatile

`volatile 主要保证变量对多个线程的可见性。`

注意边界：

`volatile 不能替代所有锁，它不能保证复合操作的原子性。`

比如 `count++` 看起来一行，实际包含读取、加一、写回，不是天然原子。

### 3. ConcurrentHashMap

`ConcurrentHashMap 是线程安全的 Map，适合并发读写场景。`

面试不要讲源码细节，先讲用途：

`如果多线程环境下要共享一个 key-value 容器，普通 HashMap 风险比较大，可以考虑 ConcurrentHashMap。`

## 七、JVM：够用版只抓三件事

JVM 不是玄学，先理解成：

`JVM 是 Java 程序运行的虚拟机，负责执行字节码、管理内存、做垃圾回收。`

### 1. 字节码和跨平台

Java 源码先编译成 `.class` 字节码，再由不同平台上的 JVM 执行。

所以 Java 才能说：

`一次编写，到处运行。`

### 2. 内存大概分区

你先知道这些关键词：

- `堆`：主要放对象
- `栈`：主要放方法调用、局部变量
- `方法区 / 元空间`：放类信息等

面试稳妥答法：

`我不把自己包装成 JVM 调优专家，但我理解 Java 对象主要在堆上分配，方法调用会用到栈，JVM 会通过垃圾回收管理不再使用的对象。`

### 3. GC 是什么

`GC 是垃圾回收，用来回收不再被引用的对象，避免内存一直增长。`

面试稳妥答法：

`GC 的价值是自动管理内存，但它不是完全没有代价。如果对象创建过多、生命周期混乱，仍然可能带来性能问题。所以业务代码里也要避免无意义地创建大量临时对象。`

---

## 八、最容易被追问的 8 句话

1. `List 有序可重复，Set 去重，Map 存 key-value。`
2. `ArrayList 适合大多数业务列表场景，HashMap 适合按 key 快速查找。`
3. `自定义对象做 Map 的 key 时，equals 和 hashCode 要保持一致。`
4. `HashMap 不是线程安全的，并发读写要谨慎。`
5. `异常处理不是简单 try-catch，而是要明确失败类型、记录日志、返回可理解结果。`
6. `线程安全的核心是共享可变数据。`
7. `volatile 保证可见性，但不能保证所有复合操作的原子性。`
8. `JVM 负责执行字节码、管理内存和垃圾回收。`

---

## 九、怎么和你的项目连起来

产品生命周期管理系统：

- `Map` 可以表达产品代码到产品信息、提醒类型到规则处理器的映射
- 异常处理用于外部数据中台、邮件/OA 接口失败
- OOP 和策略模式用于不同提醒规则扩展

TA 自动化测试：

- `List` 存文件记录
- `Map` 按申请编号聚合 expected / actual
- `Set` 检查重复申请或重复确认
- 异常和日志用于定位文件解析失败

TA 批前稽核：

- `Map` 可以把规则编码映射到规则实现
- `List` 存稽核明细
- 日志和批次号用于追溯
- 并发问题如果被问到，可以说批次类任务更强调状态控制和幂等，不会随意共享可变数据

## 十、最后的记忆法

把这块记成一句话：

`集合管数据，异常管失败，并发管共享，JVM 管运行。`

这句话顺了，Java 基础就不会散。
