# API、事务、数据一致性、可观测性面试补强

这份是给高盛工银开发岗准备的系统延展笔记。

它解决的问题是：

`当面试官不满足于问 Java 语法，而开始问你系统怎么可靠运行、数据怎么不乱、问题怎么定位时，你能稳稳接住。`

---

## 一、先记住系统面试的主线

一个企业后端系统，面试官最关心的通常不是“你会不会写接口”这么简单。

她可能真正想知道：

- 接口怎么设计，别人好不好用
- 数据怎么写入，失败了会不会乱
- 批处理怎么重跑，会不会重复
- 系统出问题，能不能定位
- 和外部系统对接，边界是否清楚

所以你的总回答可以是：

`我理解后端系统不是把接口写通就结束，而是要考虑接口契约、参数校验、事务边界、幂等、日志追踪和异常恢复。尤其金融系统里，数据正确性和可追溯性往往比单纯功能跑通更重要。`

---

## 二、API 设计：接口不是随便开一个 URL

REST API 最简单理解：

`后端通过 HTTP 把业务能力暴露给前端或其他系统。`

但面试里要多讲两层。

### 1. 一个好接口至少要清楚这些

- 这个接口做什么
- 请求参数是什么
- 哪些字段必填
- 返回结构是什么
- 失败时返回什么错误
- 是否会修改数据
- 能否重复调用

面试稳妥答法：

`我设计或理解一个接口时，会先看它是不是表达了清晰的业务动作。然后看入参、出参、错误码和幂等性。尤其是会修改数据的接口，不能只考虑成功路径，还要考虑重复提交、参数缺失和外部系统失败。`

### 2. GET 和 POST 怎么讲

- `GET`：偏查询，不应该修改数据
- `POST`：偏提交、创建或触发动作

不要背得太绝对，但要知道语义。

项目挂钩：

`产品生命周期系统里，查询产品提醒列表可以是 GET；新增或更新提醒配置更适合 POST 或 PUT；触发某个批次检查则更像一个动作型接口。`

### 3. 参数校验为什么重要

参数校验不是形式主义。

它的价值是：

- 早点发现错误
- 避免脏数据进入业务逻辑
- 让错误信息更可理解
- 减少后续排查成本

面试稳妥答法：

`参数校验越靠前，错误越容易处理。Controller 层可以做基础格式和必填校验，Service 层再做业务规则校验，比如产品状态、日期范围、权限或配置完整性。`

---

## 三、事务：一组操作要么都成功，要么都失败

事务最短定义：

`事务保证一组数据库操作作为一个整体执行，要么都成功，要么都回滚。`

### 1. 为什么金融系统特别在意事务

因为金融业务里很多数据不是孤立的。

比如：

- 主表新增了，明细表不能丢
- 批次状态更新了，结果明细不能没写
- 配置变更成功了，审计记录不能漏

面试稳妥答法：

`金融系统里很多操作有明显的一致性要求。比如保存一批稽核结果时，批次表、汇总表、明细表之间要保持一致。不能只写成功一部分，否则后续业务看到的状态就会混乱。`

### 2. 事务边界怎么理解

事务边界不是越大越好。

- 太小：可能保护不了完整业务操作
- 太大：锁时间变长，影响性能，也容易把外部调用拖进来

稳妥说法：

`事务边界应该围绕一个完整的本地数据变更来设计。数据库内部相关写入可以放在一个事务里，但外部接口调用一般要谨慎，不适合长时间占着数据库事务。`

---

## 四、幂等：重复执行，不应该把数据搞乱

幂等最短定义：

`同一个操作执行一次和执行多次，结果应该保持一致，至少不能产生重复副作用。`

这对你非常重要，因为你有批处理、测试、对账和稽核场景。

### 1. 为什么批处理要考虑幂等

批处理很可能失败后重跑。

如果没有幂等：

- 重跑会插入重复数据
- 状态会被覆盖错
- 结果会越来越乱

面试稳妥答法：

`批次类任务一定要考虑重跑。我的理解是要通过批次号、业务主键、状态字段或唯一约束控制幂等，确保失败后重跑不会重复插入或污染已有结果。`

### 2. 你的项目怎么讲

TA 自动化测试：

`同一批申请文件和确认文件对账，如果重跑一次，应该覆盖或复用同一个批次结果，而不是生成两份互相冲突的结果。`

TA 批前稽核：

`同一个产品、同一个交易日、同一个稽核批次，结果要能追溯，也要避免重复写入。`

产品生命周期提醒：

`同一提醒不应该因为任务重复执行而重复发很多封邮件，所以要有提醒记录或状态控制。`

## 五、数据一致性：不是所有一致性都靠一个事务

事务解决的是本地数据库的一致性。

但真实系统还有外部接口、文件、消息、批处理、人工修正。

所以你要有一个更稳的理解：

`一致性不是一个技术点，而是一组设计取舍。`

### 1. 强一致和最终一致

- `强一致`：操作完成后，所有相关方马上看到一致结果
- `最终一致`：中间允许短暂不一致，但最终通过补偿、重试、对账变一致

面试稳妥答法：

`在核心数据库写入里，我会优先保证强一致；但和外部系统、文件或异步流程交互时，很多场景更接近最终一致，需要靠状态、重试、补偿和对账来保证最后结果正确。`

### 2. 为什么金融系统需要对账

因为系统之间总会有边界。

边界一多，就可能出现：

- 我方成功，对方失败
- 文件收到了，数据库没更新
- 数据库更新了，通知没发出去
- 批处理跑完了，确认结果不一致

所以对账的价值是：

`把系统边界上的不一致暴露出来。`

TA 自动化测试就是很好的项目例子：

`申请文件、TA 批处理结果、确认文件、数据库状态之间，本质上就是在做一致性验证。`

## 六、可观测性：系统出事时能不能看见

可观测性不是一句潮词。

它回答的是：

`系统内部发生了什么，外面能不能通过日志、指标、链路信息看出来。`

### 1. 三类最常见信息

- `日志 Logs`：发生了什么事
- `指标 Metrics`：系统状态和数量变化
- `链路 Traces`：一次请求经过了哪些服务

面试不用讲很深，但要能说出区别：

`日志更适合看具体事件和异常，指标更适合看整体趋势和健康状态，链路追踪更适合看一次请求跨服务经过哪里、慢在哪里。`

### 2. 和小红书 CAT 怎么连

CAT 项目是你讲可观测性的好入口。

你可以说：

`我接触的小红书 CAT 监控平台，让我对应用监控有了更直观的认识。它不是业务系统本身，而是帮助开发和运维看见系统运行状态，比如调用是否成功、耗时是否异常、错误是否集中出现。`

如果被问 Transaction / Event / Heartbeat：

- `Transaction`：一次业务调用或流程
- `Event`：一个关键事件
- `Heartbeat`：系统健康状态
- `Metric`：指标数据

## 七、外部系统对接：最容易体现成熟度

Deanna 这类面试官有长期系统和数据架构经验，很可能关心：

`你对系统边界有没有概念。`

外部系统对接时，最该想清楚：

- 对方接口是否稳定
- 字段口径是否一致
- 失败怎么处理
- 超时怎么处理
- 是否需要重试
- 是否需要落日志或落库
- 是否需要人工介入

面试稳妥答法：

`我理解系统对接最重要的是边界清楚。比如数据中台、邮件/OA、TA 文件或外部代销数据，都不是我方系统内部简单方法调用。对接时要考虑字段映射、失败处理、超时、日志和补偿机制。`

## 八、怎么挂回你的项目

### 产品生命周期管理系统

可讲：

- REST API 设计
- 参数校验
- 数据中台对接
- 邮件/OA 外部接口
- 提醒幂等，避免重复通知
- 异常日志和业务可追溯

一句话：

`这个项目不是简单 CRUD，而是把特殊产品生命周期规则沉淀成接口、规则和提醒机制，同时要处理外部数据和通知链路的可靠性。`

### TA 自动化测试

可讲：

- 文件解析
- expected / actual 比对
- 数据库对账
- Jenkins Pipeline 触发
- 批次重跑和幂等
- 日志定位

一句话：

`TA 自动化测试的核心不是自动点按钮，而是把申请、处理、确认和数据库状态串成一条可验证链路。`

### TA 批前预警稽核

可讲：

- 规则配置
- 批次状态
- 稽核结果明细
- 失败重跑
- 风险前置
- 数据一致性

一句话：

`这个项目的价值是把人工经验前置成系统检查，关键是规则可扩展、结果可追溯、批次可恢复。`

### 小红书 CAT

可讲：

- Java / Maven 多模块
- 监控数据采集
- 报表聚合
- 告警链路
- 可观测性

一句话：

`CAT 让我接触到业务系统之外的运行监控视角，也就是系统上线之后如何被观察、排查和治理。`

## 九、最该背住的 12 句话

1. `后端接口不是 URL，而是业务能力的契约。`
2. `参数校验越靠前，错误越容易处理。`
3. `事务保护一组本地数据库操作的一致性。`
4. `事务边界不是越大越好，外部调用不应该随便包进长事务。`
5. `幂等解决的是重复调用或重跑时不要产生重复副作用。`
6. `批处理任务一定要考虑状态、日志、重跑和幂等。`
7. `强一致适合核心本地写入，最终一致常见于外部系统和异步流程。`
8. `对账的价值是暴露系统边界上的不一致。`
9. `日志看事件，指标看趋势，链路看路径。`
10. `可观测性解决的是系统出事时能不能看见。`
11. `外部系统对接要考虑字段口径、失败、超时、重试和补偿。`
12. `金融系统里，正确性、可追溯性和可恢复性往往比单纯跑通更重要。`

---

## 十、最后的记忆法

把系统面试记成一句话：

`接口定契约，事务保一致，幂等防重复，日志便追溯，监控看运行。`

这句话能把很多系统设计追问串起来。
