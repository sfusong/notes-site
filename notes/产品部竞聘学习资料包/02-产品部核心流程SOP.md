# 产品部核心流程 SOP

本文件用 SOP 视角整理产品部核心流程，重点看“何时触发、谁参与、准备什么、系统怎么做、哪里容易出错”。

## 1. 总览

| 流程 | 触发场景 | 核心目标 | 主要风险 |
|---|---|---|---|
| 产品方案 | 新产品、新策略、新系列、产品矩阵补位 | 判断产品是否值得做、能否做、怎么做 | 定位不清、策略与客群不匹配 |
| 产品委员会 | 产品规划、方案审批、重大产品事项 | 审批产品方案，形成跨部门共识 | 材料不足、风险收益说明不充分 |
| 定投资经理 | 新发产品进入申报前 | 明确主/副投资经理 | 投资经理信息缺失，系统导入失败 |
| 投资库与风控指标 | 新产品、新主题策略、资产范围变化 | 确认投资边界和风险监控条件 | 投资库未落地、风控指标未审批 |
| 申报登记 | 产品需取得登记编码 | 向理财登记中心报送并取得编码 | 材料不齐、导入模板口径不一致 |
| 发行登记 | 已取得登记编码，准备本次发行 | 固化募集、成立、份额、渠道、费率 | 多文件字段不一致，渠道上架受阻 |
| 销售文件生成 | 申报、发行、变更、增发份额 | 生成说明书、协议、揭示书等文件 | 模板关联错误，字段漏改 |
| TA 要素下发 | 新发、增发、变更、渠道调整 | 支撑 TA 设置和渠道销售 | TA 与说明书不一致 |
| 渠道上架 | 发行材料和 TA 要素准备完成 | 产品可在直销/代销渠道销售 | 协议、额度、客户组、代码错误 |
| 申报/发行要素修改 | 理财登字段、说明书或发行字段需修改 | 保持监管报送、销售文件和系统口径一致 | 修改流程走错、运营未同步 |
| 存续期管理 | 产品成立后发生要素或文件变化 | 持续合规运营和信息披露 | 流程选错、公告漏发、版本错用 |

## 2. 产品方案 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 销售端提出客户需求，投资端提出策略想法，产品部基于矩阵主动规划 |
| 前置条件 | 明确目标客群、渠道诉求、资产策略、风险收益特征 |
| 参与部门 | 产品部、销售端、投资端、风险合规端，必要时运营和科技参与 |
| 关键材料 | 产品方案、收益测算、情景分析、同业情况、风险说明 |
| 系统动作 | 通常未进入产品管理系统，偏方案和审批材料 |
| 注意事项 | 不要只写产品卖点，要说明适配客群、风险边界、投资范围、可操作性 |
| 产品部关注点 | 方案是否能转化成申报材料、销售文件、风险评级和系统字段 |

## 3. 产品委员会 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 定期复盘产品计划，或新产品、新策略需要审批 |
| 前置条件 | 产品方案基本成熟，销售、投资、风险等关键意见已汇总 |
| 参与部门 | 产品部牵头，销售、投资、风险合规等部门参与 |
| 关键材料 | 产品方案、产品计划、市场及同业分析、历史发行和规模数据 |
| 系统动作 | 一般不直接操作系统，审批结果影响后续产品创设 |
| 注意事项 | 明确产品是否按系列管理，后续同系列产品是否可沿用方案 |
| 产品部关注点 | 把产品规划和单只产品落地衔接起来，形成清晰审批依据 |

## 4. 定投资经理 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 新发产品准备申报前 |
| 前置条件 | 产品名称、产品代码、产品系列等基础信息基本确定 |
| 参与部门 | 产品部、投资相关部门、集中交易室或投决会相关条线、运营部 |
| 关键材料 | 投资经理初始设置与变更流程材料、共享文档、产品创设导入模板 |
| 系统动作 | 流程完成后导入产品管理系统，校验产品代码、投资经理、报备人员信息 |
| 注意事项 | 主/副投资经理如有多个，注意系统格式；新入职投资经理可能未开通用户 |
| 产品部关注点 | 该流程是申报登记前置，卡住会影响后续排期 |

## 5. 申报登记 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 产品需要向理财登记中心取得理财产品登记编码 |
| 前置条件 | 定投资经理完成，开放式产品完成流动性风险评估，风险评级表准备完成 |
| 参与部门 | 产品部、投资经理、销售经理、风险合规、运营部 |
| 关键材料 | 可行性评估报告、相关人员尽调报告、销售协议及三书、产品说明书、发行预告、流动性风险评估报告、申报登记导入模板 |
| 系统动作 | 产品管理系统发起产品申报登记信息流程，投销复核，一键生成文件，导出中债报表和申报文件 |
| 注意事项 | 公募下编码约 10 个工作日，私募下编码约 2 个工作日；申报版文件部分发行字段可能留空 |
| 产品部关注点 | 材料齐备、字段一致、流程在运营报送时间前完成 |

## 6. 投资库与风控指标 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 新产品发行、主题型产品发行、投资范围或风险监控口径变化 |
| 前置条件 | 产品方案和投资策略基本明确，主投资经理可判断适用投资库 |
| 参与部门 | 产品部、主投资部门、风险合规部、投决会相关人员，必要时运营和科技参与 |
| 关键材料 | 投资库管理与风险管控说明、投决会材料、风险指标方案、投资库落地确认 |
| 系统动作 | 资料中偏流程和审批口径，系统侧可关注投资库、风控指标、产品说明书投资范围之间的字段映射 |
| 注意事项 | 主题型产品要判断是否需新建主题投资库；风控指标需明确比例、计算规则、监控开始时间和超限处置 |
| 产品部关注点 | 把风险管控前置到产品方案和发行排期，避免发行登记前才发现投资库或风控指标未落地 |

## 7. 发行登记 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 已取得登记编码，准备某次具体募集和销售 |
| 前置条件 | 登记编码已取得，发行计划、渠道、份额、费用、期限等要素确定 |
| 参与部门 | 产品部、投资经理、销售经理、风险合规、运营部、代销渠道 |
| 关键材料 | 销售协议+四书、发行登记导入模板、委托销售产品编码导入模板、产品变更登记批量导入模板、TA 要素 |
| 系统动作 | 产品管理系统发起产品发行要素流程，投销填写复核，产品经理一键生成销售文件，导出发行登记文档压缩包 |
| 注意事项 | 常规 T 日起募、T-2 做发行登记；加急 T-1 做发行登记，去纸化流程需尽早流转至运营 |
| 产品部关注点 | 募集期、成立日、到期日、渠道、份额、费率、业绩比较基准不能错 |

## 8. 销售文件生成 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 申报登记、发行登记、销售文件修改、募集期修改、存续期修改、增发份额 |
| 前置条件 | 产品三级系列已关联对应文档模板，相关要素已填写完成 |
| 参与部门 | 产品部、销售部、投资端、风险合规，必要时运营 |
| 关键材料 | 产品说明书、投资协议书、销售协议书、风险揭示书、投资者权益须知、临时公告 |
| 系统动作 | 产品经理复核投销要素后点击一键生成或在线生成 |
| 注意事项 | 模板关联错误会导致无法生成；公告标题需与内容标题完全一致 |
| 产品部关注点 | 销售文件是投资者可见文本，必须与系统要素和报送模板一致 |

## 9. TA 要素 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 新发产品、增发子份额、渠道新增、额度修改、开放日或披露规则设置 |
| 前置条件 | 产品发行或变更要素已确定 |
| 参与部门 | 产品部、销售部、运营部、科技或系统支持 |
| 关键材料 | 产品要素导出表、TA 要素表格式转换模板、销售商额度信息 |
| 系统动作 | 接口上线前手动转换 TA 要素并提交 OA 去纸化；接口上线后多数要素自动传输 |
| 注意事项 | 产品开放日信息、净值披露规则等文字型信息可能仍需手工提交运营 |
| 产品部关注点 | TA 要素决定系统交易、额度、费率、披露和渠道展示规则 |

## 10. 风险评级 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 产品申报登记前或产品风险等级需评估调整 |
| 前置条件 | 产品投资范围、资产配置比例、估值方式等基本明确 |
| 参与部门 | 产品部、投资经理、风险合规部 |
| 关键材料 | 理财产品风险评估及评级意见审批单/风险评级表、风险等级评定细则 |
| 系统动作 | 资料中主要体现为线下填表和流程上传，具体以系统当前版本为准 |
| 注意事项 | 风险等级应与资产投向、比例、估值方法和销售适当性一致 |
| 产品部关注点 | 风险评级影响目标客群、渠道销售和风险揭示 |

## 11. 流动性风险评估 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 开放式产品申报登记前 |
| 前置条件 | 定投资经理完成，申报登记材料和风险评级材料基本准备 |
| 参与部门 | 产品部、主投资经理、风险合规经办、风险经理、相关部门负责人 |
| 关键材料 | 流动性风险评估报告、申报登记材料、风险评级表 |
| 系统动作 | OA 去纸化流程留痕，扫描版或归档文件上传产品管理系统 |
| 注意事项 | 封闭式产品不需要该流程；开放式产品需关注赎回压力和资产流动性 |
| 产品部关注点 | 流动性评估是开放式产品申报链路的重要前置材料 |

## 12. 申报要素修改 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 已申报产品的申报字段、说明书或监管报送口径需要调整 |
| 前置条件 | 明确修改字段、修改原因、是否涉及说明书修订和公告 |
| 参与部门 | 产品部、综合部或报送相关岗位、运营部、风险合规，必要时投资和销售参与 |
| 关键材料 | 申报要素修改单、修订后的说明书、相关公告或说明材料 |
| 系统动作 | 根据资料口径，涉及监管报送时需通过专网邮箱或报送流程提交；系统侧关注修改后的字段同步和留痕 |
| 注意事项 | 如果申报要素修改涉及说明书修改，应先完成报备和用印，再进入实际说明书修改链路 |
| 产品部关注点 | 申报字段、说明书、公告、运营报送和归档材料必须形成闭环 |

## 13. 发行要素修改 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 发行登记字段、募集安排、渠道、份额、费率等发行要素需要调整 |
| 前置条件 | 明确修改事项是否影响销售文件、TA、渠道、信披和理财登 |
| 参与部门 | 产品部、销售部、运营部、风险合规，必要时投资端和渠道参与 |
| 关键材料 | 发行登记要素修改表、修改后的销售文件、公告、委托销售编码导入或停用模板 |
| 系统动作 | 产品管理系统或线下表格发起修改，同步运营处理并保留审批/邮件/去纸化留痕 |
| 注意事项 | 说明书修改、TA 修改、渠道上下架可能分别触发不同流程，不能只改一个地方 |
| 产品部关注点 | 明确“改了哪个字段、影响哪些文件和系统、何时生效、谁已确认” |

## 14. 存续期变更 SOP

| 项目 | 内容 |
|---|---|
| 触发场景 | 产品成立后发生销售文件修改、要素变更、信披公告、估值条款调整、渠道增删、增发份额 |
| 前置条件 | 明确变更类型、影响范围、是否需要公告、是否涉及 TA 或理财登 |
| 参与部门 | 产品部、销售部、投资端、风险合规、运营部、渠道 |
| 关键材料 | 修改后的销售文件、临时公告、委托销售编码导入或停用模板、产品要素导出表 |
| 系统动作 | 根据场景选择销售文件管理流程、要素变更流程、信披公告管理流程、估值条款涉及变更流程 |
| 注意事项 | 有行外代销机构和无行外代销机构的流程提前量不同；销售文件更新时间影响报送和信披下载 |
| 产品部关注点 | 选对流程、控住版本、保证变更后的销售文件、TA、信披、渠道口径一致 |

## 15. 易错点清单

| 易错点 | 风险 | 建议控制 |
|---|---|---|
| 申报登记和发行登记混淆 | 排期和材料准备错误 | 申报看登记编码，发行看本次销售上架 |
| 四书、三书、销售协议混淆 | 材料清单不完整 | 发行记“销售协议+四书” |
| 投资经理设置未完成 | 后续系统导入或申报卡住 | 将定投资经理作为申报前置检查项 |
| 文档模板未关联三级系列 | 销售文件无法自动生成 | 创设产品时检查模板关联 |
| 合同费率和执行费率混淆 | 费用设置错误 | 执行费率不得高于合同费率 |
| TA 要素与说明书不一致 | 渠道上架或交易异常 | 建立字段一致性校验 |
| 存续期流程选错 | 审批节点或公告遗漏 | 先判断是否涉及四书、信披、估值条款 |
| 接口上线前后口径不清 | 重复提交或漏提交 | 明确哪些要素已自动传输，哪些仍需 OA |
| 投资库和风控指标后置确认 | 产品方案已过但发行前风险条件未落地 | 主题产品和特殊投资范围产品前置确认投资库、投决会和风控指标 |
| 申报要素修改与发行要素修改混用 | 报送口径、销售文件或运营处理漏同步 | 先判断改的是申报字段还是发行字段，再匹配材料和流程 |
