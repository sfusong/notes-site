## 【与第1讲的衔接】

第1讲建立了资管行业的整体认知框架：搞清楚了中国资管行业由哪几类机构组成、各自受谁监管、2018年资管新规如何重塑了行业格局，以及资管科技系统的"前中后台"三层架构。

## 一、学习目标

1. 能够完整描述资管产品从设计到退出的全生命周期，覆盖六个阶段的核心业务动作；
2. 能够区分关键术语：募集期与存续期、认购与申购、封闭式与开放式、业绩比较基准与预期收益率；
3. 能够说清楚产品管理系统在产品生命周期中的定位，包括它管理哪些信息、支撑哪些流程、与哪些系统发生数据交换；
4. 能够从产品经理视角分析产品生命周期各阶段对系统建设提出的核心需求；
5. 能够将"主导产品管理系统建设"的经历，转化为有说服力的面试叙事。

## 二、本讲核心结论

1. **资管产品的生命周期分为六个阶段**：产品设计→内部审批→产品登记（理财登）→发行募集→存续运营→产品终止。
2. **"认购"和"申购"是不同阶段的不同操作**：认购发生在募集期，申购发生在存续期开放日，两者在净值基准、系统处理逻辑上完全不同。
3. **产品说明书是产品生命周期的"宪法"**：所有运营行为必须严格按说明书执行，任何不一致都是合规风险。
4. **产品管理系统是连接前台业务设计和后台运营执行的"中枢"**：作为产品信息的单一权威来源，向TA、估值、信披等系统推送数据。
5. **理财子产品的生命周期管理比公募基金更复杂**：产品形态更多样，监管口径不同，母行渠道约束强，系统定制化需求高。
6. **产品生命周期的每一个"状态切换"都是系统建设的关键节点**：每次状态变更触发一系列系统联动动作，产品经理必须把这套联动逻辑设计清楚。

## 四、行业背景与业务定位

### 4.1 资管产品的分类

**按运作方式：**

| 类型 | 含义 | 典型特征 |
|------|------|----------|
| 封闭式 | 募集期结束后封闭运作，到期才能赎回 | 期限固定，流动性低，收益相对稳定 |
| 开放式 | 在存续期内设置开放日，可申购赎回 | 流动性好，但净值波动对投资者可见 |
| 滚动型 | 每隔固定周期开放一次，不开放时封闭 | 兼顾流动性与稳定性，理财子常见 |

**按投资方向（风险等级）：**

| 风险等级 | 产品类型 | 主要投资标的 |
|----------|----------|--------------|
| R1（低风险） | 现金管理类 | 货币市场工具、短期债券 |
| R2（中低风险） | 固定收益类 | 债券、存款、非标资产 |
| R3（中风险） | 固收+ | 固收打底 + 少量权益 |
| R4（中高风险） | 混合类 | 股债混合，权益占比较高 |
| R5（高风险） | 权益类 | 股票为主 |

> 理财子目前产品以R1–R3为主，这与理财子的客群以银行渠道稳健型投资者为主密切相关。

### 4.2 理财子产品的特殊背景

- **产品登记平台：理财登**：理财子常规发行新产品，须在**理财登**（全国银行业理财信息登记系统，隶属中债登旗下）完成产品信息登记，取得以 **"Z"开头** 的登记编码后方可对外销售。系统端只与理财登交互，**无需单独向金融监管总局申报**；公募基金则向证监会注册，两套体系流程存在明显差异
- **例外情况（非常态）**：若涉及特殊复杂的新业务类型，则需向属地金融监管局（如宁波金监局）报备并写文件，等待监管批准后方可开展——这是**新业务资质审批**，不是单个产品发行的常规流程
- **母行渠道约束强**：产品设计需与母行营业网点、APP的销售节奏协调，产品期限、起购金额等参数受渠道反馈影响
- **非标资产占比高**：部分产品投资非标准化债权资产，估值和清算逻辑比标准债券复杂
- **产品说明书格式不同**：使用银行理财体系的模板，与基金合同格式不同

## 五、核心概念与术语

### 5.1 募集期 vs 存续期 vs 封闭期

| 概念 | 定义 | 投资者能否操作 |
|------|------|----------------|
| 募集期 | 对外销售、接受认购的阶段，产品尚未正式成立 | 可认购，不可赎回 |
| 封闭期 | 产品成立后，某段时间内不接受申赎操作 | 不可申购，不可赎回 |
| 存续期 | 产品正式成立后到到期终止前的整个运作阶段 | 视产品类型，开放日可操作 |

### 5.2 认购 vs 申购

| 维度 | 认购 | 申购 |
|------|------|------|
| 发生时间 | 募集期内 | 产品成立后的开放日 |
| 产品状态 | 尚未成立 | 已成立，正在运作 |
| 净值基准 | 通常以1元面值计算 | 以当日净值计算 |
| 份额确认时间 | 产品成立日统一确认 | 开放日按规则确认（T日或T+1日） |
| 失败可能性 | 有（募集不足则失败，原路退款） | 无 |

### 5.3 产品说明书 vs 风险揭示书

- **产品说明书**：产品的"宪法级"文件，详细描述投资范围、运作规则、费率结构、开放安排、分红方式等。所有运营行为必须与说明书一致。
- **风险揭示书**：专门向投资者揭示产品可能面临的各类风险，是合规销售的必备文件。

### 5.4 业绩比较基准 vs 预期收益率

- **预期收益率**（旧模式）：产品明确承诺"年化收益4.5%"，实质是刚性兑付，违反资管新规
- **业绩比较基准**（新模式）：设定参考标准（如"3年期国债收益率+1.5%"），用于衡量管理人投资表现，**不承诺**一定能达到

### 5.5 开放日（Open Day）

允许投资者进行申购或赎回操作的特定日期，频率由产品说明书规定：每日/每周/每月/每季度开放。开放日是TA系统工作量最集中、运营风险最高的时间节点。

### 5.6 投资委员会（IC）

资管机构内部负责审核产品投资策略、审批重大投资决策的最高议事机构。新产品上线前通常需经过IC审批，确认投资方向和风险承受能力符合公司整体策略。

```

  │         → 起草产品说明书 → 确定业绩比较基准和费率结构           │
  └──────────────────────────┬───────────────────────────────────┘
                             │ 提交审批材料
                             ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  阶段二：内部审批                                              │
  │  产品部门提交 → 合规审核 → 风险审核 → 投资委员会审议            │
  │  ✓ 通过 → 生成内部审批单据，进入下一阶段                       │
  │  ✗ 不通过 → 退回修改，重新提交                                │

  ┌──────────────────────────────────────────────────────────────┐

  │  理财登隶属中债登，系统端只与理财登交互                            │
  │  取得登记编码 → 方可对外发行销售                                  │
  └──────────────────────────┬───────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────┐
  │  阶段四：发行募集                                               │
  │  制作销售材料 → 对接代销渠道 → 发布产品公告                     │
  │  开放认购通道 → 募集期内接受投资者认购                          │
  │  募集期结束：                                                   │
  │    ✓ 达到成立门槛 → 产品成立，发布成立公告                      │
  │    ✗ 未达成立门槛 → 募集失败，退还认购资金                      │
  └──────────────────────────┬───────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────┐
  │  阶段五：存续运营（持续进行，直至产品终止）                       │
  │  · 每日估值（计算NAV）                                         │
  │  · 开放日申赎处理（TA系统）                                    │
  │  · 定期信息披露（净值/持仓/报告）                               │
  │  · 投资组合调整（交易执行）                                    │
  │  · 收益分配/分红（如有）                                       │
  │  · 监管报送（月报/季报/年报）                                   │

                             ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  阶段六：产品终止                                               │
  │  到期终止：公告→停止申赎→资产处置→清算→份额清偿→清算报告        │
  │  提前终止：触发条件→持有人大会（如需）→公告→清算流程            │
  └──────────────────────────────────────────────────────────────┘
```

### 6.2 产品设计阶段的核心决策参数

| 决策维度 | 具体内容 | 对系统的影响 |
|----------|----------|--------------|
| 产品类型 | 封闭式/开放式/滚动型 | 影响TA系统申赎规则配置 |
| 运作期限 | 1个月/3个月/1年/无固定期限 | 影响产品到期提醒和自动终止逻辑 |
| 投资范围 | 债券/非标/股票等 | 影响估值方法选择（摊余成本法/市值法） |
| 风险等级 | R1–R5 | 影响销售渠道的客户适当性核验逻辑 |
| 起购金额 | 1元/1万元/10万元等 | 影响TA系统的申购下限校验逻辑 |
| 开放频率 | 每日/每周/每月等 | 直接配置在TA系统的开放日历中 |
| 费率结构 | 管理费/托管费/销售费/赎回费 | 影响清算系统的费用扣除计算逻辑 |
| 业绩比较基准 | 如"一年期LPR+1%" | 用于信息披露和业绩评估 |

### 6.3 存续运营的日常工作节奏

```
【存续期日常操作节奏（月度视角，滚动型产品举例）】

  每个交易日：
    09:30 前   前一日净值确认，推送至信息披露系统
    全天        投资经理调整持仓，交易系统执行
    15:00 后   估值系统开始计算当日净值
    17:00 前   净值计算完成，推送托管行复核
    18:00 前   托管行复核完成，净值正式确认

  每月开放日（如每月15日）：
    T-3日      TA系统开放申赎接收通道
    T日        截止申赎登记，汇总当日申赎数据
    T+1日      确认份额，生成申赎清算数据
    T+2日      资金完成清算
    T+2日      运营部门完成对账

  每季度：
    季末月     触发季度报告编写流程
    季末后15日 完成季度报告，发布至信息披露渠道

  每年：
    年末        年度报告编写，外部审计介入
    年后规定日期 年度报告正式发布
```

## 七、参与角色与部门分工

### 7.1 内部角色分工

| 部门/角色 | 主要职责 | 最集中于哪个阶段 |
|-----------|----------|------------------|
| 产品部门 | 产品设计、说明书起草、产品档案管理 | 设计阶段、审批阶段 |
| 合规法务部 | 审核产品合规性、准备理财登登记材料 | 审批阶段、登记阶段 |
| 风险管理部 | 评估产品风险，出具风险审核意见 | 审批阶段 |
| 投资委员会 | 审议产品投资策略，最终决策 | 审批阶段 |
| 投资/交易部门 | 管理产品投资组合，执行交易 | 存续运营阶段 |
| 运营结算部 | 日常申赎处理、清算、对账 | 存续运营阶段、终止阶段 |
| IT/科技部门 | 系统配置、新产品参数录入、接口维护 | 各阶段均有参与 |
| 渠道/销售部门 | 与代销渠道对接，推动产品销售 | 发行募集阶段 |
| 信息披露部门 | 编制和发布各类产品公告及报告 | 发行阶段、存续阶段、终止阶段 |

### 7.2 外部角色的介入时机

| 外部角色 | 介入时机 | 主要工作 |
|----------|----------|----------|
| 托管行 | 产品成立前签署托管协议，成立后持续介入 | 资产保管、净值复核、合规监督 |
| 代销渠道 | 发行募集阶段开始介入 | 面向投资者销售，传递申赎指令 |
| 监管机构（金融监管总局） | 日常监督及特殊新业务审批 | 制定规则、日常合规监督；新业务类型需报备审批 |
| 理财登（中债登旗下） | 产品登记阶段、存续期信息披露 | 接收产品登记信息，发放Z开头登记编码，接收存续期净值等数据更新 |
| 审计机构 | 年度审计阶段 | 出具审计报告，配合年度信息披露 |

## 八、系统支撑与数据流转

## 九、站在科技 / 产品 / 项目经理视角应该关注什么

### 9.1 产品"状态机"设计是核心难题

```
【产品状态机（简化版）】

  [草稿] ──提交审批──▶ [审批中] ──审批通过──▶ [待发行]
                           │                      │
                        驳回│                  启动募集│
                           ▼                      ▼
                         [草稿]              [募集中]
                                                  │
                                         成立条件满足│  募集失败
                                                  ▼       ▼
                                             [存续中]  [已失效]
                                                  │
                                           到期/触发│
                                                  ▼
                                             [终止中]
                                                  │
                                         清算完成  │
                                                  ▼
                                             [已终止]
```

每一次状态切换，都会触发对应的系统动作：
- **[待发行] → [募集中]**：触发TA系统开放认购通道，触发信披系统发布募集公告
- **[募集中] → [存续中]**：触发TA系统关闭认购通道、开放正式申赎规则，触发估值系统初始化产品净值
- **[存续中] → [终止中]**：触发TA系统关闭申赎通道，触发清算系统启动清算流程

### 9.2 产品参数录入的"第一公里"问题

新产品上线时最常见的问题不是系统Bug，而是**参数录入错误**：
- 开放日历录入错误（导致非开放日也能申赎）
- 费率配置错误（导致费用计算偏差）
- 投资范围配置错误（导致风控系统误报警）
- 产品代码重复或格式不规范（导致下游系统无法识别）

解决方案：产品管理系统需要完善的**参数校验规则**和**四眼原则**（录入人员与复核人员分离）。

### 9.3 系统建设优先级判断

| 优先级 | 系统/功能 | 理由 |
|--------|-----------|------|
| P0（必须有） | 产品基础信息管理 | 没有这个什么都做不了 |
| P0（必须有） | 审批流转记录 | 合规留痕要求 |
| P1（尽快有） | 产品状态管理 | 各下游系统的信息同步依赖它 |
| P1（尽快有） | 与TA系统的产品参数对接 | 减少手工录入错误 |
| P2（有余力做） | 产品档案全文检索 | 提升运营效率 |
| P2（有余力做） | 监管报送数据自动生成 | 减少合规风险 |

## 十一、常见误区与易混点

### 误区1："认购"和"申购"可以混用

不可以。两个词对应不同阶段、不同的系统处理逻辑和净值基准。混用会被有经验的面试官直接扣分。

### 误区2：产品说明书只是"文案材料"

产品说明书是具有法律效力的合同类文件，是产品所有运营行为的约束边界。系统配置必须与说明书完全一致，任何不一致都是合规风险。

### 误区3：产品管理系统就是"产品信息的Excel升级版"

这个理解太浅。产品管理系统的核心价值在于：流程管控（审批流程数字化）、数据一致性（单一权威来源）、状态联动（状态变更触发系统动作）、合规支撑（确保运营符合说明书约束）。

### 误区4：产品成立后就进入"自动驾驶"模式

存续期的运营工作非常密集：每日估值、开放日申赎、定期报告、监管报送、异常处理……任何环节出问题都可能影响投资者权益或触发合规风险。

### 误区5：产品终止等于产品到期

产品有三种终止方式：**到期终止**（最常见）、**提前终止**（规模低于最低存续规模等触发条件）、**主动清盘**（管理人决定）。提前终止处理流程比到期终止复杂得多。

## 十二、案例分析

### 案例：某理财子新产品上线"参数不一致"引发的系统事故

**背景：**

某理财子推出一款固收类滚动开放型理财产品，产品说明书约定每月第15个**自然日**为开放日（遇非工作日顺延至下一工作日）。产品成立后，IT工程师在TA系统中配置开放日历时，误将"每月15号自然日"理解为"每月第15个工作日"，导致开放日期与说明书不一致。

**事故经过：**

产品上线第二个月，投资者发现实际开放日比说明书晚了约3–5个工作日，纷纷投诉。合规部门介入调查后发现是系统配置错误，修复期间当月开放日临时暂停，部分急需资金的投资者无法按期赎回，引发舆情风险。

**根本原因分析：**

```
根本原因：
  1. 产品说明书与系统配置之间，没有"逐条核对"的标准流程
  2. IT工程师对业务规则理解存在偏差，且没有业务人员复核
  3. 上线前测试用例中，没有包含"特殊日期顺延"的边界测试

直接原因：
  · "自然日第15日" ≠ "第15个工作日"
  · 没有四眼原则（业务+IT各自独立确认参数）
```

**产品经理应该如何避免：**

1. **建立产品参数核对清单**：说明书中每一个影响系统配置的参数，都要有对应的系统字段映射表，上线前业务+IT各自独立核对签字
2. **设计边界测试用例**：开放日历的测试必须包含节假日、月末、月初等特殊情况
3. **产品管理系统统一管理规则**：开放日历的业务规则应在产品管理系统中定义并推送给TA系统，而不是由TA系统独立配置

**案例核心价值：** 说明了为什么需要统一的产品管理系统——当产品信息在多个系统中分散维护时，不一致问题几乎无法避免。

能说清楚产品管理系统向TA系统推送开放日历和申赎规则，向估值系统推送产品类型和估值方法，向信披系统推送产品基础信息。这种系统集成视角，是技术背景产品经理的核心竞争力之一。

**第5点：案例（参数不一致事故）可以用作面试中"你遇到过什么系统建设难题"的素材**

这类案例帮助在面试中表现出"我理解这类风险是怎么发生的，以及如何用系统设计来预防它"的成熟度，是业务理解深度的体现。

## 十八、课堂提问

> 以下5个问题是老师在课上最可能追问的，建议合上笔记口头作答后再核对。

---

**问题1（理解型）：募集期和存续期有什么区别？投资者在这两个阶段做的操作，为什么叫"认购"和"申购"而不是同一个词？**

> 💡 提示方向：从产品是否已成立、净值基准是否确定、份额确认时间三个维度展开。想想如果募集期失败了，已"认购"的资金怎么处理？

---

**问题2（理解型）：产品说明书对系统建设有什么约束？如果系统配置和说明书不一致，会带来什么风险？**

> 💡 提示方向：产品说明书是运营行为的"宪法"，想想本讲案例中参数不一致引发了什么。合规、投资者权益、监管三个维度都要想到。

---

**问题3（分析型）：产品管理系统为什么需要"状态机"设计？如果不设计状态机，会出现什么问题？**

> 💡 提示方向：想想产品从募集中变成存续中的那一刻，有多少系统需要同时知道这个变化并做出响应。如果没有统一的状态管理，会出现什么数据不一致情况？

---

**问题4（分析型）：你说产品管理系统是"单一权威来源"，这个设计原则解决了什么问题？如果每个系统各自维护一份产品信息，会出现哪三类风险？**

> 💡 提示方向：参数版本不一致、修改成本高（改一处需同步5个系统）、审计难（不知道哪个系统的数据才是准的）。

---

**问题5（综合型）：一个有5年经验的资管业务人员和一个有5年经验的互联网产品经理，同时应聘理财子产品管理系统的产品经理岗位，你认为各自的优劣势是什么？你的背景相比这两类人有什么差异化优势？**

> 💡 提示方向：业务人员懂业务但可能不懂系统设计；互联网PM懂产品方法但不懂资管业务规则和监管约束。你的差异化是同时有计算机背景和资管系统建设实战经验，关键要能讲出具体例证，而不是泛泛而谈。

## 十九、我应该记住的关键句

> 以下9句话可直接用于面试表达和业务沟通，建议反复练习直到脱口而出。

---

**① `[业务理解]`**
> "认购和申购是两个完全不同的操作：认购发生在募集期，产品还没成立，投资者是在'预定'份额；申购发生在存续期的开放日，产品已经在运作了，按当日净值买入。混淆这两个词，说明对产品生命周期没有基础认知。"

---

**② `[业务理解]`**
> "资管新规之后，'预期收益率'这个词基本从产品说明书里消失了，取而代之的是'业绩比较基准'。两者的本质区别是：前者是承诺，后者是参考标准。这个变化背后，是整个行业从刚性兑付文化向净值化文化的转变。"

---

**③ `[系统表达]`**
> "产品管理系统最重要的设计原则是'单一权威来源'：产品的所有基础信息只在一个地方维护，TA系统、估值系统、信披系统都从这里取数，而不是各自维护一份。一旦需要修改，改一处就够了，不会出现各系统数据打架的情况。"

---

**④ `[系统表达]`**
> "产品管理系统的核心难点不是存数据，而是管状态。产品从草稿到审批中、从募集中到存续中，每一次状态切换都会触发一系列下游系统的联动动作。如果状态机设计不清楚，就会出现'某个系统还以为产品在募集期'的数据不一致问题，直接影响投资者的申赎操作。"

---

**⑤ `[业务理解]`**
> "很多人以为产品成立之后就进入'自动驾驶'了，其实存续期才是运营最繁重的阶段：每天要估值、开放日要处理申赎、每季度要出报告、每年要接受审计，任何一个环节出了问题都是合规风险。支撑这一切的，是每天都在跑的后台系统。"

---

**⑥ `[系统表达]`**
> "产品说明书对于系统建设来说，是最重要的需求文档之一。开放日历、申赎规则、费率结构，这些系统参数都必须以说明书为准逐条核对，不能靠'我理解你的意思'来配置。一个字的偏差，比如'自然日'和'工作日'，可能导致整个开放日流程错误。"

---

**⑦ `[自我定位]`**
> "我在理财子做产品管理系统，本质上是在把一套本来靠人工和Excel维持的产品生命周期管理流程，用系统来标准化、自动化。这不只是IT项目，更是业务流程再造——我需要同时理解业务规则和系统实现，才能做出一个业务部门愿意用、IT部门能建出来的系统。"

---

**⑧ `[面试收尾]`**
> "理财子的产品管理系统建设，跟公募基金不一样的地方在于：公募的供应商方案已经很成熟，直接部署就能用；理财子的业务特点跟公募差异很大，供应商标准产品往往只能覆盖60–70%的需求，剩下30–40%需要定制化开发。这也是为什么理财子需要懂业务的甲方PM，而不只是会管项目的项目经理。"

---

**⑨ `[行业认知]`**
> "产品生命周期管理是资管机构所有业务的'主干线'——TA系统处理的申赎、估值系统计算的净值、信披系统发布的公告，都是围绕着产品某个生命周期阶段在运转的。理解了产品生命周期，你才能理解为什么每个系统要在那个时间点做那件事。"

---
