# 产品生命周期管理系统面试资料

## 1. 30 秒项目介绍

这是一个内部业务系统，主要解决两类产品生命周期事件管理问题：

- 分红封闭式产品：需要根据成立日、基础日历和分红计划，提前提醒产品经理和投资经理做分红准备
- 目标盈产品：当年化收益率达到阈值时，需要触发提前到期提醒，并通知产品、投资、销售等相关方处理

我在里面参与了实际开发，技术上使用了 `Java Spring Boot + React`，本质上是把业务规则转成系统可执行的提醒逻辑，并在产品管理系统正式上线前先行投入业务使用。

## 2. 可确认事实

- 是宁银理财内部系统
- 技术栈：`Java Spring Boot + React`
- 目标：支持分红封闭式产品和目标盈产品的关键事件提醒
- 规则来源：成立日、基础日历、分红计划、收益达标条件
- 价值：在大产品管理系统上线前先承担生命周期管理需求

## 3. 组件和技术解释

### Java

这里主要承担后端业务逻辑实现。你可以这样理解：
- 接收前端请求
- 处理业务规则
- 读写数据库
- 返回提醒结果或任务状态

### Spring Boot

Spring Boot 是 Java 企业应用开发框架，适合快速搭建：
- REST API
- 业务服务层
- 定时任务
- 数据访问层
- 配置管理

如果面试官问“为什么适合这个项目”，你可以答：

因为这个系统核心不是算法，而是业务规则、提醒逻辑和接口交互。Spring Boot 在企业业务系统里成熟度高，开发效率高，也方便后续继续并入更大的产品管理系统。

### React

React 是前端框架，适合做：
- 产品事件列表页面
- 提醒状态页面
- 规则配置页面
- 任务确认页面

你可以说它在这个系统里主要承担业务操作界面和前后端联调。

### 规则触发

这是这个项目最值得讲的点。  
不要讲成“做了个定时提醒工具”，要讲成：

`基于业务日历和产品规则的事件驱动提醒系统`

### 基础日历

基础日历可以理解为：
- 交易日/非交易日判断
- 业务日期推算
- 分红准备提前天数计算

如果面试官问“为什么要有基础日历”，你可以答：

因为提醒时点不能只按自然日算，必须考虑交易日、节假日和业务实际处理时间，所以基础日历是规则计算的基础。

## 4. 面试可用的架构理解

下面这版是“合理推断版”，可以作为你面试时的稳定说法：

```text
React 前端
  -> 产品事件列表 / 提醒任务页面 / 配置页面
  -> 调用 Spring Boot API

Spring Boot 后端
  -> 产品主数据查询
  -> 生命周期规则引擎/判断逻辑
  -> 定时任务或批处理扫描
  -> 提醒记录落库
  -> 返回给前端或推送给相关用户

数据库
  -> 产品基础信息
  -> 分红计划
  -> 目标盈规则
  -> 事件提醒记录
  -> 状态流转记录
```

你可以补一句：

“虽然这个系统规模不大，但它本质上已经具备了典型企业业务系统的后端规则处理、定时扫描和前后端协同结构。”

## 5. 面试官最可能问的问题

### Q1. 为什么会做这个系统，而不是直接放进产品管理系统？

建议回答：

因为业务需求是即时存在的，而产品管理系统建设有周期。生命周期管理，尤其是分红准备和目标盈到期提醒，是比较刚性的运营需求，所以先做一个更聚焦的系统承接需求，等主系统成熟后再并入更大的平台能力。

### Q2. 这个系统最核心的技术难点是什么？

建议回答：

最核心的难点不是技术框架本身，而是如何把业务规则准确转成系统逻辑。因为提醒时点不是固定的，而是要结合成立日、产品日历、分红计划和收益阈值综合判断，所以关键是规则表达和触发逻辑设计。

### Q3. 这个系统怎么判断“该提醒了”？

你可以这样说：

系统会根据产品类型读取不同规则。对分红封闭式产品，会结合成立日、基础日历和分红计划计算准备窗口；对目标盈产品，会结合收益达标条件判断是否进入提前到期处理流程。一旦命中规则，就生成提醒任务并通知相关角色。

### Q4. 为什么要用 Spring Boot？

回答：

因为它适合做这种规则驱动的企业业务系统，接口层、服务层、定时任务、配置和数据访问都比较成熟，而且后续更容易和公司内部其他系统对接。

### Q5. 这个系统的前端 React 主要做什么？

回答：

前端主要承载提醒任务查看、状态确认、列表筛选以及相关操作入口。重点不是复杂交互，而是把规则触发后的任务状态、处理节点和相关产品信息清晰地呈现出来。

### Q6. 这个系统上线后价值是什么？

回答：

价值主要是把原来容易被遗漏或依赖人工记忆的生命周期事件管理系统化，提前提醒相关角色，降低运营风险，也为后续主系统上线前先行支撑业务协同。

### Q7. 你具体写了什么代码？

稳妥回答：

我在这个项目里有实际开发投入，主要参与了后端规则逻辑和系统实现，也参与了前后端联调。它不是纯需求推进型项目，而是一个真实的内部业务系统开发项目。

如果面试官继续追：

“偏后端还是偏前端？”

你可以答：

我更偏后端和业务规则实现，但前端联调和页面配合我也有参与。

## 6. 面试官可能追问的技术点

### 1. 定时任务怎么做？

如果问到：

“你们是不是定时扫一遍数据？”

你可以答：

大概率是通过后端定时任务或批量扫描机制去判断哪些产品进入提醒窗口。具体实现可以是 Spring 的定时任务能力，核心是按规则扫描并生成提醒记录。

这里是“合理推断版”，你如果不确定有没有真的用定时任务，可以改口成：

“从系统逻辑上，这类提醒一般会有定时扫描或批量计算机制，我参与的重点更多在规则和系统实现层。”

### 2. 提醒记录怎么防止重复生成？

一个比较像样的回答：

通常会通过“产品 + 事件类型 + 业务日期/批次”这种维度去做唯一性约束，或者在生成前先查历史状态，避免重复提醒。

### 3. 如果某个提醒已经处理了，后面怎么办？

回答：

一般会有提醒状态流转，比如待处理、已处理、忽略、失效等。这样系统不只是发通知，还能记录后续处理状态。
