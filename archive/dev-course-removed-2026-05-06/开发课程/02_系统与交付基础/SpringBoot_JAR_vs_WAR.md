# Spring Boot 为什么常打 jar 不打 war

这也是 Java 开发岗里很常见的一道题。

如果只记一句话，你可以先记这个：

`因为 Spring Boot 通常会把 Web 容器一起打进应用里，所以直接打成 jar 部署更简单。`

---

## 1. 先说结论

传统 Java Web 项目经常打成 `war`，
再部署到外部 `Tomcat`。

但 Spring Boot 项目更常见的方式是：

- 打成 `jar`
- 内嵌 `Tomcat`
- 直接 `java -jar` 启动

所以 Spring Boot 常打 `jar`，
核心原因不是“不能打 war”，
而是：

`打 jar 更符合 Spring Boot 的设计理念，也更方便部署和运维。`

---

## 2. 先回忆一下 jar 和 war 的区别

- `jar` 更像普通 Java 程序包
- `war` 更像 Java Web 应用部署包

传统模式里：

- 项目打成 `war`
- 放到外部 `Tomcat`
- 由 `Tomcat` 运行

Spring Boot 模式里：

- 项目打成 `jar`
- 应用内部已经带了内嵌 `Tomcat`
- 不需要外部再准备一个 `Tomcat`

---

## 3. Spring Boot 为什么更适合 jar

## 3.1 因为它自带内嵌容器

这是最根本的原因。

Spring Boot 的一个重要特点就是：

- 可以把 `Tomcat`、`Jetty` 这类 Web 容器作为依赖引入
- 启动应用时，容器会跟着应用一起启动

也就是说，
以前是：

- 先装好 `Tomcat`
- 再把 `war` 部署进去

现在变成：

- 应用自己就带着容器
- 直接启动应用就行

所以自然更适合 `jar`。

---

## 3.2 因为部署更简单

如果是 `jar`，
部署通常就是：

```bash
java -jar app.jar
```

这比：

- 安装外部 `Tomcat`
- 管理 `Tomcat` 版本
- 把 `war` 放进指定目录
- 配置容器环境

要简单很多。

所以对开发和运维来说，
`jar` 模式通常更轻量。

---

## 3.3 因为环境更一致

如果应用把容器一起带上，
那测试环境、预发环境、生产环境跑的就更容易保持一致。

否则如果依赖外部 `Tomcat`，
就可能出现：

- 各环境 `Tomcat` 版本不一致
- 容器配置不一致
- 部署方式不一致

而 Spring Boot 打成 `jar` 后，
应用和容器绑定得更紧，
更容易做到：

`“同一个包，在不同环境按同样方式运行。”`

这对排查问题和版本交付都很重要。

---

## 3.4 因为更适合微服务

Spring Boot 很常用在微服务架构里。

微服务的特点之一是：

- 服务数量多
- 每个服务都希望独立部署
- 启停要方便
- 交付方式要统一

如果每个服务都依赖外部 `Tomcat`，
管理起来会比较重。

而直接打成 `jar`，
每个服务自己就是一个完整可运行单元，
更符合微服务部署习惯。

---

## 3.5 因为更方便配合容器化

虽然你面试时不一定要主动展开 Docker，
但这个逻辑你可以知道。

如果项目打成 `jar`，
在做容器化部署时通常很自然：

- 把 `jar` 放进镜像
- 启动命令写成 `java -jar`

这比依赖外部 `Tomcat` 的 `war` 部署，
通常更直接。

---

## 4. Spring Boot 能不能打 war

可以。

这点你要知道，
不然面试官一问你就容易说死。

Spring Boot 也支持打成 `war`，
然后部署到外部 `Tomcat`。

但一般只有这些场景更可能这么做：

- 公司历史架构就是统一外部 `Tomcat`
- 运维规范要求必须部署到应用服务器
- 需要和老系统保持一致

所以更准确的说法不是：

`“Spring Boot 不能打 war”`

而是：

`“Spring Boot 可以打 war，但大多数新项目更常见的是打 jar。”`

---

## 5. 从工程角度怎么理解

你可以这样总结：

- `war` 模式：应用依赖外部容器
- `jar` 模式：应用自带容器

所以 Spring Boot 常打 `jar`，
本质上是把“应用”和“运行它的 Web 容器”打包成了一个整体。

这样会带来几个好处：

- 部署简单
- 环境一致
- 独立性更强
- 更适合微服务和自动化交付

---

## 6. 面试时怎么答最稳

如果面试官问：

“Spring Boot 为什么常打 jar 不打 war？”

你可以这样答：

“因为 Spring Boot 通常会把内嵌 Tomcat 一起打进应用里，所以项目本身就是一个完整可运行单元，直接通过 `java -jar` 就能启动，不需要再单独准备外部 Tomcat 去部署 war。这样做的好处是部署更简单、环境更一致，也更适合微服务和自动化交付。Spring Boot 其实也支持打 war，只是大多数新项目里打 jar 更常见。” 

---

## 7. 更短一点的口语版

“传统 Java Web 项目经常打 war，是因为它们依赖外部 Tomcat；Spring Boot 常打 jar，是因为它把 Tomcat 内嵌进应用里了，应用自己就能跑，所以部署更方便。” 

---

## 8. 面试官可能继续追问什么

### 8.1 打 jar 后 Tomcat 去哪了

你可以答：

“不是没有 Tomcat，而是 Tomcat 作为依赖被内嵌到 Spring Boot 应用里了，启动应用时也会把内嵌容器一起启动。” 

### 8.2 什么场景下还会打 war

你可以答：

“如果公司沿用传统架构，或者运维规范要求统一部署到外部 Tomcat，Spring Boot 也可以打 war。只是相对来说，新项目里 jar 更常见。” 

### 8.3 打 jar 有什么好处

你可以答：

“我理解主要是部署简单、环境一致、服务独立性更强，也更适合微服务场景。” 

---

## 9. 你只要记住这 4 句话

1. 传统项目常打 `war`，因为依赖外部 `Tomcat`
2. Spring Boot 常打 `jar`，因为内嵌了 `Tomcat`
3. `jar` 部署更简单，通常直接 `java -jar`
4. Spring Boot 也能打 `war`，只是新项目里 `jar` 更常见

---

## 10. 一句总结

`Spring Boot 常打 jar，不是因为不能打 war，而是因为它通过内嵌 Web 容器把应用做成了一个完整可运行单元，这样部署更简单、环境更一致，也更符合现代 Java 服务的交付方式。`
