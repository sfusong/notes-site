## 一、项目 1：产品生命周期管理系统

## 1. 最稳的一句话定位

`这是我参与开发的一套内网前后端系统，用来承接正式产品管理系统上线前的特殊产品生命周期节点管理和协同提醒需求。`

它不是：

- 一个纯提醒小工具
- 后续统一产品管理系统本身

它更像：

- `正式统一平台上线前，先承接紧急业务需求的自研辅助系统`

---

## 2. 为什么要做这套系统

你可以这样讲：

当时正式的产品管理系统还没有完全上线，但业务已经有比较急的真实需求，尤其是像：

- 目标盈产品
- 分红封闭式产品
- 定期分红产品

这类产品在成立后不是只等到期，而是在存续过程中会不断触发关键节点，比如：

- 分红准备
- 收益达标
- 提前到期判断

如果这些节点长期靠人工记忆和线下沟通，很容易漏掉，所以需要一套系统把这些规则和提醒先沉淀下来。

---

## 3. 这套系统技术上怎么工作的

按你现有材料，最稳的技术理解是：

- `前端`：基于公司内部 React 低代码/组件化平台
- `后端`：Java Spring Boot
- `数据来源`：数据中台 + 产品要素数据
- `提醒出口`：系统页面 + 邮件 + 内部 OA

它的基本链路可以理解成：

1. 页面录入或展示产品相关要素
2. 后端读取产品基础信息、个性化参数和工作日历等数据
3. 后端规则判断哪些事件应该触发提醒
4. 提醒结果落库
5. 页面展示事件和状态
6. 邮件 / OA 接口把提醒推送给相关业务角色

---

## 4. 这条项目你可以认领的开发工作

这条项目最适合你认领的是：

### 后端

- 后端服务设计
- 业务规则落地
- 数据库设计
- 数据中台联调
- 提醒接口对接

### 前端

- 参与页面交互和接口联动
- 不是唯一前端负责人，但确实参与开发

### 系统层面

- 不是只提需求
- 是把业务规则真正落到系统里

你可以直接讲：

`这条项目里我主做的是后端这条线，包括规则判断、数据库设计、和数据中台联调，以及和邮件、OA 提醒接口的对接；前端我也有参与接口联动和部分页面开发，但不是唯一前端负责人。`

---

## 5. 数据库你可以怎么讲

目前最稳的表结构层次是：

1. `产品基础要素表`
   - 产品编号
   - 产品名称
   - 产品类型
   - 成立相关日期

2. `产品个性化要素表`
   - 分红规则
   - 目标收益阈值
   - 特殊产品附加参数

3. `工作日历表`
   - 用来支持提醒节点按工作日规则判断

4. `提醒记录 / 提醒状态表`
   - 产品编号
   - 提醒类型
   - 触发时间
   - 当前状态
   - 是否已发送

如果被追问为什么这么设计，你就说：

`因为这个系统不是只要把产品信息存起来，而是要基于产品类型、个性化规则和工作日历去触发事件判断，所以除了基础信息，还必须有规则要素层和提醒结果层。`

---

## 6. 你最适合举的功能：目标盈提醒

这条功能最适合拿来证明你确实做过开发。

### 业务背景

目标盈产品不是只看一个固定到期日，而是要结合收益表现判断是否接近或达到目标收益条件，一旦触发，业务侧就要关注后续可能的提前到期处理。

### 技术链路

你可以这样讲：

1. 后端先读取产品基础信息和目标收益相关参数
2. 再从数据中台获取收益或净值相关数据
3. 按规则判断是否达到提醒条件
4. 如果触发，就把提醒结果落到提醒记录表
5. 页面上展示对应事件状态
6. 同时通过邮件和 OA 发送提醒

### 你做了什么

- 参与这条规则的后端实现
- 参与取数和联调
- 参与提醒结果落库
- 参与邮件 / OA 对接

### 最稳的话术

`如果举一个最能体现我开发参与度的功能，我会讲目标盈提醒。它不是按固定日期提醒，而是要结合产品信息和收益相关数据做规则判断。我主要参与的是后端这一侧，包括规则逻辑、数据联调、提醒结果落库，以及和邮件、OA 的提醒对接。`

---

## 7. 这条项目的完整讲法

你可以直接这样说：

`产品生命周期管理系统是我参与开发的一套内网前后端系统，用来承接正式产品管理系统上线前的一些特殊产品节点管理需求。因为像目标盈产品、分红封闭式产品这类业务，不是只靠一个固定日期就能管理，而是要结合产品类型、分红计划、收益条件等规则去判断关键节点，所以当时我们先做了一套自研辅助系统。技术上后端是 Spring Boot，前端基于内部 React 平台。我这边主做的是后端这条线，包括规则判断、数据库设计、和数据中台联调，以及和邮件、OA 提醒接口的对接。前端我也有参与接口联动和部分页面开发。对我来说，这条项目最能体现的是我确实参与过把金融业务规则转成系统能力的过程。`

## 二、项目 2：TA 批前预警里的静态参数稽核

## 1. 最稳的一句话定位

`这是我参与开发的一套批前静态参数稽核能力，用于在 TA 夜间批处理前，对关键配置和基础状态做规则化检查，把问题前置暴露出来。`

这条能力不是：

- 通用监控平台
- 纯报表系统

而是：

- `运营控制和业务检查能力`

---

## 2. 它和交易类批前预警的区别

你一定要讲成两层：

### 交易类批前预警

更偏：

- 当前份额
- 新申购规模
- 募集规模
- 销售额度
- 持有限额
- 产品成立条件

这部分更像：

- 恒生 AOP 的能力

### 静态参数稽核

更偏：

- 参数有没有漏设
- 参数有没有错设
- 日期和状态有没有冲突
- 配置之间有没有矛盾

这部分更像：

- 团队自研水星稽核系统

---

## 3. 你已经能确认的静态稽核项

从清单里已经能确定的典型自研稽核项有：

- 托管账号信息缺失
- 销售商清算账号信息缺失
- 手续费分成比例缺失
- 定开产品开放期设置检查
- 已销售额度检查
- 连续开放日校验
- 校验日期相关信息
- 私募产品交易限制检查
- 行情数据完整性检查
- 产品状态不相符交易检查
- 普通非交易过户检查
- 周期产品到期日与母行代销到期日对比检查
- 对比前一日系统级参数检查
- 对比前一日销售商文件列表检查
- 存储上一日系统级参数
- 存储上一日销售商文件列表
- 跑批情况汇总
- 请款与回款金额检查
- 客户浮动管理费设置
- 费率参数检查
- 产品募集规模检查
- 销售商已销售规模检查
- T+1 行情数据完整性检查
- 份额冻结异常确认检查

## 4. 这条能力技术上大概率怎么工作的

基于你现有材料，这条能力最稳的技术理解是：

1. 先从 TA、产品、账户、费率、销售商、行情等相关系统/表里取数
2. 对每类稽核项定义规则
3. 执行一轮批前检查
4. 把异常项汇总成结果
5. 提供给运营/科技在跑批前查看和处理

它底层大概率更像：

- `SQL + 规则判断 + 结果汇总`

而不是复杂的 AI/风控引擎。

这条项目不要讲成“做了个平台”，要讲成：

这背后的价值是：

## 三、项目 3：TA 自动化测试

## 1. 最稳的一句话定位

`这是我参与开发的一套围绕 TA 与代销商之间申请/确认文件交互闭环做的场景化自动化回归方案。`

这条项目最核心的，不是普通页面自动化，而是：

- 环境准备
- 批处理触发
- 文件生成与解析
- 数据库结果比对
- 回归验证标准化

---

## 2. 这套方案解决了什么问题

你可以这样讲：

TA 和代销商对接时，最关键的链路不是一个按钮，而是：

1. 代销申请文件进入 TA
2. TA 批处理
3. TA 生成确认文件
4. 确认结果回给代销商

如果每次版本升级后都靠人工准备环境、导数、跑批、查库、查文件、看结果，成本很高，也很依赖经验。

所以自动化建设的目标就是把这条核心闭环固化成可重复执行的回归方案。

---

## 3. 技术链路你必须讲清楚

按现有材料，这条项目最稳的技术链路是：

### 阶段 1：环境准备

- 用 Oracle SQL 清理/重置数据
- 修改日期参数
- 处理测试前置状态
- 必要时配合 Linux 启停服务
- 确认服务注册恢复正常

### 阶段 2：场景触发

- 通过 Jenkins Pipeline 统一触发
- 按场景执行自动化任务

### 阶段 3：核心业务闭环

- 自动生成/准备申请文件
- 进入 TA 批处理链路
- 系统生成确认文件

### 阶段 4：结果校验

- 解析确认文件
- 和数据库结果做比对
- 必要时结合文件、节点、日志一起判断

### 阶段 5：异常定位

- 判断是工具问题
- 还是测试数据问题
- 还是系统本身问题

---

## 4. 你可以认领的开发工作

这条项目里，你现在最能明确认领的是：

- 自动化思路提出
- 业务闭环拆解
- 参与技术选型
- 确认文件解析代码
- 数据库结果比对代码
- 对 TA 相关数据库结果口径的理解

你可以这样讲：

`这条项目里我不只是做场景设计，也参与了关键实现。尤其是确认文件解析和数据库结果比对这部分，我自己是写过代码的，所以我对这套自动化不是只停留在业务层，而是对核心实现链路也比较清楚。`

---

## 5. 你在这里最适合强调的技术点

### 1. Jenkins Pipeline

它是统一入口，不是简单跑脚本。

### 2. Oracle SQL

既用于环境准备，也用于结果断言。

### 3. 确认文件解析

这是你最适合 owning 的“编码点”。

### 4. 数据库比对

这是你最适合 owning 的第二个“编码点”。

### 5. 闭环自动化

这不是某个小功能，而是一整条申请/确认链路的标准化回归。

---

## 6. 这条项目的完整讲法

你可以直接这样说：

`TA 自动化测试是我参与开发的一套场景化自动化回归方案，核心是把 TA 和代销商之间最重要的申请文件进入、TA 批处理、确认文件回传这条闭环标准化。因为这类场景如果完全靠人工准备环境、跑批、查库、查文件去做回归，成本非常高，也很依赖测试经验，所以我们把高频场景沉淀到 Jenkins Pipeline 上做统一触发。技术上前面会结合 Oracle SQL 做环境准备和数据清理，中间触发批处理，后面再通过确认文件解析和数据库结果比对来判断结果。我在这个项目里既参与了思路和场景设计，也参与了确认文件解析和数据库比对这部分实现，所以我对这套方案的技术链路是比较熟的。`
