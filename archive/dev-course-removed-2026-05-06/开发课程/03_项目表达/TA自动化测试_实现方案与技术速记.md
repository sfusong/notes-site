# TA 自动化测试：实现方案与技术速记

这份是面试速记版。

目标不是讲全，而是让你：

- 知道这个工具大概怎么实现
- 知道每个技术在里面干什么
- 面试被问时能稳稳答出来

---

## 一、先讲一句话

`这是一个围绕 TA 与代销商核心文件交互闭环做的场景化自动化回归方案。`

它不是简单点页面，
而是把：

- 环境准备
- 场景触发
- 文件交互
- 结果校验
- 异常定位

串成一条自动化流水线。

---

## 二、实现方案怎么理解

你就记 5 步。

### 1. Jenkins 统一触发

- 测试人员在 Jenkins 页面选环境、选场景、点执行
- Jenkins 负责把整条流程跑起来

### 2. SQL 做环境准备

- 改日期
- 改参数
- 清理中间数据
- 重置状态

为什么要这样做：

- TA 这类批处理系统强依赖时间和状态
- 不先准备环境，后面跑出来的结果不可信

### 3. Linux 脚本做服务控制

- 服务启停
- 环境恢复
- 配合时间切换让配置生效

### 4. 执行业务场景

核心不是单点接口，而是业务链路：

- 产品分发给代销商
- 代销商申请进入 TA
- TA 做确认、份额、净值处理
- 确认结果再回传

### 5. 多通道结果校验

校验不只看一个地方，而是一起看：

- 数据库
- 缓存
- 文件
- 批处理节点
- Jenkins 结果

## 三、技术上用了什么

你就按“每一层干什么”记。

### 1. Jenkins

作用：

- 统一入口
- 参数化执行
- 跑自动化场景
- 记录执行结果

面试一句话：

`Jenkins 在这里更像自动化执行总控台。`

### 2. Jenkins Pipeline

作用：

- 把多个步骤串起来
- 形成标准执行流程

比如：

- 准备环境
- 执行案例
- 校验结果

面试一句话：

`Pipeline 就是把整条自动化流程标准化。`

### 3. Oracle SQL

作用：

- 环境初始化
- 数据准备
- 结果查询
- 状态断言

面试一句话：

`SQL 是这个项目里环境准备和结果校验的核心手段。`

### 4. Linux 脚本

作用：

- 服务启停
- 环境恢复
- 配合批处理时点切换

面试一句话：

`Linux 脚本主要承担环境控制，不是业务逻辑本身。`

### 5. Eureka

作用：

- 看服务重启后是否恢复正常注册

面试一句话：

`Eureka 检查是为了避免服务没恢复导致的假失败。`

### 6. Redis

作用：

- 检查缓存状态是否刷新正确

面试一句话：

`因为数据库对了，不代表缓存就对了。`

### 7. 文件校验

作用：

- 看申请/确认文件有没有生成
- 文件时间是否正确
- 更成熟的做法是继续校验文件内容

面试一句话：

`文件校验是这个项目里很关键的一层，因为它本身就在验证 TA 和代销商之间的文件交互闭环。`

### 8. 节点监控 / 预警页

作用：

- 看批处理跑到哪里
- 看哪个节点失败
- 看报错详情

面试一句话：

`节点监控页主要用于定位链路卡在哪一层。`

## 四、这个项目的架构怎么讲

你不用讲太复杂，就说：

`Jenkins 负责统一触发，SQL 和 Linux 负责环境准备，TA 系统负责执行业务链路，数据库/缓存/文件/节点监控负责联合校验。`

这句话已经够面试用了。

## 五、如果面试官问“这个工具最后长什么样”

你可以说：

“我理解它不是一个完全独立自研的大一统前端平台，而更像由 Jenkins 页面作为统一触发入口，再结合业务页面、批处理监控页面和查库查文件结果共同组成的一套自动化回归方案。” 

---

## 六、如果面试官问“文件内容自动校验怎么做”

你就记这句：

`先解析文件，再从数据库生成预期结果，然后按关键字段逐条对账，最后把差异输出到流水线结果里。`

关键字段可以举例：

- 申请编号
- 客户号
- 产品号
- 确认状态
- 确认金额
- 确认份额
- 净值

## 七、你最稳的项目介绍

“这个项目本质上是围绕 TA 和代销商之间核心申请/确认文件交互做的场景化自动化回归方案。我的角色更偏业务提出方和场景 owner，主要负责梳理哪些高频核心链路最值得做自动化，定义前后置条件和关键校验点，再协同开发测试同事推进落地。技术上，这套方案主要由 Jenkins 统一触发，前面通过 SQL 和 Linux 脚本做环境准备，中间执行业务场景，后面再结合数据库、缓存、文件和批处理节点做联合校验。” 

---

## 八、你最容易被问的 5 个问题

### 1. 为什么要用 Jenkins

答：

`因为要把复杂、重复、依赖人的执行步骤标准化，形成统一入口。`

### 2. 为什么 SQL 这么重要

答：

`因为 TA 批处理场景很多关键状态只能通过数据库层高效准备和验证。`

### 3. 为什么还要看文件

答：

`因为这个项目本身就在验证代销商和 TA 的文件交互闭环，只看库不够。`

### 4. 最难的点是什么

答：

`最难的不是单个测试脚本，而是把时间、状态、服务、文件、节点这些依赖串成一条稳定的验证链路。`

### 5. 你在里面到底做了什么

答：

`我主要负责场景梳理、规则拆解、推动落地和结果验收，不是底层唯一实现者。`
