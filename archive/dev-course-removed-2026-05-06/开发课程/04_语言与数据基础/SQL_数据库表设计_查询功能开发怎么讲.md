# SQL / 数据库表设计 / 查询功能开发怎么讲

这份材料的目标，不是把你训练成 DBA，而是让你在开发岗面试里，能把：

- `SQL 是做什么的`
- `表结构设计大概怎么想`
- `查询功能开发大概怎么做`
- `历史数据回填怎么理解`
- `你在广发证券项目里到底做了什么`

讲得稳、讲得像做过中后台开发的人。

---

## 一、SQL 是什么

最简单理解：

`SQL` 是和关系型数据库交互的语言。

它主要做几件事：

- 查询数据
- 新增数据
- 更新数据
- 删除数据
- 做表结构变更

所以你面试时最稳的答法是：

“SQL 是和关系型数据库交互的语言，既可以做数据查询和增删改，也可以做表结构定义和变更。在业务系统开发里，SQL 是后端实现数据读写和查询能力的基础。”

---

## 二、什么是表结构设计

最简单理解：

`表结构设计` 就是在数据库里定义：

- 要存什么数据
- 每个字段是什么意思
- 字段类型是什么
- 哪些字段必填
- 哪些字段要建立索引
- 表和表之间怎么关联

所以表结构设计本质上不是“画几个表”，而是：

`把业务对象稳定地映射成数据结构。`

这句话很适合面试时说。

---

## 三、表结构设计时最常见会考虑什么

你现在先记最常见的 5 点就够：

1. `字段是否够表达业务`
2. `字段类型是否合适`
3. `是否需要主键`
4. `哪些字段会高频查询，需要索引`
5. `后续扩展和历史兼容性`

比如一个授信变动查询功能，最容易想到的字段可能包括：

- 客户号
- 授信额度
- 变动前值
- 变动后值
- 变动时间
- 变动原因
- 操作人

所以你面试里最好体现出一个意识：

`表结构设计不仅是为“存下来”，也是为后续查询和展示服务。`

---

## 四、什么是表结构变更

最简单理解：

就是原来的表不够用了，需要：

- 新增字段
- 修改字段类型
- 增加约束
- 增加索引

在企业项目里，这非常常见。

因为很多系统不是从零开始的，而是已有系统随着需求变化不断演进。

所以你如果讲自己做过“表结构变更”，会很像真实做过中后台开发的人。

---

## 五、什么是历史数据回填

这个概念和你广发证券项目特别相关。

最简单理解：

`历史数据回填` 就是在表结构变化后，把原来存量数据补齐到新口径下。`

比如原来没有某个字段，后来为了支持查询功能新增了这个字段，那旧数据怎么办？

就需要：

- 根据历史数据规则补值
- 根据已有记录推算
- 做一次批量更新

所以历史数据回填的本质是：

`让老数据也能适配新功能和新结构。`

---

## 六、查询功能开发一般怎么理解

查询功能开发不是“写一个 select 就完了”，更完整地看，通常包括：

1. 明确查询条件
2. 明确返回字段
3. 设计排序和分页
4. 考虑性能
5. 处理导出需求

也就是说，一个中后台查询页背后，通常至少有这些问题：

- 按什么筛选
- 查哪些字段
- 一次返回多少
- 怎么排序
- 是否支持导出

这就是为什么“查询功能开发”在面试里并不低级，它其实很能体现中后台开发的基本功。

---

## 七、如果面试官问：一个查询功能你会怎么做

你可以这样答：

“我通常会先确认查询对象、筛选条件和返回字段，再判断是否需要分页、排序和导出。之后会看哪些字段是高频过滤条件，是否需要索引来支撑性能。对中后台系统来说，查询功能的核心不是把数据查出来，而是让查询口径、性能和页面使用方式一起收得住。”

这句很适合你。

---

## 八、和你的广发证券项目怎么绑定

你简历里这条是：

`客户授信管理系统“授信变动查询”功能的设计与后端开发，基于 PostgreSQL 设计表结构变更方案并完成历史数据回填，支持风控人员对授信调整进行列表查询、统计及 CSV 导出。`

这条项目非常适合拿来讲：

1. `查询功能开发`
2. `表结构变更`
3. `历史数据回填`

所以你后面最稳的讲法应该是：

“这个项目本质上是一条典型的中后台查询功能开发。我做的不是简单加个页面，而是围绕授信变动记录的查询需求，先梳理查询口径，再设计表结构变更方案，并处理旧数据如何适配新结构的问题，最后提供列表查询、统计和 CSV 导出能力。”

这句话很有力量。

## 十、如果面试官问：为什么需要历史数据回填

你可以答：

“因为表结构调整后，新的查询功能依赖新的字段和口径，如果只支持新产生的数据，旧数据就无法统一展示或统计。所以需要通过历史数据回填，让存量记录也能适配新的结构和查询需求。”

---

## 十一、如果面试官问：为什么查询功能也值得单独讲

你可以答：

“因为中后台系统里很多核心工作其实就是围绕数据录入、状态流转、查询统计和导出展开的。查询功能看起来基础，但它能很直接体现对业务口径、数据库结构、性能和页面使用方式的理解。”

---

## 十二、如果面试官问：查询功能最容易踩什么坑

你可以这样答：

“我理解最容易踩的坑主要有三个：第一是查询口径不清，导致业务理解和系统结果不一致；第二是表结构和索引没有考虑到高频过滤条件，导致性能不好；第三是只考虑新数据，没有处理历史数据适配，导致查询结果不完整。”

这句会显得很成熟。

---

## 十三、表结构设计和后端分层怎么连起来

你可以把这一层也讲顺：

- 前端传查询条件
- Controller 接请求
- Service 组织查询逻辑和业务口径
- Repository / DAO 执行 SQL
- 数据库按表结构和索引返回结果

这样你前面学的 Spring Boot / REST API 和这节数据库内容就连起来了。

## 十四、面试可直接复述的一版总答案

“SQL 是和关系型数据库交互的基础语言，在业务系统开发里既用于数据查询和增删改，也用于表结构定义和变更。表结构设计本质上是把业务对象稳定映射成数据结构，不只是为了存数据，也要服务后续查询和展示。  
在我做的广发证券项目里，我参与的是一条比较典型的中后台查询功能开发。具体包括梳理查询口径、设计表结构变更方案、处理历史数据回填，以及支持列表查询、统计和 CSV 导出。对我来说，这类项目的价值不在于功能表面简单，而在于它会把业务口径、数据库结构、查询性能和历史数据兼容性一起拉进来考虑。”

1. SQL 是和关系型数据库交互的语言
2. 表结构设计是在定义业务对象如何映射成数据结构
3. 表结构设计不仅服务存储，也服务查询和展示
4. 表结构变更在企业系统里非常常见
5. 历史数据回填是让老数据适配新结构和新功能
6. 查询功能开发不只是写 select，还包括口径、分页、排序和导出
7. 查询功能很能体现中后台开发基本功
8. 广发证券这条项目最适合讲查询功能、表结构变更和历史数据回填
9. 查询功能最容易踩的坑是口径、性能和历史兼容性
10. 数据库这课最好和 Spring Boot / REST API 的后端分层一起讲
